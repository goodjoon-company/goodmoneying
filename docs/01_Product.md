# 제품 기준

Status: Accepted
Last Updated: 2026-07-11

## 문서 역할

이 문서는 goodmoneying의 제품 목표, 범위, 정책, 요구사항, 마일스톤(Milestone), 사용자 화면을 정의하는 단일 기준(source of truth)이다. 현재 구현 사실과 앞으로의 제품 우선순위를 분리하며, 설계·구현·검증 상세는 아키텍처(Architecture), 계약(Contract), GitHub Issue, `docs/Test/`, `docs/History/`에서 관리한다.

현재 기준은 GitHub Issue #4의 제품 의도 결정과 Issue #5의 구현 산출물 분류를 반영한다. 브레인스토밍(Brainstorming) 또는 과거 계획과 이 문서가 다르면 이 문서를 우선한다.

## 제품 원칙

- goodmoneying은 운영 콘솔이 아니라 개인 투자 의사결정을 돕는 데이터 기반 작업대다.
- 운영 화면은 데이터 신뢰성을 확인하는 내부 도구다. 사용자 가치의 최종 화면은 관심 대상 탐색, 비교, 분석, 전략 실험이다.
- 가까운 로드맵(roadmap)은 1~2단계만 구체화하고, 이후 단계는 진입 조건을 확인하는 결정 게이트로 둔다.
- 구현된 사실, 확정된 제품 결정, 후보 아이디어, 의도적으로 보류한 범위를 서로 다른 상태로 관리한다.
- "나중에 쓸 수도 있는 데이터"보다 다음 투자 판단에 실제로 쓰는 데이터와 설명 가능성을 우선한다.
- M1~M3에서 정의한 최소 기능 제품(MVP, Minimum Viable Product) 구현은 이미 완료됐다. 현재 로드맵은 초기 구현 이후 제품화(Post-MVP Productization)와 사용자 가치 검증을 다룬다.
- 개인 단일 사용자 제품을 전제로 하며 투자 수익 보장이나 법적 의미의 매수·매도 추천을 목표로 하지 않는다.

## 제품 목표

### 현재 목표

다음 1개월의 목표는 운영 안정화 자체가 아니라, 신뢰 가능한 업비트(Upbit) 데이터 기반 위에서 사용자가 관심 코인을 비교하고 투자 후보를 좁힐 수 있는 작업대로 이동하는 것이다.

현재 제품화 단계의 성공은 다음 결과로 판단한다.

1. 사용자가 관심 코인의 현재 상태와 과거 데이터 범위를 이해한다.
2. 사용자가 여러 관심 코인을 같은 기준으로 비교한다.
3. 사용자가 비교 결과를 바탕으로 더 살펴볼 투자 후보를 좁힌다.
4. 데이터가 부족하거나 오래됐을 때 그 사실과 복구 경로를 알 수 있다.

수집 성공, 커버리지(Coverage), 최신성, 결측 복구는 이미 구현된 초기 제품 기준선이다. 현재 단계의 완료는 이 기반을 사용해 위 사용자 결과를 실제로 달성하는 것으로 판단한다.

### 장기 방향

장기적으로 코인과 주식의 시장 데이터를 수집·분석하고, 뉴스·공시·리포트와 대규모 언어 모델(LLM, Large Language Model) 신호를 결합해 전략을 실험한다. 충분히 검증된 전략은 봇(Bot) 시뮬레이션(Simulation), 모의매매(Paper Trading), 실제 자동매매(Live Trading)로 단계적으로 확장한다.

이 장기 방향은 확정된 실행 계획이 아니다. 각 영역은 직전 단계에서 사용자 가치, 필요한 데이터, 비용, 리스크를 다시 확인한 뒤 별도 요구사항과 아키텍처 결정 기록(ADR, Architecture Decision Record)으로 구체화한다.

## 완료된 초기 제품 기준선

M1~M3의 업비트 수집 운영 기능은 구현과 자동화 검증을 마쳤다. 현재 작업은 아래 기능을 다시 MVP로 구현하는 것이 아니라, 제품 기반으로 유지하거나 사용자 가치 흐름에 맞게 재배치하는 단계다. 상세 근거는 GitHub Issue #5와 `docs/History/2026-07-10-P0-3-현재-구현-산출물-분류.md`에 있다.

| 기반 | 제품상 위치 | 현재 처리 |
|---|---|---|
| 업비트 수집 후보군 상위 100개와 활성 수집 대상 50개 | 신뢰 가능한 코인 데이터 기반 | 유지한다. 후보 선정과 활성 대상 계약의 신뢰성을 우선한다. |
| 관심종목 화면 | 투자 후보 탐색의 중심 화면 | 유지하고 비교·탐색 흐름을 다음 단계에서 강화한다. |
| 코인 상세 레이어 | 관심 대상 분석의 진입점 | 유지하고 운영 상세보다 투자 분석 맥락을 강화한다. |
| Backfill 관리 | 과거 데이터 준비와 결측 복구 도구 | 유지하되 별도 제품 중심 화면으로 확장하지 않는다. |
| 운영 상태 대시보드 | 데이터 신뢰성을 확인하는 내부 운영 도구 | 유지하되 투자 후보 탐색보다 앞서 확장하지 않는다. |
| 실시간 체결 히트맵 | 투자 판단 가치가 검증되지 않은 구현 | 현재 구현만 보존하고 추가 확장을 보류한다. |

호가 원천 스냅샷(Snapshot) 저장과 다중 워커·고가용성(High Availability)은 초기 제품에 포함되지 않은 미구현 범위다. 현재 필요성이 확인되지 않았으므로 호가 요약(Orderbook Summary)과 단일 워커 운영을 유지하고 각각 전략 실험·확장성 결정 게이트에서 재검토한다.

현재 런타임(Runtime) 기능 중 즉시 삭제할 제품 기능은 없다. 과거 상세 Task 문서는 Milestone 요약과 Test/History 증적으로 이관한 뒤 제거했다.

## 현재 제품화 범위

현재 단계는 업비트 수집 MVP를 추가 구현하는 범위가 아니다. 완료된 데이터 기반을 사용해 관심 코인을 비교하고 투자 후보를 좁히는 제품 경험을 만드는 범위다.

- 업비트 수집 후보군 상위 100개와 활성 수집 대상 최대 50개, 캔들(Candle), 현재가, 거래대금, 호가 요약 수집은 완료된 기반으로 유지한다.
- 관심종목에서 현재가, 거래대금, 등락률, 기준 시각, 캔들 데이터 범위와 데이터 신뢰도를 같은 기준으로 비교한다.
- 코인 상세에서 가격 흐름과 분석에 사용한 데이터 맥락을 이해한다.
- 데이터가 부족하거나 오래됐을 때 그 상태와 Backfill 복구 경로를 알 수 있다.
- 비교·정렬·필터·상세 연결을 통해 다음에 분석할 투자 후보를 좁힌다.
- 관심종목과 코인 상세는 사용자 가치 화면으로, 운영 상태와 Backfill은 이를 지지하는 내부 도구로 구분한다.

### 제품 화면 경계

| 구분 | 화면 | 역할 |
|---|---|---|
| 사용자 가치 | 관심종목 | 관심 코인을 같은 기준으로 비교하고 후보를 좁힌다. |
| 사용자 가치 | 코인 상세 | 선택한 코인의 가격 흐름과 데이터 맥락을 분석한다. |
| 내부 운영 | 운영 상태 | 수집 파이프라인과 데이터 신뢰성을 진단한다. |
| 내부 운영 | Backfill 관리 | 과거 데이터 준비, 결측 복구, 작업 제어를 수행한다. |

## 현재 범위 밖

다음 항목은 현재 제품화 범위에 포함하지 않는다. `Deferred`는 의도적인 보류이고 `Candidate`는 제품 가치와 범위를 아직 결정하지 않았다는 뜻이다.

- 국내 주식과 미국 주식 데이터 기반
- 뉴스, 공시, 증권사 리포트 수집과 원천 보존
- LLM 요약과 구조화 신호(Signal)
- 호가 원천 스냅샷의 고빈도 장기 보존과 재생(Replay) 분석
- 실시간 체결 히트맵의 추가 확장
- 시장 스캐닝(Scanning), 전략 설계, 백테스트(Backtest)
- 봇 설계, 시뮬레이션, 모의매매, 실거래
- 현재 병목 근거가 없는 다중 워커, 다중 인스턴스, 메시지 큐(Message Queue), 고가용성 고도화

다중 사용자, 권한(Authorization), 과금(Billing), 팀 공유 기능은 장기 후보가 아니라 계속 제외하는 범위다.

## 사용자 시나리오

### 초기 사용자

초기 사용자는 본인의 투자 분석과 자동화 기반을 구축하는 개인 단일 사용자다.

### 현재 제품화 시나리오

1. **데이터 신뢰 확인**
   - 사용자는 관심종목과 코인 상세에서 최신 데이터 시각과 저장된 과거 범위를 확인한다.
   - 데이터가 부족하거나 오래됐으면 화면에서 그 상태를 이해하고 Backfill 또는 운영 상태로 이동한다.

2. **관심 코인 비교와 후보 선택**
   - 사용자는 관심 코인의 현재가, 거래대금, 등락률, 데이터 범위와 분석 기준을 같은 화면에서 비교한다.
   - 정렬과 필터로 후보를 좁히고 더 살펴볼 코인을 선택한다.

3. **코인 상세 분석 진입**
   - 사용자는 선택한 코인의 가격 흐름과 데이터 품질을 확인한다.
   - 어떤 기간과 데이터가 분석에 쓰였는지 이해한 상태에서 다음 판단으로 이동한다.

4. **과거 데이터 준비와 복구**
   - 사용자는 필요한 코인의 Backfill 계획을 만들고 작업 상태를 확인한다.
   - 시스템은 이미 저장된 구간을 다시 요청하지 않고 결측 구간만 수집하며 실패 작업을 안전하게 재개한다.

5. **운영 문제 진단**
   - 운영자는 내부 운영 화면에서 수집 성공률, 최신성, 결측, 워커(Worker), 저장량 변화를 확인한다.
   - 운영 문제 해결은 사용자 가치 화면의 신뢰성을 회복하기 위한 보조 흐름이다.

### 후속 결정 시나리오

- P3 진입 전에는 어떤 분석 지표와 조건이 실제 후보 판단에 쓰이는지 먼저 검증한다.
- 호가 원천 저장은 호가 요약만으로 답할 수 없는 전략 실험이 정의됐을 때 다시 결정한다.
- 주식, 뉴스, LLM, 봇, 거래는 직전 단계의 사용자 가치가 확인된 뒤 별도 시나리오와 위험을 정의한다.

## 수집 품질 정의

수집 품질 지표는 사용자에게 데이터의 신뢰 여부를 설명하고 운영자가 원인을 진단하기 위한 공통 기준이다.

| 지표 | 정의 |
|---|---|
| 성공률(Success Rate) | 관찰 구간의 기대 수집 횟수 대비 성공 횟수 |
| 커버리지(Coverage) | 목표 구간 중 저장된 고유 슬롯(slot)의 비율 |
| 결측률(Missing Rate) | 목표 구간 중 저장되지 않은 슬롯의 비율 |
| 중복률(Duplicate Rate) | 전체 수집 이벤트 중 동일 자연키가 반복된 비율 |
| 최신성 지연(Freshness Lag) | 마지막 성공 데이터 시각부터 현재까지의 지연 |
| 저장 지연(Storage Lag) | 거래소 데이터 시각부터 저장 완료까지의 지연 |
| 저장량 변화(Storage Delta) | 현재 관찰 구간의 저장 행 또는 바이트 증가량 |

기대 횟수는 활성 수집 대상, 데이터 유형, 관찰 구간, 수집 주기, 거래소 달력을 함께 고려한다. 코인은 24시간 시장 기준을 사용하며, 향후 주식은 정규장·확장장·휴장일과 데이터 공급원 제약을 반영한다. 저장량 변화는 수집 정지의 직접 증거가 아니므로 최신성, 성공률, 워커 heartbeat와 함께 판단한다.

### 측정 계약

| 기호 | 의미 |
|---|---|
| `W = [start_at, end_at)` | 품질을 계산하는 관찰 구간(Window) |
| `E(d, i, W)` | 데이터 유형 `d`, 종목 `i`, 구간 `W`에서 수집 계획과 시장 시간 정책상 저장되어야 하는 기대 슬롯(Expected Slot) 수 |
| `S(d, i, W)` | 저장소에 실제 존재하는 고유 슬롯(Unique Slot) 수 |
| `M(d, i, W)` | 결측 구간(Missing Range)으로 판정된 슬롯 수 |
| `A(d, i, W)` | 수집 시도(Collection Attempt) 수 |
| `A_success(d, i, W)` | 성공한 수집 시도 수 |
| `R(d, i, W)` | 저장된 원천 행(Raw Row) 수 |
| `U(d, i, W)` | 저장된 고유 키(Unique Key) 수 |
| `latest_collected_at(d, i)` | 해당 데이터 유형의 최신 수집 시각 |
| `latency_p95(d, i, W)` | 원천 데이터 시각부터 DB 저장 완료까지 걸린 시간의 95퍼센타일(Percentile) |

| 지표 | 산식 |
|---|---|
| 성공률 | `A_success / max(A, 1)` |
| 커버리지 | `S / max(E, 1)` |
| 결측률 | `M / max(E, 1)` |
| 중복률 | `max(R - U, 0) / max(R, 1)` |
| 최신성 지연 | `now_kst - latest_collected_at` |
| 저장 지연 | `latency_p95` |
| 오늘 저장 행 | `count(rows where collected_at >= today_kst_start)` |

품질 점수(Quality Score)는 데이터 유형별 최신성 임계값 `freshness_threshold(d)`와 저장 지연 임계값 `latency_threshold(d)`를 사용한다.

```text
freshness_score = max(0, 1 - freshness_lag / freshness_threshold(d))
latency_score = max(0, 1 - latency_p95 / latency_threshold(d))

quality_score = round(100 * (
  0.35 * coverage
  + 0.25 * freshness_score
  + 0.20 * success_rate
  + 0.10 * (1 - min(duplicate_rate, 1))
  + 0.10 * latency_score
))
```

`quality_score >= 90`이고 필수 지표가 임계값을 넘지 않으면 정상, `70 <= quality_score < 90` 또는 단일 필수 지표 경고가 있으면 주의로 본다. `quality_score < 70`, 최신성 지연이 임계값의 3배 초과, 커버리지 80% 미만, 워커 heartbeat 지연 중 하나라도 있으면 장애로 본다.

| 데이터 유형 | 기대 슬롯 `E(d, i, W)` | 기본 최신성 임계값 |
|---|---|---|
| 현재가 스냅샷(Ticker Snapshot) | 수집 계획상 활성인 1분 버킷(bucket) 수 | 180초 |
| 1분 원천 캔들(Source Candle) | 구간 안의 완성된 1분 캔들 수 | 캔들 종료 후 180초 |
| 호가 요약 | 수집 설정의 샘플링 버킷 수 | 180초 |
| 체결 이벤트(Trade Event) | 기대 슬롯 기반 커버리지에서 제외하고 수신률, 저장 지연, 저장 행 변화로 판정 | 180초 |

## 정책

- 시스템 프로세스와 코인 수집은 업비트 시장 특성에 맞춰 24시간 365일 운영한다.
- 향후 주식 수집은 거래소 운영 시간, 휴장일, 데이터 공급원 제약을 반영하며 비수집 시간은 장애로 취급하지 않는다.
- 원천 데이터 보존은 재처리와 감사(Audit) 가치, 비용, 실제 분석 필요성을 함께 고려한다.
- 호가 원천 데이터가 없는 과거 구간에서는 호가 기반 전략을 완전하게 검증할 수 없음을 명시한다.
- LLM 분석 결과는 투자 판단의 보조 입력이며 사실 데이터와 같은 확정 정보로 취급하지 않는다.
- 제품은 투자 수익을 보장하지 않으며 매수·매도 추천의 법적 책임을 제품 목표로 두지 않는다.
