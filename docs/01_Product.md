# 시스템 트레이딩 플랫폼 제품 요구사항(PRD, Product Requirements Document)

상태: 승인됨(Accepted)

최종 갱신: 2026-07-18

제품 범위: 업비트(Upbit) 가상자산

## 1. 문서 역할

이 문서는 goodmoneying의 제품 목표, 범위, 정책, 성공 기준, 단계별 계획을 정의하는 단일 기준(source of truth)이다. 기존 문서와 충돌하면 이 문서를 우선한다. 기술 경계는 `docs/02_Architecture.md`, 기계 검증 가능한 인터페이스는 `docs/contracts/`, 되돌리기 어려운 결정은 `docs/ADR/`, 실행 단위는 GitHub Issue, 검증 증적은 `docs/Test/`, 인계는 `docs/History/`에서 관리한다.

## 2. 제품 비전

goodmoneying은 단순 시세 조회 도구가 아니라 다음 흐름을 하나의 감사 가능한 작업 공간에서 제공하는 개인용 상용 수준 시스템 트레이딩(Systematic Trading) 플랫폼이다.

`시장·계좌 원천 수집 → 품질 검증 → 지표 연구 → 전략 설계 → 재현 가능한 백테스트 → 모의거래 → 그림자 거래 → 위험 통제가 적용된 실거래 준비`

초기 자산 범위는 업비트 가상자산으로 제한한다. 주식 데이터, 주식 주문, 다른 거래소는 구현하지 않는다. 내부 모델은 향후 어댑터를 추가할 수 있도록 거래소·자산·거래쌍 식별자를 명시하지만 현재 제품 화면에 존재하지 않는 기능을 노출하지 않는다.

## 3. 사용자와 핵심 결과

초기 사용자는 자신의 연구, 운영, 계좌와 위험을 직접 책임지는 단일 운영자다. 사용자는 다음 결과를 얻어야 한다.

1. 전체 업비트 거래쌍에서 수집 대상을 고르면 별도 시작 버튼 없이 과거 백필(Backfill)과 실시간 수집이 자동 시작·재개된다.
2. 사용 가능, 무거래, 복구 필요, 획득 불가, 미검증을 구분해 데이터 신뢰도를 판단한다.
3. 형식이 지정된 비순환 그래프(Typed DAG)로 전략을 만들고 불변 버전으로 고정한다.
4. 동일한 데이터셋·전략·엔진 버전과 파라미터로 동일한 백테스트 결과를 재현한다.
5. 전략을 모의·그림자 봇에 장착하고 계좌 배분·주문·체결·포지션·손익·위험을 한 흐름으로 관찰한다.
6. 모든 실거래 경로가 명시적 승인 전에는 차단되어 있고, 긴급 정지(Kill Switch)가 신규 주문을 즉시 봉쇄함을 확인한다.
7. Upbit Lab에서 기존 공개·비공개 REST 및 WebSocket 시험 기능을 계속 사용한다.

## 4. 절대 제품 정책

- 실제 자금이 움직이는 실거래는 운영자의 별도 명시적 활성화 전까지 비활성이다. 자동화 에이전트와 테스트는 실제 주문을 생성하지 않는다.
- 출금 기능과 출금 권한은 요구하거나 사용하지 않는다.
- 비밀정보는 저장소, 일반 설정 테이블, 로그, 감사 이벤트, 테스트 증적에 평문으로 남기지 않는다.
- 업비트가 제공하지 않거나 과거에 수집하지 않은 데이터는 생성·보간해 원천 데이터로 가장하지 않는다.
- 가격, 수량, 수수료, 손익, 지표 계산은 부동소수점 대신 정밀 수치(Decimal)를 사용한다.
- 내부 시각은 UTC로 저장하고 사용자 인터페이스(UI, User Interface)는 KST를 명시해 표시한다.
- 기존 사용자 데이터, 인증정보, Upbit Lab 기능은 보존한다. 호환성 없는 데이터 삭제는 백업·비교·복구 리허설 없이는 수행하지 않는다.
- 업비트 동작은 작업 시점의 공식 개발자 문서 최신 버전을 우선한다.

## 5. 제품 범위

### 5.1 시장과 수집 정책

- 공식 API에서 전체 거래쌍을 주기적으로 동기화하고 상장·주의·거래 지원 종료·상태 이력을 시점 기준(Point-in-time)으로 보존한다.
- 시장 카탈로그 요청의 endpoint·query·요청/응답 시각·HTTP 상태·원문 JSON·checksum·수집기/스키마 버전을 manifest로 보존하고 상태 이력이 성공 manifest를 참조한다. 실패·빈 성공 응답은 상태를 바꾸지 않고 실패 manifest만 남긴다.
- 초기 기본 정책은 모든 KRW 거래쌍, 2024-01-01부터 현재까지, 가능한 캔들 백필과 지속 수집이다.
- 새 KRW 거래쌍은 자동 편입하되 사용자가 제외·일시정지할 수 있다.
- 대상별 시작일 또는 과거 N년, 데이터 유형, 기준 주기, 보존 기간, 우선순위, 활성 상태를 설정한다.
- P1의 `retention_days` 실행 범위는 원본 호가 receipt·snapshot·level뿐이다. 정규화 캔들·체결·티커 정리는 P7 용량 정책, 백업·복구 검증 전에는 수행하지 않는다.
- 정책 활성화는 백필 작업 생성, 요청 수 제한 기반 스케줄링, 미완료 구간 재개, 실시간 구독, 재시작 복구를 하나의 멱등 흐름으로 수행한다.
- BTC·USDT 인용 통화(Quote Currency)는 공식 지원과 저장 용량을 확인한 뒤 같은 모델로 확장한다.
- Coverage & Quality 화면의 KRW 필터는 P1 범위다. BTC·USDT 화면 일반화와 자동 편입은 후속 범위다.

### 5.2 데이터 의미와 품질

모든 시간 구간은 다음 상태 중 하나를 가진다.

| 상태 | 의미 |
|---|---|
| `available` | 성공한 원천 요청에서 실제 원천 행이 관찰됨 |
| `no_trade` | 성공 응답의 양쪽 인접 분 캔들로 완전히 경계된 내부 분에 체결이 없어 공식 캔들이 생성되지 않음 |
| `missing` | 수집이 기대됐으나 성공 증적과 원천 행이 없어 복구가 필요함 |
| `unavailable` | 선택 정책·거래지원·원천 기능상 데이터가 제공될 수 없음 |
| `unverified` | 아직 요청 성공 여부나 의미를 확정하지 못함 |

빈 성공 응답과 페이지 선두·후미 공백은 `no_trade`로 확정하지 않는다. 원천 캔들에는 인위적인 무거래 캔들을 저장하지 않는다. 연속 시계열은 분석 계층에서만 채움 정책(Fill Policy)을 적용하며 전략 버전과 결과에 정책, 전방 채움 여부, 결측 수, 획득 불가 비율, 처리 방식을 기록한다.

원천 계층은 시장 이력, 캔들, 체결, 호가, 티커, 개인 주문·체결, 수집 실행, 범위·품질, 원천·요청 파라미터·수집기·스키마 버전을 추적한다. 집계 계층은 1·3·5·10·15·30분, 1·4시간, 일·주·월을 지원한다. 지표 계층은 SMA, EMA, RSI, MACD, 볼린저 밴드, ATR, 스토캐스틱, ADX, 거래량·거래대금, 변동성, 모멘텀, 돌파, 최대 낙폭, 주문장 불균형, 체결 강도와 사용자 정의 지표를 지원한다.

P2-4의 미시구조(Microstructure) 첫 버전은 실시간 수집 이후의 1분 구간으로 제한한다. 기본 호가 단위(`level=0`)의 마지막 유효 스냅숏에서 스프레드, 중간가격 대비 스프레드, 상위 10단계 잔량과 주문장 불균형을 계산하고, 체결 원천에서 매수(`BID`)·매도(`ASK`) 표시별 건수·수량·불균형·체결 강도를 계산한다. 업비트가 공식 문서에서 제공하지 않는 공격자(aggressor)·메이커/테이커 의미는 추론하지 않는다. 0 분모, 미검증 원천, 불완전한 호가 단계와 체결 대사는 0·무한대·상한값으로 바꾸지 않고 상태와 `null`로 노출한다. 다주기 미시구조와 과거 미수집 호가·체결 합성은 범위 밖이다.

### 5.3 전략 스튜디오

- 노드는 시장 입력, 주기 변환, 지표, 비교·교차, 논리, 시장 체제(Regime), 진입·청산, 손절·익절, 추적 손절, 포지션 크기, 계좌 배분, 거래 시간, 재진입 대기, 위험 제한, 봇 출력을 포함한다.
- 검증기는 순환, 자료형, 주기 호환성, 미래 데이터 참조(Look-ahead Bias), 파라미터 범위, 결측 처리, 워밍업, 데이터 요구량을 실행 전에 차단한다.
- 조건 결과는 어떤 입력과 노드 때문에 참·거짓이 됐는지 설명 가능해야 한다.
- 모든 변경은 불변 전략 버전을 생성하며 실행 중인 봇에 초안을 직접 반영하지 않는다.

### 5.4 백테스트

- 라이브 전략 평가와 동일한 전략·주문 상태 경로를 사용한다.
- 캔들·이벤트 실행, 수수료, 슬리피지, 지연, 부분 체결, 최소 주문 금액, 호가 단위, 시장가·지정가, 다중 거래쌍, 자금 배분, 동시 포지션 제한을 모델링한다.
- 호가 데이터가 없을 때는 체결 가정을 숨기지 않고 결과에 기록한다.
- 학습·검증·테스트 구간, 워크포워드 분석(Walk-forward Analysis), 민감도, 부트스트랩·몬테카를로, 벤치마크 비교를 지원한다.
- 결과는 누적·연환산 수익률, 최대 낙폭, Sharpe, Sortino, Calmar, 이익계수, 승률, 평균 손익비, 거래 빈도, 회전율, 노출, MFE·MAE, 기간·거래쌍별 기여, 수수료·슬리피지, 데이터 품질 영향을 포함한다.
- 골든 데이터셋(Golden Dataset) 재생으로 백테스트·모의·라이브 평가 신호 동등성을 검증한다.
- Backtest Lab은 저장된 백테스트 run 목록을 먼저 읽고 선택한 run의 성과, 체결, 산출물, 입력·결과 hash를 읽기 전용으로 조회한다. API client는 published 전략 version과 sealed 데이터셋 version을 입력으로 새 queued 백테스트 run을 생성할 수 있으며, 대용량 체결·자산곡선은 별도 페이지 API로 조회한다. 진행 WebSocket은 연결 직후 현재 run summary에서 파생한 snapshot을 제공한다.

### 5.5 포트폴리오, 봇, 주문과 위험

봇 단계는 `draft → backtest → paper → shadow → live-ready → live`이며 `paused`, `stopped`, `faulted`로 전이할 수 있다. 봇은 전략 버전, 계좌, 총·거래쌍별 배정, 동시 포지션, 주문 한도, 일일 손실, 최대 낙폭, 시장 노출, 집중도, 실행 주기, 재진입 대기, 가격·데이터·API 이상 차단 정책을 가진다.

주문 의도(Order Intent)와 거래소 주문을 분리하고 멱등 식별자, 상태 기계, REST·개인 WebSocket·잔고 대사를 사용한다. 중복·유령 주문, 부분 체결, 연결 종료, 시간 초과 후 결과 불명을 명시적으로 처리한다. 자전거래 체결 방지(SMP)는 공식 제약을 따르며 `post_only`와 동시에 사용하지 않는다.

P5 포트폴리오는 운영자 토큰이 필요한 멱등 명령으로 생성하며, owner별 목록 조회는 cursor 문맥을 검증한다. API 명령으로 생성된 포트폴리오는 요청 ID, 멱등 키, 요청 시각, 요청 hash를 DB에 보존한다. 이 단계는 paper·shadow 운용 설정의 기반만 열며 실제 주문 또는 live 전이를 만들지 않는다.

paper 실행은 `approved` 주문 의도를 별도 작업 queue로 받아 모의 주문, 모의 체결, 포지션 projection을 같은 transaction으로 기록한다. 이 worker는 실제 Upbit 주문 제출·취소·조회, private WebSocket, 주문 테스트 API를 호출하지 않는다.

실거래 진입은 백테스트·모의·그림자 승인, 위험 설정, 주문 테스트 API, 계좌·체결 대사, 운영자 명시 활성화와 감사 사유가 모두 있어야 한다. 제품 구현 완료와 실거래 활성화는 별개의 상태다.

### 5.6 API와 실시간 통신

- HTTP/REST는 초기 부트스트랩, 범위·대용량 조회, 파일 내보내기, 헬스, 설정·버전 생성, 멱등 명령을 담당한다.
- WebSocket은 시세, 수집·품질·작업자, 백테스트, 신호, 봇, 주문·체결, 포지션·계좌, 위험, 운영 이벤트를 전달한다.
- 내부 이벤트는 버전, 토픽, 단조 증가 시퀀스, 재개 커서, 하트비트, 유실 감지, 중복 제거, 역압(Backpressure), 권한, 재연결 지수 백오프를 계약으로 가진다.
- 유실이 감지되면 REST 스냅숏과 커서로 복구한다.

### 5.7 전문 UI

UI는 데이터 준비, 전략 연구, 재현 백테스트, 모의·그림자 봇 운영, 주문·위험 감시와 장애 복구라는 제품 결과를 한 흐름에서 달성하게 해야 한다. 정확한 화면명·내비게이션·레이아웃·시각 토큰·화면별 정보·상호작용·반응형·QA 기준은 `docs/ui/`만 단일 기준으로 관리하며 이 문서에서 복제하지 않는다.

제품 불변 조건은 포인터 없이 전략을 편집할 대안, 차트의 표 대안, 모바일의 경고 확인·봇 일시정지·긴급 정지, WCAG 2.2 AA 목표다. 기존 Upbit Lab의 REST·WebSocket 시험 기능과 테스트는 삭제하지 않고 새 UI 체계에 통합한다.

## 6. 비범위

- 주식시장, 주식 데이터·지표·주문, 타 거래소 실제 연동
- 출금·입금 자동화와 출금 권한
- 다중 사용자, 공개 가입, 과금, 소셜 기능
- AI가 임의로 실거래를 활성화하거나 실제 주문을 생성하는 동작
- API가 제공하지 않는 과거 호가·티커·체결의 합성
- 측정 근거 없는 별도 메시지 브로커, 시계열 데이터베이스, 객체 저장소 도입

## 7. 비기능 요구사항

### 7.1 재현성·정확성

- 데이터셋, 전략, 엔진, 스키마, 파라미터 버전을 고정한다.
- 동일 입력은 결정론적(Deterministic) 결과를 생성한다.
- 거래소 시각, 수신 시각, 저장 시각을 구분한다.
- 거래쌍 상태 이력으로 생존 편향(Survivorship Bias)을 방지한다.

### 7.2 성능

- 운영 사용자 75백분위수에서 LCP 2.5초 이하, INP 200ms 이하, CLS 0.1 이하를 목표로 한다.
- 첫 유용 셸과 핵심 상태는 3초 안에 표시하고 정상 실시간 변화는 1초 안에 브라우저에 반영되는지 측정한다.
- 긴 목록은 가상화하고 화면별 코드 분할, 차트 지연 로딩, 집계·범위 조회를 사용한다.
- 초기 화면에서 전체 과거 데이터를 내려받지 않고 WebSocket event마다 전체 화면을 다시 조회하지 않는다.
- React rendering, route·chart chunk와 전체 bundle 크기를 운영 build와 브라우저에서 실측한다.

### 7.3 접근성

- 명확한 focus, 전체 keyboard navigation, screen reader label, 색상 외 상태 표현을 제공한다.
- `prefers-reduced-motion`을 존중하고 pointer가 아닌 keyboard·touch 대안을 제공한다.
- 터치 대상 크기, 대비와 chart table 대안을 WCAG 2.2 AA 기준으로 자동·수동 검증한다.

### 7.4 보안·감사

- 시세 조회, 계좌 조회, 주문 권한을 가능한 범위에서 분리하고 최소 권한을 사용한다.
- 토큰, JWT, 쿼리 해시, 개인 식별정보를 마스킹한다.
- 전략·봇 변경, 활성화·일시정지, 주문 의도, 위험 차단을 변경 불가능한 감사 이벤트로 기록한다.
- 의존성, 이미지, 비밀 노출, 권한, 입력 검증을 릴리스 게이트에서 검사한다.

### 7.5 복구·운영

- 작업 큐는 PostgreSQL과 `FOR UPDATE SKIP LOCKED`를 기본으로 재시도, 임대 만료, 중복 방지, 우선순위, 실패 격리, 재개를 제공한다.
- 마이그레이션은 dbmate와 확장-수축(Expand-Contract) 방식, 첫 적용·두 번째 적용·빈 DB·샘플 행 보존·스냅숏 일치·API 검증을 통과한다.
- 배포는 기존 GitHub Actions와 `mac-mini-m4` 자체 호스팅 러너만 사용한다. 운영 서버에서 수동 빌드·수정하지 않는다.
