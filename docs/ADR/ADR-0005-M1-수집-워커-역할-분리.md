# ADR-0005-M1-수집-워커-역할-분리

Status: Accepted
Date: 2026-06-21
Supersedes: `docs/ADR/ADR-0003-M1-단일-워커와-M3-5-다중화-게이트.md`

## 맥락

M1 수집 워커(Collection Worker)는 실시간 증분 수집(Realtime Incremental Collection)과 백필(Backfill)을 모두 수행할 수 있었지만, 로컬 개발과 운영 배포의 기본 실행 경로는 실시간 수집만 반복했다. 승인된 백필 작업(Backfill Job)이 DB에 `pending`으로 있어도 백필 루프를 별도 옵션으로 실행하지 않으면 처리되지 않았다.

백필은 장시간 실행될 수 있고 실시간 수집 주기를 밀 수 있다. 반대로 M1에서 메시지 큐(Message Queue)를 도입하면 현재 데이터량과 병목이 확정되기 전 과설계가 된다.

## 결정

M1은 수집 실행 단위를 두 역할로 분리한다.

- 실시간 수집 워커(Realtime Collection Worker): 활성 수집 대상의 후보 유니버스(Candidate Universe), 현재가 스냅샷(Ticker Snapshot), 호가 요약(Orderbook Summary), 1분 원천 캔들(Source Candle)을 주기적으로 수집한다.
- 백필 수집 워커(Backfill Collection Worker): 승인된 백필 작업을 DB 상태 폴링(Polling)으로 10초마다 확인하고, `pending` 또는 `running` 작업이 있으면 점유해 백필을 수행한다.

작업 분배와 제어의 단일 기준(Source Of Truth)은 DB의 `backfill_jobs`와 `backfill_job_targets` 상태다. CLI(Command Line Interface) 옵션으로 워커 모드를 바꾸는 방식은 제거하고, 실행 모듈과 프로세스 이름으로 역할을 구분한다.

## 대안

| 대안 | 장점 | 단점 |
|---|---|---|
| 실시간/백필 워커 역할 분리 | 실시간 수집 지연을 줄이고, 백필을 기본 실행 경로에 포함할 수 있다 | 업비트 API rate limit을 두 프로세스가 함께 쓰므로 백필 처리량을 보수적으로 둬야 한다 |
| 단일 워커 루프에서 실시간 수집 후 백필 처리 | 프로세스 수가 적고 구현이 단순하다 | 백필 작업이 길어지면 실시간 수집 주기가 밀릴 수 있다 |
| M1에서 메시지 큐 도입 | 작업 분배와 재시도 모델이 명확하다 | 현재 규모에서는 운영 복잡도가 크고 M4 결정 범위를 앞당긴다 |

## 결과

- 로컬 개발 실행 단위는 `realtime-collection-worker`와 `backfill-collection-worker`로 나눈다.
- prod-home 배포에서도 같은 서비스 이름과 컨테이너 이름을 사용한다.
- 백필 수집 워커는 기본 10초 폴링을 사용하고, 운영 조정은 `GOODMONEYING_BACKFILL_POLL_SECONDS` 환경변수(Environment Variable)로만 한다.
- M1의 백필 동시성(Concurrency)은 1로 유지한다. 코인별 병렬 백필과 분산 rate limiter는 M4에서 결정한다.

## 후속 작업

- M4에서 메시지 큐 기술과 분산 rate limiter를 결정한다.
- 백필 수집 워커가 실시간 수집에 영향을 주는지 운영 지표로 관찰한다.
