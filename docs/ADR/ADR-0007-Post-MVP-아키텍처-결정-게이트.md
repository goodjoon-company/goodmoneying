# ADR-0007-Post-MVP 아키텍처 결정 게이트

Status: Accepted
Date: 2026-07-10
관련 제품(Related Product): `docs/01_Product.md`
관련 Task(Related Task): GitHub Issue #7
미래 결정 게이트 정제(Refines Future Gates): `ADR-0002`, `ADR-0003`, `ADR-0004`, `ADR-0005`, `ADR-0006`

## 맥락

M1~M3에서 업비트 수집, Backfill, 운영 상태, 관심종목, 코인 상세, 실시간 체결 데이터 기반을 구현하고 검증했다. 현재 단계는 이 기능을 다시 최소 기능 제품(MVP, Minimum Viable Product)으로 구현하는 범위가 아니라, 완료된 데이터 기반을 투자 후보 탐색과 분석 경험으로 연결하는 초기 구현 이후 제품화(Post-MVP Productization) 단계다.

기존 아키텍처와 ADR은 호가 원천 스냅샷(Snapshot), 다중 워커, 메시지 큐(Message Queue), 고가용성(High Availability), PostgreSQL 파티셔닝(Partitioning)·복제(Replication), 국내·미국 주식 확장을 M3, M4, M5 같은 고정된 후속 일정에 배치했다. 그러나 현재 제품 기준에서는 해당 기능의 사용자 가치, 병목, 저장 비용, 복구 목표가 확인되지 않았다.

기존 ADR의 구현 결정은 유효하지만 미래 기술을 특정 단계에서 반드시 도입한다는 일정은 현재 제품 기준과 충돌한다. 과거 결정을 덮어쓰지 않고 미래 진입 조건을 현재 기준으로 다시 정의해야 한다.

## 결정

- 현재 업비트 데이터 기반은 PostgreSQL 단일 저장소, 실시간·Backfill 워커 역할 분리, DB 상태 기반 작업 제어, 호가 요약(Orderbook Summary)을 유지한다.
- 호가 원천 스냅샷 저장은 전략 실험에서 호가 요약만으로 답할 수 없는 시나리오가 승인될 때까지 보류한다. 진입 시 저장 해상도, 깊이, 보존 기간, 재생(Replay), 압축, 비용을 먼저 결정한다.
- 다중 워커, 메시지 큐, 분산 rate limiter, 운영 서버 다중 인스턴스, 고가용성, PostgreSQL 파티셔닝·복제·장애 조치(Failover)는 고정 마일스톤에서 자동 착수하지 않는다.
- 확장성 고도화는 다음 중 하나가 측정될 때 새 ADR과 계약 설계를 거쳐 착수한다.
  - 현재 워커 처리량이나 외부 API rate limit 때문에 수집 목표를 지속적으로 충족하지 못한다.
  - 단일 인스턴스 장애 복구 시간이 승인된 복구 목표를 넘는다.
  - 저장량, 조회 시간, 백업·복구 시간이 운영 임계값을 넘는다.
  - 승인된 새 시장 또는 전략 실험을 현재 구조로 지원할 수 없다.
- 국내·미국 주식, 뉴스·공시·리포트, 대규모 언어 모델(LLM, Large Language Model), 전략, 봇(Bot), 거래 확장은 아키텍처 일정이 아니라 제품 결정 게이트다. 사용자 가치와 데이터 요구가 승인된 뒤 모듈 경계, 계약, 비용, 운영 위험을 별도 설계한다.
- 운영 상태와 Backfill 화면은 데이터 신뢰성을 위한 내부 운영 표면으로 유지한다. 새로운 투자 후보 탐색 기능보다 운영 화면 확장을 먼저 추진하지 않는다.
- 이번 결정은 현재 API, DB, 메시지 계약과 런타임 동작을 변경하지 않는다. 미래 게이트를 통과하면 계약을 코드보다 먼저 갱신한다.

## 대안

| 대안 | 장점 | 단점 |
|---|---|---|
| M4·M5 같은 고정 일정에 확장 기술 도입 | 실행 순서가 단순하고 선행 준비가 빠르다. | 사용자 가치와 병목 증거 없이 과설계하고 운영 복잡도와 비용을 앞당긴다. |
| 주식·전략·봇까지 공통 골격을 지금 선구축 | 미래 모듈의 표면상 일관성이 높다. | 아직 확정되지 않은 데이터와 사용자 흐름이 현재 계약을 왜곡할 수 있다. |
| 미래 확장 범위를 제품에서 제거 | 현재 범위가 가장 단순해진다. | 장기 제품 방향과 단계적 확장 가능성을 잃는다. |
| 측정 가능한 결정 게이트로 유지 | 현재 제품화에 집중하고 필요할 때 근거 기반으로 확장한다. | 임계값과 새 요구사항을 진입 시점에 다시 정의해야 한다. |

## 결과

- 현재 아키텍처 문서는 완료된 M1~M3 기준선과 Post-MVP 제품화 범위를 중심으로 설명한다.
- 과거 ADR의 PostgreSQL, 워커 역할 분리, 백필 실행 규칙은 계속 유효하다.
- 과거 ADR에 적힌 M3·M4·M5 미래 일정은 현재 실행 계획이 아니라 이 ADR로 대체된 역사적 계획으로 해석한다.
- `docs/contracts/api/openapi.yaml`, `docs/contracts/db/schema.sql`, `docs/contracts/protobuf/`는 변경하지 않는다.
- 원천 호가나 확장 인프라를 선구축하지 않아 단기 복잡도와 저장 비용을 억제한다.
- 결정 게이트 진입이 늦으면 확장 준비도 함께 늦어질 수 있으므로 운영 지표와 제품 승인 근거를 추적해야 한다.

## 후속 작업

- P1·P2에서 관심종목과 코인 상세의 데이터 신뢰·비교 시나리오를 먼저 검증한다.
- P3 전략 실험 준비 단계에서 호가 원천 스냅샷 필요성을 다시 판단한다.
- 시장 확장 또는 확장성 임계값이 확인되면 새 GitHub Issue와 ADR로 구체화한다.
- GitHub Issue #8에서 제품, 아키텍처, 계약, ADR의 최종 정합성을 검증한다.
