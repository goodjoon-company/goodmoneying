# ADR-0009: 캔들 집계 테이블과 자동 집계 워커

Date: 2026-07-14
Status: Accepted
Related Issue: [#13](https://github.com/goodjoon-company/goodmoneying/issues/13)

## 맥락

분석 WebSocket 구독마다 1분 원천봉을 5·10·30분·시·일·주·월 봉으로 다시 묶으면, 장기 범위와 동시 구독이 늘수록 DB 읽기와 CPU 비용이 반복된다. Backfill이 과거 원천봉을 추가한 뒤에도 어떤 집계가 뒤처졌는지 운영자가 알 수 있어야 한다.

## 결정

- 1분 원천봉과 직접 수집된 일봉은 원천 데이터로 계속 보존한다.
- 지원 분석 단위의 OHLCV 집계 봉은 별도 집계 테이블에 멱등 upsert한다.
- 별도 집계 워커가 원천 워터마크와 집계 워터마크를 비교해 자동 작업을 생성·재개하고, 코인·단위별 진행 상태를 저장한다.
- 분석 조회는 최신 집계 테이블을 우선 사용한다. 작업 진행 중의 빈 구간은 원천봉 즉시 파생으로 보완해 데이터 정확성을 우선한다.
- 시스템 관리 화면은 수집·Backfill·집계의 실제 heartbeat와 작업/대상 행을 표시한다.
- 워커 폴링 루프(poll loop)가 실행되는 동안 처리 행 수와 독립된 단일 5초 주기 하트비트(heartbeat) 실행기를 유지해 장기 실행 대상을 `stale`로 오판하지 않게 한다. 작업마다 실행기를 만들지 않아 하트비트가 영구 차단되어도 데몬 스레드(daemon thread)가 반복 누적되지 않는다.
- 운영 PostgreSQL의 하트비트는 집계와 다른 저장소 인스턴스·연결·트랜잭션(transaction)으로 기록한다. 연결 제한(connect timeout)과 문장 실행 제한(statement timeout)은 각각 2초다. 이는 모든 네트워크·디스크 I/O(Input/Output)의 완료 상한을 뜻하지 않는다.
- 실행기는 정상·예외 종료 모두 중지 이벤트(stop event)를 설정하고 최대 3초 합류(join)하며, 제한을 넘으면 저하 상태를 오류 로그로 남기고 본 처리의 반환을 막지 않는다. 영구 차단 시 남을 수 있는 데몬은 프로세스당 최대 하나다.
- 격리 테스트용 파일 SQLite는 별도 연결을 사용하고 잠금 대기 제한(busy timeout)을 2초로 둔다. 이는 잠금 획득 대기만 제한하며 모든 SQLite I/O 완료 상한이 아니다. 연결을 공유할 수 없는 인메모리(in-memory) SQLite는 같은 저장소를 재사용한다.
- SQLite 집계는 재진입 잠금(Reentrant Lock) 안에서 원천 조회 전에 `BEGIN IMMEDIATE`로 쓰기 트랜잭션을 확보한다. 모든 집계 UPSERT 뒤 명시적으로 커밋(commit)하고 예외 시 롤백(rollback)해 원천 스냅샷(snapshot)과 집계 결과를 일치시킨다.

## 대안과 트레이드오프

- 매 분석 구독 때 즉시 집계: 저장 공간은 적지만 반복 CPU·DB 비용과 운영 가시성이 부족해 채택하지 않는다.
- DB materialized view: PostgreSQL에는 적합하지만 SQLite 개발 환경과 작업별 진행률·자동 복구를 같은 방식으로 제공하기 어려워 채택하지 않는다.
- 원천봉 수집 경로에서 동기 집계: 최신성은 좋지만 실시간 수집과 Backfill의 실패 격리를 약화하므로 별도 워커를 택한다.

## 결과

- 세 번째 워커인 집계 워커의 배포·heartbeat·상태 계약이 필요하다.
- 집계 테이블과 작업·대상 테이블은 `docs/contracts/db/schema.sql`의 단일 기준에 추가한다.
- 집계 지연은 분석 정확성 장애가 아니라 성능·운영 경고이며, 원천봉 보조 경로로 표시를 유지한다.
