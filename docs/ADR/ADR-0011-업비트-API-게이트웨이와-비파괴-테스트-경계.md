# ADR-0011: 업비트 API 게이트웨이와 비파괴 테스트 경계

Status: Accepted
Date: 2026-07-16
Related Issue: [#19](https://github.com/goodjoon-company/goodmoneying/issues/19)

## 맥락

업비트 전체 REST·WebSocket 작업대는 공개 시세뿐 아니라 인증 조회와 공식 주문 생성 테스트를 다룬다. 브라우저 직접 호출은 `Origin` 헤더가 있는 시세 조회 REST·공개 WebSocket 요청을 10초당 1회로 제한하고, 비공개 WebSocket 인증과 키 비노출 요구를 충족하지 못한다. 기존 운영 서버에 결합하면 제품 데이터 API와 외부 API 시험의 장애·키·제한 책임이 섞인다.

## 결정

- `apps/upbit_gateway/`를 운영 서버와 독립된 FastAPI 런타임 경계로 둔다.
- 브라우저는 업비트 키를 보유하지 않고 업비트에 직접 연결하지 않는다.
- 게이트웨이는 임의 URL 대신 `docs/contracts/upbit/upbit-api-catalog.yaml`의 `endpoint_id`만 받는다.
- 공식 REST·WebSocket 기능, 파라미터, 요청 제한 그룹, 안전 등급은 카탈로그를 단일 기준(source of truth)으로 둔다.
- 조회는 `read`, `POST /v1/orders/test`는 `test`, 실제 주문·모든 취소·자산 이전·입출금 생성·취소·트래블룰 검증은 `blocked`로 분류한다.
- `blocked`는 로컬 미리보기와 정책 오류까지만 허용하고 업비트에는 전송하지 않는다.
- Issue #19에서는 health·catalog와 501 실행 경계만 만들며 인증이나 상향 네트워크 호출은 구현하지 않는다.

## 대안과 트레이드오프

| 대안 | 장점 | 단점 | 판단 |
|---|---|---|---|
| 브라우저 직접 호출 | 서버 코드가 작다. | Origin 제한, 키 노출, 비공개 인증 제약이 있다. | 기각 |
| 기존 운영 서버에 결합 | 프로세스가 늘지 않는다. | 제품 API와 시험 API의 보안·장애·요청 제한 책임이 결합된다. | 기각 |
| 별도 업비트 API 게이트웨이 | 키, 제한, 안전, 추적 경계가 명확하고 독립 검증·배포가 가능하다. | 런타임과 배포 대상이 하나 늘어난다. | 채택 |

## 결과

- 카탈로그와 계약 테스트가 공식 문서 드리프트(drift) 및 상태 변경 안전 등급 회귀를 차단한다.
- 운영 서버와 수집 워커는 기존 역할을 유지하고 게이트웨이 장애의 영향을 직접 받지 않는다.
- 게이트웨이의 인증 정보 저장, 실행 제한기, 재연결, 운영 배포와 모니터링은 후속 구현이 필요하다.

## 후속 작업

- Issue #20에서 REST 실행 엔진·JWT·요청 제한·민감 정보 마스킹과 가짜 업비트 서버 통합 테스트를 구현한다.
- 프론트엔드, WebSocket 중계, 실제 허용된 비파괴 연동은 분리된 후속 Issue에서 구현한다.
