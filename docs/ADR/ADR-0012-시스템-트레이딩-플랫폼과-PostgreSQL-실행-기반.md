# ADR-0012: 시스템 트레이딩 플랫폼과 PostgreSQL 실행 기반

- 상태: Accepted
- 날짜: 2026-07-17
- 대체: ADR-0003, ADR-0007의 범위 보류 결정

## 맥락

기존 제품은 업비트 수집·분석 도구였고 전략, 백테스트, 봇, 주문과 위험 통제를 미래 결정 gate로 보류했다. 새 PRD는 이들을 정식 제품 범위로 승인했다. 처리량 근거 없이 별도 메시지 브로커나 데이터베이스를 도입하면 운영 복잡성과 복구 경로가 늘어난다.

## 결정

플랫폼을 시장 수집, 품질, 집계·지표, 전략, 백테스트, 포트폴리오·봇, 주문·대사, 위험, 운영의 경계로 분리한다. 초기 내구성 작업 큐, outbox, lease와 재시도는 PostgreSQL 및 `FOR UPDATE SKIP LOCKED`로 구현한다. 역할은 독립 worker process로 실행하지만 같은 도메인·계약을 공유한다.

별도 브로커, 시계열 DB, 객체 저장소는 처리량·지연·보존 비용·복구 목표 위반을 측정하고 별도 ADR이 승인될 때만 도입한다.

## 대안

1. **초기부터 Kafka·Redis·시계열 DB 도입**: 확장 여지는 크지만 현재 규모에서 운영·대사·복구 복잡성이 과도하다.
2. **단일 worker에 모든 책임 유지**: 배포는 단순하지만 장애 격리, 재시도, 처리량 관측과 독립 확장이 어렵다.
3. **외부 관리형 퀀트 플랫폼 사용**: 구현량은 줄지만 데이터·전략·주문 감사와 기존 배포 환경 통제가 약해진다.

## 트레이드오프

- PostgreSQL이 초기 병목이 될 수 있어 queue latency, lease contention, 처리량을 계측해야 한다.
- 역할별 process가 늘어나 배포·heartbeat·운영 화면이 복잡해진다.
- 대신 하나의 백업·트랜잭션·대사 경계를 유지하고 운영자가 복구 경로를 이해하기 쉽다.

## 결과와 후속 작업

- DB 계약에 공통 job lease, outbox, worker heartbeat를 정의한다.
- 각 worker는 idempotency, retry, dead-letter와 resume E2E를 가진다.
- DB 변경 배포는 ADR-0015의 분리 백업, checksum, 복원 rehearsal과 forward recovery gate를 필수로 통과한다.
- P7에서 측정 결과로 분리 저장소·브로커 필요성을 다시 판단한다.
