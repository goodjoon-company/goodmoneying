# ADR-0014: 실거래 기본 비활성과 주문 안전 경계

- 상태: Accepted
- 날짜: 2026-07-17
- 대체: ADR-0011의 실제 주문 전면 차단 정책

## 맥락

기존 Upbit Gateway는 실제 주문·취소를 전면 차단해 비파괴 API 시험을 보장한다. 새 제품은 주문 어댑터와 실거래 준비 기능을 요구하지만 자동화가 실제 자금을 움직이면 안 된다. 구현 가능성과 운영 활성화를 분리하고 다중 안전 gate가 필요하다.

## 결정

주문 도메인과 Upbit 실제 주문 어댑터는 구현하되 기본 운영 상태는 `live_disabled`다. `paper`와 `shadow`는 어떤 경우에도 실제 주문 endpoint를 호출하지 않는다. `live-ready`는 `POST /v1/orders/test`, private `myOrder`, REST 대사, 권한·위험 검증까지만 수행한다.

`live` 전이는 운영자 권한, 사유, 승인된 전략·백테스트·paper·shadow·위험 정책, 계좌 대사, 전역 kill switch 상태를 하나의 감사 트랜잭션에서 검증해야 한다. AI, CI, E2E는 이 전이를 수행할 권한이 없다. 출금 endpoint와 권한은 영구 거부한다.

global live capability의 권위 상태는 감사 가능한 PostgreSQL 저장소에 둔다. 초기 설치, 재시작, 상태 조회 실패, DB·cache 불일치, 새 SHA 배포에서는 `live_disabled`로 닫힌다. 환경변수, API Key 존재, 봇 상태만으로 활성화하지 않는다. 운영 승인 주체만 별도 명령으로 해제할 수 있고 CI·AI·service token에는 권한을 발급하지 않는다.

실제 주문은 주문 의도, 멱등 식별자, 사전 위험 검사, outbox 기록 후에만 제출한다. 시간 초과 결과 불명은 재주문하지 않고 대사 상태로 보낸다. SMP는 공식 `cancel_taker`, `cancel_maker`, `reduce`만 허용하고 `post_only`와 동시 사용을 거부한다.

## 대안

1. **실제 주문 코드 전면 미구현**: 가장 안전하지만 live-ready 대사와 상태 기계의 운영 준비도를 검증할 수 없다.
2. **환경변수 하나로 live 활성화**: 구현은 단순하지만 오설정·자동화·감사 누락에 취약하다.
3. **거래소 키만 있으면 자동 활성화**: 운영 의도를 증명하지 못하므로 기각한다.

## 트레이드오프

- 여러 gate와 대사 로직으로 구현·운영 복잡성이 증가한다.
- 실제 matching·체결 lifecycle을 발생시키는 격리 sandbox는 제공되지 않고 `/v1/orders/test`는 사전 검증 API이므로 실제 체결 전 완전한 검증은 불가능하다.
- 대신 코드 완성도와 실제 자금 이동 권한을 분리하고 실패가 안전한 방향으로 닫힌다.

## 결과와 후속 작업

- 주문·봇·위험 상태 기계와 권한 계약을 기계 검증 가능하게 정의한다.
- 운영 배포 검증은 권위 상태의 `live_disabled`, 실제 주문 endpoint 거부 negative test, kill switch, 출금 endpoint·권한 부재를 확인한다.
- 실제 live 활성화는 이 제품 구현과 별도의 운영 변경 요청·승인을 요구한다.
