# ADR-0016: KRW 자동 수집 정책과 PostgreSQL 내구성 조정

- 상태: Accepted
- 날짜: 2026-07-17
- 관련 이슈: [#28](https://github.com/goodjoon-company/goodmoneying/issues/28)
- 대체 범위: 상위 100 후보·활성 50 제한과 수동 백필 시작 전제

## 맥락

기존 수집기는 24시간 거래대금 상위 후보와 최대 50개 활성 대상을 운영자가 고른 뒤 수동 백필을 시작했다. 시스템 트레이딩 연구는 당시 거래 가능했던 전체 KRW 시장, 재시작 뒤 이어지는 과거 수집, 수집 정책과 실제 실시간 구독의 차이를 재현 가능하게 설명해야 한다. 프로세스 메모리의 큐나 고정 WebSocket 목록은 이 기준을 충족하지 못한다.

## 결정

1. Upbit 공식 시장 목록 전체를 주기적으로 저장하고 상태 변경을 유효 기간 이력으로 보존한다.
2. 모든 공식 KRW 시장에 2024-01-01 UTC 시작, 1분 원천 캔들, 실시간 체결·호가·티커, 연속 수집, 보존 기한 없음, 우선순위 100의 기본 정책을 자동 적용한다.
3. 시장과 데이터 유형·주기 조합을 수집 대상 사양(collection target specification)으로 두고, 운영자의 일시정지·제외는 카탈로그 재동기화가 덮어쓰지 않는다.
4. 과거 캔들은 PostgreSQL 작업 행과 안정된 멱등 키로 생성한다. 소유권은 `FOR UPDATE SKIP LOCKED`로 선택하면서 임대 소유자·만료·시도 횟수를 같은 트랜잭션에서 갱신한다. 만료된 작업만 회수한다.
5. 실시간 수집은 희망 상태와 단조 증가 세대를 DB에 저장한다. 작업자는 연결을 주기적으로 정상 회전하고 연결 성공 뒤 자신이 읽은 정확한 `(target_spec_id, generation)`만 적용 완료로 표시한다.
6. 별도 메시지 브로커는 실제 PostgreSQL 병목과 복구 한계를 측정하기 전 도입하지 않는다.
7. 살아 있는 임대는 같은 worker instance도 중복 claim하지 않으며 진행·결과 쓰기는 현재 owner와 만료 전 lease를 확인한다. 실패 target만 재시도하고 성공 target·부분 진행은 보존한다.
8. 시도 예산이 남으면 `retry_wait`, 소진하면 `dead_letter`로 전이한다. 최종 실패는 미확인 coverage와 품질 이벤트에도 반영한다. 429·418은 업비트 공식 냉각 시간보다 빨리 재시도하지 않는다.
9. 운영 내구성의 단일 구현은 PostgreSQL이다. SQLite는 전환기 테스트 호환 어댑터이며 운영 대체 저장소로 취급하지 않는다.

## 대안

### 기존 상위 50개와 수동 시작 유지

저장량은 작지만 연구 유니버스에 현재 거래대금 편향이 생기고 신규 상장·재시작 복구에 숨은 수동 절차가 남는다.

### 프로세스 내부 큐와 WebSocket 동적 명령

구현은 단순하지만 프로세스 종료 시 희망 상태·적용 상태·진행률을 잃고 중복 작업을 안전하게 회수하기 어렵다.

### 메시지 브로커 선도입

구독·작업 분리는 쉬워지지만 현재 규모에서 운영 복잡도와 새로운 장애 경계만 늘어난다.

## 결과

- 대상 지정은 자동 백필과 실시간 희망 구독을 함께 만든다.
- 신규 KRW 시장은 자동 편입되고 명시적 제외는 유지된다.
- 작업자 강제 종료와 DB 재시작 뒤에도 DB 상태로 재개할 수 있다.
- 실시간 연결 회전은 최대 갱신 주기만큼 정책 반영 지연을 만들지만 연결 전 적용 완료 오판을 막는다.
- 모든 KRW의 2024년 이후 캔들은 큰 저장 공간을 요구하므로 P7에서 실제 행 크기, 인덱스, 디스크 여유, 처리율과 백업·복구 시간을 배포 게이트로 측정한다.

## 후속 조건

- `no_trade`는 빈 응답 하나가 아니라 추가 거래소 증거가 있을 때만 확정한다.
- PostgreSQL 임대 경합이나 실시간 부하가 측정 임계값을 넘으면 ADR-0012의 실행 기반 게이트에 따라 파티셔닝·큐·별도 저장소를 비교한다.
