# ADR-0017: 캔들 개정 원장과 버전 집계 계보

- 상태: Accepted
- 날짜: 2026-07-17
- 관련 Issue: [#29](https://github.com/goodjoon-company/goodmoneying/issues/29)
- 확장: ADR-0009의 집계 저장 결정

## 맥락

업비트가 같은 자연키의 과거 캔들 내용을 정정하거나 늦은 응답이 도착할 수 있다. 최신 행만 덮어쓰면 백테스트 시점에 알 수 있었던 값과 집계 입력을 재현할 수 없고, 집계 결과만 저장하면 어떤 원천 개정과 계산 규칙을 사용했는지 설명할 수 없다.

## 결정

- `source_candles`는 수집 시각상 최신 내용을 제공하는 조회 투영(projection)으로 유지한다.
- 모든 서로 다른 의미 내용은 같은 트랜잭션에서 `source_candle_revisions`에 추가 전용(append-only) 개정으로 기록한다. 같은 내용은 해시로 멱등 처리하고, 늦게 도착한 과거 변경도 원장에는 남기되 최신 투영을 퇴행시키지 않는다.
- 원천 내용 해시는 Decimal 표기 스케일과 DB 대리키(surrogate key)를 제외한 의미 값으로 계산한다.
- `candle_rollups`의 자연키에 `calculation_version`을 포함하고 입력 개정 ID, 입력 내용 해시, `source_as_of`, `knowledge_at`, 품질, 완전성을 저장한다.
- 집계 완전성은 행 수가 아니라 반개방 커버리지 구간을 함께 사용한다. `no_trade`는 빈 슬롯을 완전하게 만들 수 있지만 `missing`, `unavailable`, `unverified`와 판정되지 않은 슬롯은 불완전하다.
- 지원 단위는 `1m, 3m, 5m, 10m, 15m, 30m, 1h, 4h, 1d, 1w, 1M`이며 주·월 경계와 계산은 UTC·Decimal 기준이다.

## 대안

1. DB 트리거만으로 개정을 생성하면 애플리케이션이 같은 트랜잭션에서 신규 개정 목록을 후속 처리에 전달하기 어렵다.
2. 최신 투영을 뷰로 만들면 모든 읽기가 원장을 집계해야 해 현재 조회 경로와 운영 비용이 커진다.
3. 집계 결과를 계산 버전 없이 덮어쓰면 알고리즘 변경 전후 결과를 동시에 보존할 수 없다.

## 결과

- 009 이후 마이그레이션은 개정 원장과 버전별 집계 행을 보존하는 forward-only 계약이다.
- 런타임 역할은 개정 원장 `SELECT·INSERT`, identity 시퀀스 `USAGE`, 집계 테이블 `SELECT·INSERT·UPDATE`의 최소 권한이 필요하다.
- P2-2 변경 전파는 같은 트랜잭션에서 새로 생성된 개정 목록을 소비할 수 있다.
