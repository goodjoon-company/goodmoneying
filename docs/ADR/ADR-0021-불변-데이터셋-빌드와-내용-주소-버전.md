# ADR-0021: 불변 데이터셋 빌드와 내용 주소 버전

- 상태: Accepted
- 날짜: 2026-07-17
- 관련 Issue: [#29](https://github.com/goodjoon-company/goodmoneying/issues/29)
- 확장: ADR-0013의 시간·결측, ADR-0017~0020의 불변 입력 계보

## 맥락

백테스트와 전략 연구는 같은 데이터셋을 다시 읽을 때 원천 정정이나 신규 수집 뒤에도 같은 사건을 보아야 한다. 선택 조건만 해시한 레시피(recipe)는 이후 늦게 저장된 과거 개정과 품질 판정이 섞일 수 있고, 모든 값을 별도 복제하면 다중 시장·장기간 데이터의 저장 비용이 과도하다. 또한 큰 범위의 입력을 요청 트랜잭션에서 모두 해석하면 API 지연과 잠금 범위가 커진다.

## 결정

- 변경 가능한 생성 수명주기 `dataset_builds`와 게시 후 불변인 `dataset_versions`를 분리한다.
- 생성 명령을 수락하는 반복 읽기(REPEATABLE READ) 트랜잭션에서 입력·출력 UTC 반개방 범위, `asOf`, 상품, 데이터 종류·주기, 정의·계산 버전, 원천·품질·물질화·시장 상태 ID 상한과 정책을 즉시 고정한다. worker 대기 시간은 선택 프런티어를 바꾸지 않는다.
- worker는 고정된 상한과 `knowledge_at <= asOf`를 모두 적용해 정확한 입력 member를 선택하고, 최종 version·series·coverage·시장 상태 snapshot·typed member를 한 트랜잭션으로 게시한다.
- candle member는 원천 개정 또는 집계 물질화 중 정확히 하나를 참조한다. 지표, 시장 통계, 미시구조는 각 추가 전용 물질화를 FK로 참조한다. 대리키와 상한은 provenance와 늦은 삽입 차단에만 사용한다.
- 내용 해시는 대리키·생성 시각·표시 메타데이터를 제외한 정규화 JSON으로 계산한다. 시장 자연키, 범위, `asOf`, 데이터 종류·주기, 버전·정책, 시점별 시장 상태, 품질 snapshot, 정렬된 member의 내용 해시와 자연키가 내용 동일성을 구성한다.
- 같은 멱등 키와 같은 명령은 같은 build를 반환하고, 같은 키의 다른 명령은 충돌로 거부한다. 같은 의미 manifest는 내용 해시 고유 제약으로 같은 version을 재사용한다.
- 최종 의미 테이블은 UPDATE·DELETE를 DB trigger로 거부한다. 읽기 API는 저장된 정확한 member만 조회하고 `null`, `drop`, carry-forward point를 합성하거나 새 계산·쓰기를 수행하지 않는다.
- 가용성 정책은 `point-in-time-v1`로 고정한다. 데이터셋 member는 `occurred_at`과 `knowledge_at`을 보존하며, 백테스트 평가 시점 `t`는 두 시각이 모두 `t` 이하이고 당시 알려진 시장 상태가 거래 가능할 때만 입력을 볼 수 있다.
- 채움과 결측 처리를 분리한다. P2-5는 `none`과 확정 `no_trade` candle 전용 `no_trade_carry_forward_v1`, 결측 처리 `fail|null|drop`을 데이터셋 버전에 고정한다. `fail`은 불완전한 coverage의 게시를 차단하지만 완전한 `no_trade`는 허용한다. `null|drop|no_trade_carry_forward_v1`의 실제 적용은 P3 전략 검증과 P4 백테스트 소비 계층이 담당하며 데이터셋 member를 변경하지 않는다. `missing|unavailable|unverified`, 원시 호가와 미시구조에는 전방 채움을 허용하지 않는다.
- coverage snapshot은 원천 품질 사건이 있으면 `data_quality_events.detected_at`, 시장 비가용성이 있으면 그 상태의 `observed_at`, 둘 다 있으면 두 시각 중 늦은 값을 `knowledge_at`으로 쓴다. 명시적 증거가 없어 빌드가 관측 개수로 추론한 구간은 실제 추론 시각인 빌드 `frozen_at`을 사용하며 내용 해시에 포함한다. 후속 소비자는 평가 시점 `t`보다 늦게 알려진 품질 상태를 사용할 수 없다.
- 원시 호가 자체는 P2-5 데이터셋 종류에서 제외한다. 불변 미시구조 물질화와 그 기존 보존 핀만 참조한다.
- 큰 데이터셋은 트랜잭션 범위의 임시 테이블에 고정 선택을 스테이징하고 4,096행 서버 커서로 정규 JSON 배열 SHA-256을 증분 계산한다. 전역·시계열 해시는 동일한 대리키 제외 member 표현을 사용하고 typed member·시장 상태·coverage는 집합 기반으로 삽입한다.
- 긴 게시는 별도 연결 heartbeat가 120초 임대를 30초마다 연장한다. 갱신과 성공·재시도·실패 전이는 모두 상태·소유자·임대 세대·미만료 조건을 확인하며, 소유권을 잃으면 전체 게시 트랜잭션을 롤백한다. 예상 밖 실패는 제한된 지수 지연 재시도 뒤 dead-letter로 이동한다.

## 대안

1. 레시피만 해시하면 동일 URL과 조건이 늦은 과거 정정 뒤 다른 행을 선택한다.
2. 모든 결과 행을 별도 복제하면 내용은 명확하지만 저장량과 생성 시간이 입력 규모에 선형으로 중복된다.
3. worker 시작 시 프런티어를 고정하면 큐 대기 시간과 재시도 시점에 따라 결과가 달라진다.
4. 전역 `asOf`만 적용하면 해당 시각에 알려진 과거 정정을 더 이른 평가 시점에도 노출해 미래 참조를 만든다.
5. 현재 `coverage_intervals`와 `market_status_history.valid_to`를 직접 읽으면 변경 가능한 현재 투영이 게시된 version의 의미를 바꾼다.

## 결과

- 데이터셋 생성은 긴 작업이어도 요청 수락 시점의 프런티어를 재현한다.
- 이후 수집·정정·품질 변경은 기존 version의 member와 내용 해시를 바꾸지 않고 새 version으로만 나타난다.
- 백테스트는 데이터셋 내용 해시와 member별 지식 시각을 함께 사용해 결정론과 시점 인과성을 검증할 수 있다.
- P2-6 Data Lab은 서버가 계산한 build 상태·coverage·series를 표시하고, 완료 version 편집 대신 새 build 복제를 제공한다.
