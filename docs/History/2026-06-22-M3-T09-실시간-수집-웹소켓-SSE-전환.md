# 2026-06-22 M3-T09 실시간 수집 웹소켓(WebSocket) + SSE(Server-Sent Events) 전환

## 변경 요약

- 실시간 수집 live 프로필을 업비트 웹소켓(WebSocket) 기반 스트림 수집으로 전환하는 워커 모듈을 추가했다.
- `ticker`, `orderbook`, `candle.1m` 웹소켓 메시지를 기존 내부 모델(`TickerSnapshot`, `OrderbookSummary`, `SourceCandle`)로 변환하고 기존 `record_incremental_collection()` 저장 경로를 재사용한다.
- 운영 API에 `/v1/dashboard/summary/stream` SSE(Server-Sent Events) 엔드포인트를 추가했다.
- 운영 화면은 EventSource로 `dashboard` 이벤트를 받아 dashboard 상태를 갱신하고, 기존 React Query HTTP 폴링(Polling)을 보조 경로로 유지한다.
- 화면 갱신 기준 배지를 `SSE 실시간`으로 변경했다.
- 로컬 SQLite 저장소에서 SSE 요청과 일반 대시보드 요청이 같은 연결(connection)을 동시에 쓰지 않도록 대시보드 요약 계산을 잠금으로 직렬화했다.
- 웹소켓 스트림이 flush 주기 전에 예외로 종료되어도 버퍼의 마지막 시세를 저장한 뒤 예외를 전파하도록 보정했다.
- 웹소켓 flush 성공 시 Realtime worker heartbeat를 `running`으로 갱신해 실시간 수집 중 상태가 `지연`으로 표시되는 문제를 보정했다.
- 오늘 저장 Row Count와 저장량 breakdown에서 수집 결과 로그(`target_collection_results`)를 제외하고 캔들(Candle), 현재가(Ticker), 호가(Orderbook) 원천 시장 데이터만 표시하도록 정정했다.

## 리스크와 후속 작업

- 실제 업비트 웹소켓 live 연결 검증은 별도 운영/스테이징 환경에서 수행해야 한다.
- 웹소켓 재연결 backoff, 누락 이벤트 보정, 구독 재설정, 메시지 큐(Message Queue) 분리는 후속 Task가 필요하다.
- 캔들 웹소켓은 체결 기반 이벤트라 시간당 고정 기대 건수 계산과 맞지 않는다. 실시간 히트맵은 최신성(Freshness), 수신률(Receive Rate), 저장 지연(Persist Latency) 기준으로 재설계해야 한다.
