# P0-2 제품 의도 재확인과 원칙 정리

## 배경

GitHub Issue #4는 `docs/History/2026-07-04-제품-요구사항-로드맵-재정렬-계획.md`의 Task 2다. 현재 제품 문서는 최종 제품 비전을 넓게 담고 있지만 MVP(Minimum Viable Product)가 운영 콘솔 중심으로 읽히는 문제가 있었다.

## 결정

| 질문 | 결정 |
|---|---|
| 다음 1개월 목표 | 운영 안정화 자체가 아니라 투자 분석 화면으로 넘어가기 위한 데이터 기반과 관심 대상 탐색 흐름을 정리한다. |
| 업비트 수집 운영 콘솔의 위치 | 제품 중심 화면이 아니라 데이터 신뢰성을 확인하는 내부 운영 도구로 둔다. |
| MVP 성공 기준 | "데이터가 안정적으로 쌓인다"에서 멈추지 않고, 사용자가 관심 코인을 비교해 투자 후보를 좁힐 수 있는 상태를 성공 기준으로 둔다. |
| 코인/주식 확장 우선순위 | 가까운 MVP는 코인으로 끝까지 검증하고, 국내/미국 주식은 전략 실험 준비 단계 이후 별도 결정 게이트를 통과한 뒤 확장한다. |
| 호가 원천 스냅샷(Snapshot) 저장 | 당장 시작하지 않고 전략 실험 필요성이 확인될 때까지 보류한다. 현재는 호가 요약(Orderbook Summary)을 우선 유지한다. |

## 제품 원칙

- goodmoneying은 운영 콘솔이 아니라 개인 투자 의사결정을 돕는 데이터 기반 작업대다.
- 운영 화면은 데이터 신뢰성을 확인하는 내부 도구이며, 사용자 가치의 최종 화면은 관심 대상 탐색, 분석, 전략 실험이다.
- 가까운 로드맵(roadmap)은 1~2단계만 구체화하고, 이후 단계는 결정 게이트로 둔다.
- 요구사항은 구현된 사실, 확정된 제품 결정, 보류된 아이디어를 분리해 표시한다.
- 저장소와 화면은 "나중에 쓸 수도 있는 데이터"보다 "다음 투자 판단에 실제로 쓰는 데이터"를 먼저 드러낸다.

## 문서 반영

- `docs/01_Product.md`에 `제품 의도 결정`과 `제품 원칙`을 추가했다.
- 호가 원천 스냅샷(Snapshot) 보류 결정과 충돌하지 않도록 `GM-PROD-006`과 기존 M3 로드맵 문구를 조정했다.
- 계약(Contract), 데이터베이스(Database, DB), API(Application Programming Interface), 런타임 코드는 변경하지 않았다.
- 후속 P0-3은 이 결정을 기준으로 현재 구현 산출물을 계속 가져갈 것, 운영 도구로 격하할 것, 보류할 것, 삭제 후보로 분류한다.
