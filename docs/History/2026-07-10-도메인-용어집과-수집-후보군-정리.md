# 도메인 용어집과 수집 후보군 정리

## 변경 요약

- 루트 `UBIQUITOUS_LANGUAGE.md`를 도메인 용어 단일 기준(source of truth)으로 만들었다.
- 후보 유니버스(Candidate Universe)의 표준 용어를 수집 후보군(Collection Candidate Pool)으로 바꿨다.
- 시스템이 계산한 수집 후보군, 사용자가 선택한 관심목록(Watchlist), 워커가 처리하는 활성 수집 대상(Active Collection Target)을 별도 개념으로 정의했다.
- `CONTEXT.md`의 중복 용어 정의를 제거하고 용어집 링크와 사용 규칙만 남겼다.
- 제품, 아키텍처(Architecture), 모듈 설계, OpenAPI 설명, 아키텍처 결정 기록(ADR, Architecture Decision Record), 사용 설명서의 현재 표현을 표준 용어로 동기화했다.
- 루트와 문서 지도에서 용어집을 연결하고, 루트 설명을 초기 구현 이후 제품화(Post-MVP Productization) 기준으로 고쳤다.

## 결정

### 표준 용어

`수집 후보군`을 선택한 이유는 이 집합이 전체 투자 가능 자산 세계가 아니라, 현재 수집 대상으로 선택할 수 있도록 시스템이 계산한 상위 100개 집합이기 때문이다. `후보 유니버스`보다 범위와 목적이 이름에 직접 드러난다.

### 의미 경계

- 수집 후보군은 시스템 계산 결과다.
- 관심목록은 사용자의 비교·관찰 의도와 순서다.
- 활성 수집 대상은 수집 워커의 실행 상태다.
- 현재 구현에서는 관심목록과 활성 수집 대상이 같은 저장값을 공유하지만 미래에 비교와 수집을 분리할 수 있으므로 개념은 합치지 않는다.

### 호환성

기존 `/v1/candidate-universe`, `CandidateUniverseResponse`, `CandidateUniverseEntry`, `in_universe`, `out_of_universe`는 외부 계약과 코드 영향이 있는 식별자(identifier)이므로 유지한다. 이름 변경은 소비자 영향과 이행 계획을 갖춘 별도 계약 변경에서만 검토한다.

## 리뷰 결과

- 요구사항 적합성: Issue #9의 목표, 범위, 비범위, 완료 기준을 충족한다.
- 제품 적합성: `docs/01_Product.md`의 완료 기준선과 P0 범위를 새 용어와 Issue #9까지 동기화했다.
- 아키텍처 적합성: 데이터 흐름이나 모듈 경계는 바꾸지 않고 기존 세 개념의 관계를 명확히 했다.
- 계약 적합성: OpenAPI 설명만 변경했고 경로, 연산, 요청·응답 스키마, 열거형, DB 계약은 변경하지 않았다.
- 추적성: 제품(Product) P0 → GitHub Issue #9 → 용어집 → 테스트(Test) → 인계 이력(History) 링크를 연결했다.
- 보안·성능·운영 안정성: 런타임 변경이 없어 영향이 없다.
- 문서 품질: 용어 정의, 피할 별칭, 관계, 예시 대화, 남은 모호성을 포함하며 `CONTEXT.md`와 정의를 중복하지 않는다.

## 남은 위험과 후속 작업

- 운영 화면 문구와 일부 코드 오류 메시지는 기존 후보 유니버스 표현을 유지한다. 사용자 화면 문구를 표준화할 때는 코드·테스트를 먼저 수정하고 자동화된 종단간 시험(E2E Test)을 수행한다.
- 관심목록과 활성 수집 대상을 분리해야 하는 사용자 시나리오가 생기면 Product, API, DB 계약을 먼저 별도 설계한다.
- 수집 후보군 갱신 주기를 사용자에게 보장해야 할 때 제품 정책과 운영 서비스 수준 목표(SLO, Service Level Objective)를 확정한다.
- P0 최종 검증 Issue #8은 Issue #9의 용어집과 문서 동기화까지 포함해 실행한다.

## 롤백(Rollback)

문서 변경을 되돌려도 런타임 데이터와 API 호환성에는 영향이 없다. 표준 용어를 다시 바꿀 때는 용어집을 먼저 수정하고 현재 Product, Architecture, 계약 설명을 함께 동기화한다.
