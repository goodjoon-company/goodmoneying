# 로컬 E2E 격리 테스트 서버 복구

## 변경 요약

- Playwright 기본 API 서버가 루트 `.env`의 PostgreSQL을 사용하던 경로를 제거했다.
- `tests/e2e/seeded_api.py`에 명시적으로 seed된 SQLite 테스트 API 진입점을 추가했다.
- SSE(Server-Sent Events)와 쓰기 요청이 겹칠 때 SQLite 커서가 경쟁하지 않도록 테스트 전용 repository 프록시(proxy)에서 호출을 직렬화했다.
- API와 웹에 같은 `E2E_OPERATOR_TOKEN`을 전달하고 토큰·호스트·포트는 셸 문자열 보간 대신 Playwright `env` 속성을 사용한다.
- 후보 100개와 활성 수집 대상 50개를 실제 E2E API 응답에서 직접 검증한다.
- Playwright 성공 뒤 API·웹 시험 서버 포트가 닫혔는지 자동 검증한다.
- CI에 Chromium 의존성 설치와 `npm run e2e` 품질 게이트를 추가했다.
- 아키텍처 기준, README, M1 사용 설명서에 운영 런타임·로컬 격리 E2E·배포 환경 E2E의 경계를 기록했다.

## 결정

### 테스트 전용 직접 주입

런타임 `GOODMONEYING_DEMO_DATA=1` 경로를 되살리지 않는다. fixture는 `tests/e2e/seeded_api.py`가 `SQLiteOperationsRepository`에 직접 주입하며 제품 API 생성 함수에는 완성된 repository만 넘긴다.

### 공개 실행 인터페이스

개발자와 CI의 공개 진입점은 `npm run e2e` 하나다. Playwright가 API와 웹 서버의 시작·대기·종료를 책임지며 기본 포트는 API `18000`, 웹 `15173`이다. 이미 배포된 환경을 검증할 때만 `E2E_SKIP_WEBSERVER=1`과 외부 URL·토큰을 사용한다.

로컬 실행이 성공하면 종료 검증기가 두 포트의 연결 불가를 자동 확인한다. 배포 환경의 서버는 테스트가 소유하지 않으므로 `E2E_SKIP_WEBSERVER=1`에서는 종료 검증도 건너뛴다.

### SQLite 동시성

제품 repository 구현을 E2E 때문에 변경하지 않는다. 테스트 전용 프록시가 OperationsRepository 공개 호출 전체를 재진입 잠금(Reentrant Lock)으로 직렬화해 동기 endpoint와 SSE가 같은 SQLite 연결을 동시에 소비하지 않게 한다.

### CI 품질 게이트

CI는 단위·통합 테스트와 웹 빌드 뒤 Chromium E2E를 실행한다. 따라서 로컬에서만 동작하고 Linux CI에서 테스트 서버를 준비하지 못하는 회귀를 병합 전에 차단한다.

## 남은 위험

- SQLiteOperationsRepository 자체는 하나의 연결에서 execute와 fetch의 잠금 범위가 분리돼 있다. 이번 Issue는 테스트 프록시로 격리했으며, 제품 런타임은 PostgreSQL을 사용하므로 제품 repository 변경으로 확대하지 않았다.
- CI의 첫 Playwright 실행은 Chromium과 운영체제 의존성 설치로 기존보다 시간이 늘어난다. 30분 제한 안에서 실제 실행 시간을 관찰한다.
- Starlette의 `httpx` 사용 중단 예정 경고는 기존 비차단 경고로 남는다.

## 롤백(Rollback)

이 변경을 되돌리면 로컬 `npm run e2e`가 다시 루트 `.env`의 PostgreSQL에 의존한다. 롤백이 필요하면 CI E2E 단계와 격리 API·Playwright 설정·문서·검증 증적을 같은 커밋 단위로 되돌리고, 운영 E2E의 `E2E_SKIP_WEBSERVER=1` 경로는 유지한다.
