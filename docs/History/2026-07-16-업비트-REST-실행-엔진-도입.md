## 변경 요약

- 카탈로그 허용 목록 기반 REST 실행기에 HS512 JWT, 새 nonce, 입력 순서 쿼리 해시(query hash)를 추가했다.
- 공식 그룹별 요청 제한(rate limit), `Remaining-Req`, `Origin` 별도 제한, 429·418 냉각(cooldown)을 추가했다.
- `read`와 공식 주문 생성 테스트 `test`만 실행하고 실제 주문·취소·자산 이전·입출금·트래블룰 검증은 상향 호출 전에 차단한다.
- 고정 Upbit URL과 명시적 루프백(loopback) 테스트 URL만 허용하고 브라우저 `Origin`은 상향 전달하지 않는다.
- 키·JWT·Authorization·쿼리 해시(query hash)를 제거한 REST 추적 봉투(Trace Envelope)를 OpenAPI와 런타임에 동기화했다.
- `dev.sh`, Dockerfile, Compose, 환경 샘플, CI에 독립 `upbit-gateway` 런타임을 연결했다.

## 검증

실제 게이트웨이 프로세스와 가짜 Upbit HTTP 서버를 연결해 공개 조회, 인증 조회, 공식 주문 생성 테스트, 400·401·429·418 전달을 자동 검증했다. 차단된 실제 주문 경로의 가짜 상류 호출 수는 0회였다. 자세한 명령과 결과는 연결된 검증 증적을 따른다.

## 리스크

- Upbit 공식 인증·요청 제한 정책은 변경될 수 있으므로 후속 기능 작업 때 최신 공식 문서를 다시 확인해야 한다.
- 프로세스 메모리 제한기는 단일 게이트웨이 인스턴스 경계이며 다중 복제 운영은 공유 제한 저장소 설계가 필요하다.
- 418의 `Retry-After`가 없으면 보수적으로 60초 냉각하며 자동 재시도하지 않는다.

## 후속 작업

- 프론트엔드 REST 작업대는 Issue #21, WebSocket 중계는 Issue #22 이후 실행 단위에서 구현한다.
- 실제 상태 변경 기능은 후속 범위로 확장하지 않고 계속 `blocked`로 유지한다.
