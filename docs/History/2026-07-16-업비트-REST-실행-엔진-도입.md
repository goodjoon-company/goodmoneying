## 변경 요약

- 카탈로그 허용 목록 기반 REST 실행기에 HS512 JWT, 새 nonce, 배열·불리언(boolean)·입력 순서를 실제 URL과 동일하게 쓰는 단일 정규 쿼리 문자열(canonical query string) 및 쿼리 해시(query hash)를 추가했다.
- 공식 그룹별 요청 제한(rate limit), `Remaining-Req`, 429·418 냉각(cooldown)을 추가했다. 상향에서 제거한 브라우저 `Origin`에는 별도 1회/10초 제한을 적용하지 않고 실제 endpoint 그룹 한도를 사용한다.
- `read`와 공식 주문 생성 테스트 `test`만 실행하고 실제 주문·취소·자산 이전·입출금·트래블룰 검증은 상향 호출 전에 차단한다.
- 고정 Upbit URL과 명시적 루프백(loopback) 테스트 URL만 허용하고 브라우저 `Origin`은 상향 전달하지 않는다.
- 키·JWT·Authorization·쿼리 해시(query hash)를 제거한 REST 추적 봉투(Trace Envelope)를 모든 상향 상태와 OpenAPI·런타임에 동기화했다. 로컬 정책 오류와 상태가 겹치면 본문 스키마(schema)로 구분한다.
- `dev.sh`, Dockerfile, Compose, 환경 샘플, CI에 독립 `upbit-gateway` 런타임을 연결하고 읽기 전용 Compose 비밀정보(Secret) 추가 병합 파일을 제공했다.

## 검증

실제 게이트웨이 프로세스와 가짜 Upbit HTTP 서버를 연결해 공개 조회, 배열·불리언 인증 조회, 공식 주문 생성 테스트, 400·401·429·418 전달과 418 냉각을 자동 검증했다. 연속 브라우저 `Origin` 요청은 공식 시세 그룹 처리량을 유지했고, WebSocket 식별자는 422로 거부됐으며, 차단된 실제 주문 경로의 가짜 상류 호출 수는 0회였다. 응답과 프로세스 표준 출력·오류에 가짜 키가 없음을 확인했다. 자세한 명령과 결과는 연결된 검증 증적을 따른다.

## 리스크

- Upbit 공식 인증·요청 제한 정책은 변경될 수 있으므로 후속 기능 작업 때 최신 공식 문서를 다시 확인해야 한다.
- 프로세스 메모리 제한기는 단일 게이트웨이 인스턴스 경계이며 다중 복제 운영은 공유 제한 저장소 설계가 필요하다.
- 418의 헤더와 공식 JSON 본문에 유효한 기간이 없으면 보수적으로 60초 냉각하며 자동 재시도하지 않는다.
