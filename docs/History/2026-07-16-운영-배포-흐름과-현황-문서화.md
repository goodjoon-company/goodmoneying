# 운영 배포 흐름과 현황 문서화

날짜: 2026-07-16
관련 문서: `docs/03_Operations.md`
관련 검증: `docs/Test/2026-07-16-운영-배포-흐름과-현황-문서-검증.md`
관련 ADR(Architecture Decision Record): `docs/ADR/ADR-0004-prod-home-CICD와-배포-프로필.md`

## 변경 요약

- `docs/03_Operations.md`를 운영 토폴로지, 배포 흐름, 실제 운영 반영 버전과 상태의 단일 기준(source of truth)으로 추가했다.
- `docs/README.md`의 읽는 순서와 문서 책임 지도에 운영·배포 문서를 연결했다.
- 저장소의 최신 `main` 배포 정의와 현재 운영 `release-1db7d40`을 분리해 기록했다.
- 운영 API·Web·컨테이너·DB 접근·워커 heartbeat·최근 적재량을 실제로 조회해 시점별 현황으로 남겼다.
- 배포 실패, 수동 롤백, 상태 재확인 절차와 현재 차단 요인을 운영자 관점에서 정리했다.

## 현재 판단

- 현재 운영 서비스는 API·Web 응답과 실시간·백필(Backfill) 데이터 수집을 유지하고 있다.
- `main`은 운영 `release`보다 108커밋 앞서 있으나 최신 CI가 Web 시간대 테스트 실패 상태이므로 운영 승격 준비가 완료되지 않았다.
- Upbit Gateway, 캔들 집계 워커, dbmate migration은 최신 `main` 배포 정의에는 있으나 현재 운영에는 반영되지 않았다.

## 남은 위험

- 현재 장비에서 Mac Mini M4 SSH가 시간 초과되어 공식 healthcheck 전체를 통과시키지 못했다.
- healthcheck가 캔들 집계 워커와 실제 데이터 증가량을 자동 확인하지 않는다.
- ADR-0004와 배포 프로필 설명의 `release-{short-sha}` 표현이 최신 `main` 워크플로의 전체 SHA 태그와 다르다.
- 배포 후 실패에 대한 자동 롤백과 DB migration 하향 복구가 없다.
- `main`과 `release`의 큰 차이를 한 번에 승격하면 변경 범위와 migration 위험이 크므로 배포 전 별도 검토가 필요하다.
