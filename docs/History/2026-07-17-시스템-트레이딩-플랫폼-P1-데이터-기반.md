# 시스템 트레이딩 플랫폼 P1 데이터 기반

날짜: 2026-07-17
GitHub Issue: [#28](https://github.com/goodjoon-company/goodmoneying/issues/28)

## 변경 이유

기존 상위 100 후보·활성 50 대상·수동 백필 흐름은 현재 유니버스 편향과 재시작 복구 공백을 만든다. P1은 공식 시장 이력과 정책을 권위 상태로 바꾸고 대상 선택이 과거·실시간 데이터 준비로 자동 이어지게 한다.

## 변경 요약

- 확장 전용 dbmate migration으로 시장·상태 이력·정책·대상 사양·fetch manifest·5상태 coverage·품질 이벤트·희망 구독·내구성 백필 필드를 추가했다.
- 기존 instrument·target·원천 행을 삭제하지 않고 새 식별자와 provenance를 nullable 연결해 무손실 전환 경계를 유지했다.
- 시장 동기화 worker와 Data Foundation API, Coverage & Quality 화면을 추가했다.
- 실시간 worker가 DB 구독 세대를 읽어 연결을 정상 회전하고 정확 세대만 적용 완료로 기록하도록 바꿨다.
- 백필은 살아 있는 임대 중복 claim 금지, owner·expiry 쓰기 검증, 부분 진행 보존, 429·418 냉각 시간, 실패 target만 재시도, 예산 소진 격리와 품질 전이를 구현했다.
- 원천 캔들·체결·호가·티커의 실제 저장 경로가 체크섬 manifest와 사건·수신·저장·지식 시각을 남기며, 실제 자연키 행이 저장된 구간만 `available`로 만든다.
- 웹소켓 원본 프레임은 연결·순번별 수신 영수증으로 append-only 저장하고, 동일 경제 상태의 호가 스냅숏과 전체 30레벨을 분리해 기존 10레벨 요약과 한 트랜잭션에서 기록한다.
- 백필 페이지는 실제 endpoint·query·요청/응답 시각·상태·원본 JSON을 manifest로 보존하며, 응답 없는 전송 오류도 실제 cursor와 오류 유형을 남긴다.
- coverage는 `available|no_trade|missing|unavailable|unverified`만 사용하고 모든 생성·전이를 이전 상태·새 상태·범위·사유·manifest가 있는 품질 이벤트로 기록한다.

## 결정과 호환성

- PostgreSQL이 운영 조정의 단일 기준이다. SQLite 상태 기계는 기존 빠른 테스트용 호환 어댑터로 남기며 운영 parity를 주장하지 않는다.
- 자동 경로는 `leased`를 중간 저장하지 않고 claim과 함께 `running`으로 전이한다. `planned|paused|stopped|failed`는 수동 운영·전환기 API 호환 상태다.
- 429는 다음 초 경계보다 빠르게 재시도하지 않는다. 418은 응답의 차단 기간을 따르고 기간이 없으면 5분을 사용한다.
- 단순 빈 캔들 응답은 `no_trade` 증거가 아니므로 성공 manifest만 남기고 coverage를 추정하지 않는다.
- 같은 성공 페이지의 양쪽 인접 캔들로 완전히 경계된 내부 분 공백만 `no_trade`로 확정한다.

## 남은 위험과 P2·P7 후속

- 실제 KRW 전체 장기 백필의 행 크기·인덱스·처리량·디스크·백업 시간은 P7 배포 게이트에서 운영 규모로 측정한다.
- available coverage는 요청 batch 단위로 압축되지만 장기간 누적 인접 interval 병합은 P7 성능 측정 후 최적화한다.
- P2는 원천 계약을 변경하지 않고 다중 주기 집계, 지표, 데이터셋 버전과 내부 WebSocket 복구 계약을 추가한다.
