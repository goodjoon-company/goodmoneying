# P4-4 백테스트 run 목록 cursor 인계

날짜: 2026-07-18
추적: GitHub Issue #31
관련 검증: `docs/Test/2026-07-18-P4-4-백테스트-run-목록-cursor-검증.md`

## 변경 요약

- `GET /v1/backtest-runs` REST API를 추가했다.
- 목록 응답은 상세 `BacktestRun`이 아니라 `BacktestRunSummary` 배열과 `nextCursor`를 반환한다.
- `PostgresBacktestStore.list_runs()`는 `id DESC` keyset pagination을 사용한다.
- cursor는 `backtest-run-list-v1` 문맥, 첫 페이지 최대 ID 상한(`ceiling`), 마지막 ID(`lastId`), HMAC-SHA-256 digest를 담는 불투명 값이다.
- cursor 변조·문맥 불일치는 `BacktestCursorMismatchError`와 `BACKTEST_CURSOR_CONTEXT_MISMATCH` 409로 수렴한다.
- Backtest Lab은 하드코딩된 run ID 없이 저장된 run 목록을 먼저 읽고 첫 run을 자동 선택한다.
- `nextCursor`가 있으면 “다음 run 목록 불러오기”로 다음 페이지를 누적 로드한다.
- 목록 선택 후 상세 성과·체결·산출물은 기존 단건 조회 API로 읽는다.
- seeded API와 Web test fixture에 백테스트 run 목록 fixture를 추가했다.

## 설계 결정

- 목록은 체결·equity·artifact payload를 포함하지 않는다. Backtest run 결과는 커질 수 있으므로 목록은 발견(discovery)과 선택에 필요한 summary만 담당한다.
- 정렬은 현재 P4 최소 범위에서 `id DESC`로 고정했다. DB migration에는 `(created_at DESC, id DESC)` 인덱스가 있지만, 첫 페이지 상한 고정과 단순한 재개 cursor를 우선했다. created_at 기반 정렬이 필요해지면 tuple cursor로 확장한다.
- cursor digest는 공개 checksum이 아니라 HMAC-SHA-256이다. `GOODMONEYING_CURSOR_HMAC_SECRET`이 없으면 프로세스 시작 시 생성한 비밀로 서명하므로 재시작 전 cursor만 유효하다.
- DB status `queued`는 API status의 `pending`으로 매핑한다. DB 내부 큐 의미와 외부 제품 상태 이름을 분리하기 위해서다.
