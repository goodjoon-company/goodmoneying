## 범위

- 포함: 수집 대상/설정 화면에서 체크된 코인으로 백필 계획 생성 버튼을 제공한다.
- 포함: 백필 계획 생성 시 수집 범위와 백필 옵션을 입력하는 레이어 팝업(layer popup)을 제공한다.
- 포함: 백필 계획 생성 결과를 백필 승인 패널에 계획 목록으로 표시한다.
- 포함: 계획 추가/삭제 후 백필 계획 승인 버튼을 활성화한다.
- 포함: 승인 시 기존 `POST /v1/backfill/jobs` 계약으로 계획별 백필 작업(Backfill Job)을 만든다.
- 포함: 워커(Worker)는 이미 저장된 1분 캔들(Source Candle) 앞 구간을 중복 요청하지 않고 첫 빈 구간부터 백필한다.

## 비범위

- DB/API 계약의 새 엔드포인트 추가
- 삭제 후 재수집(Destructive Rebuild)
- 다중 워커(Message Queue) 기반 작업 분배
- 실제 업비트(Upbit) 라이브 백필 장기 실행

## 현재 맥락

기존 UI는 운영 상태 대시보드의 코인 행 상세에서 단일 백필 계획을 만들 수 있었다. 사용자는 백필 데이터 수집 흐름을 `백필 후보 선택 -> 백필 계획 생성 -> 백필 계획 목록 구성 -> 백필 계획 승인 -> 워커 지속 수집`으로 정리했고, 백필 계획 생성과 승인은 수집 대상/설정 화면에 있어야 한다고 요구했다.

## 설계 메모

- 기존 OpenAPI(OpenAPI) 계약은 `POST /v1/backfill/plans`와 `POST /v1/backfill/jobs`가 있어 새 엔드포인트 없이 화면 흐름을 재배치한다.
- 프론트엔드(Frontend)의 백필 승인 패널은 승인 전 계획 목록을 로컬 UI 상태로 구성하고, 승인 시 계획별 API 호출을 수행한다.
- 워커는 `source_candles`의 유니크 키 `(instrument_id, source, candle_unit, candle_start_at)`에만 의존하지 않고, 요청 전에 기존 캔들 시작 시각을 확인해 API 요청 시작점을 조정한다.
- 기간이 조정되면 목표 시작일부터 재검사하되, 시작일 데이터가 있으면 그 뒤 첫 빈 1분봉부터 요청한다.

## 실패 케이스

- 운영 상태 화면에서 백필 계획 생성 액션이 남아 사용 흐름이 분산됨
- 선택된 코인이 없는데 백필 계획 생성 버튼이 활성화됨
- 백필 계획 생성 후 승인 패널에 계획 목록이 표시되지 않음
- 계획 삭제 후 승인 버튼 상태가 갱신되지 않음
- 승인 후 워커가 이미 저장된 시작 구간을 다시 API로 요청함
- 시작일부터 전체 재검사를 해야 하는데 기존 데이터 이후 첫 빈 구간을 찾지 못함

## 완료 기준

- 백필 계획 생성 버튼은 수집 대상/설정 화면에서 선택 코인 기준으로 동작한다.
- 백필 계획 생성 레이어에서 수집 범위 시작/종료를 설정할 수 있다.
- 생성된 계획은 백필 승인 패널 목록에 표시된다.
- 계획 목록이 있을 때만 백필 계획 승인 버튼이 활성화된다.
- 승인하면 계획 기반 백필 작업이 생성된다.
- 워커는 이미 저장된 기간을 중복 요청하지 않고 첫 빈 구간부터 요청한다.
