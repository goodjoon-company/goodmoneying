## 범위

- 포함: 일반 API 실행에서 fixture seed가 자동으로 실제 API 데이터처럼 섞이지 않게 한다.
- 포함: `qualityStatus`, `qualityDetail`, `collectionRangeDisplay`를 실제 수집 대상/계획/커버리지 상태에서 만든다.
- 포함: `coveragePercent`, `missingSegmentCount`, `coverageSegments`를 실제 저장된 버킷(bucket)과 목표 범위 기준으로 계산한다.
- 포함: 계산 비용이 큰 커버리지 결과는 저장 시점 또는 배치 계산 결과를 조회할 수 있도록 저장소(repository) 경계에 반영한다.
- 포함: 실제 근거가 없는 `rateLimitRemainingPercent` UI 표시는 제거한다.
- 포함: `storageBytesToday`는 실제 오늘 적재된 행 기반의 추정 저장량으로 의미를 좁힌다.
- 포함: `duplicateRows24h`는 유니크 키(unique key)로 중복 저장이 불가능한 현 구조에서 표시하지 않는다.

## 비범위

- 업비트(Upbit) rate limit 헤더 영속화와 상세 관측 테이블 추가
- PostgreSQL 물리 저장 용량을 `pg_total_relation_size`로 산출하는 운영 전용 최적화
- 대규모 머티리얼라이즈드 뷰(Materialized View) 도입
- 다중 워커(Message Queue) 기반 분산 계산

## 현재 맥락

코드 리뷰 결과 일부 운영 UI 값이 실제 DB 값이 아니라 fixture, rank 기반 휴리스틱(heuristic), 고정 문자열, 고정 숫자, row size 추정값으로 표시되고 있었다. 사용자는 합성 지표를 제거하고 계산되지 않은 값은 실제 계산하도록 요구했으며, 오래 걸리는 계산은 입력 시점/배치 계산/DB view를 활용해 빠르게 응답하길 원했다.

## 설계 메모

- 승인된 방향은 B안: 저장 시점/배치 계산 + 조회 전용 API다.
- SQLite 테스트 저장소와 PostgreSQL 저장소 모두 같은 의미의 지표를 반환해야 한다.
- 커버리지 계산은 목표 수집 범위의 1분 버킷 대비 저장된 `source_candles` 개수와 비어 있는 연속 구간으로 계산한다.
- ticker/orderbook 품질은 최신 성공 시각과 실제 저장 행 유무로 계산한다.
- 후보 유니버스의 품질 표시는 수집 대상이면 실제 커버리지 상태를 요약하고, 비수집 대상이면 수집 계획 없음으로 표시한다.

## 계약 변경

- `DashboardTotals.rateLimitRemainingPercent`는 실제 수집 요청 헤더 영속화가 없으므로 UI 표시 대상에서 제외한다.
- `InstrumentDetail.duplicateRows24h`는 현재 유니크 키로 중복 저장을 허용하지 않으므로 UI 표시 대상에서 제외한다.
- `storageBytesToday`는 오늘 생성된 원천/결과 행 기반 추정 저장량으로 정의한다.

## 실패 케이스

- API가 DB 설정 없이 fixture 데이터를 실제 API 데이터처럼 반환한다.
- 커버리지 값이 저장된 캔들 개수와 목표 범위를 반영하지 않는다.
- 결측 세그먼트가 실제 빈 1분 버킷과 무관한 고정 비율로 표시된다.
- 후보 유니버스 품질이 거래대금 순위만으로 정상/주의를 만든다.
- 실제 계산 근거가 없는 rate limit 여유율 또는 중복 행 수가 UI에 남아 있다.
