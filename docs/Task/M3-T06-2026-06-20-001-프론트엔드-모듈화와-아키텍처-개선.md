## 범위

- OpenAPI 계약을 따르는 프론트엔드 운영 데이터 접근 Module을 만든다.
- HTTP Adapter와 fixture Adapter를 같은 Interface 뒤에 둔다.
- 후보 유니버스(Candidate Universe), 활성 수집 대상(Active Collection Target), 백필 계획(Backfill Plan), 백필 작업(Backfill Job) 흐름을 수집 대상/백필 workflow Module로 모은다.
- 운영 화면 View Model 표시 계산을 `App.tsx`에서 분리한다.
- 운영 화면 shell의 스냅샷, 상세 레이어, fixture 분기 상태 처리를 별도 Module로 옮긴다.
- 테스트가 CSS class Implementation에 과도하게 묶인 부분을 role/label 중심 Interface로 정리한다.

## 비범위

- React Query와 HTTP 폴링(Polling) 결정을 바꾸지 않는다.
- OpenAPI 계약 자체의 필드 의미를 바꾸지 않는다.
- 백엔드(Backend) DB/API 동작을 새로 설계하지 않는다.
- 디자인 시각 스타일을 새로 변경하지 않는다.
- 차트 라이브러리 Adapter를 새로 추상화하지 않는다. 현재는 One adapter라서 명확한 Seam이 아니다.

## 현재 맥락

현재 브랜치 `codex/design_mod_v0.1`에는 운영 콘솔 디자인/계약 정렬 작업이 미커밋 상태로 남아 있다. 이번 작업은 같은 브랜치의 기존 변경 위에서 이어서 수행한다.

아키텍처 리뷰에서 확인된 주요 마찰은 다음과 같다.

- `apps/web/src/App.tsx`가 1,670줄이며 화면 shell, 대시보드, 수집 대상/백필, 시장 리스트, 상세 모달, 차트, 표시 포맷을 한 파일에 담고 있다.
- `apps/web/src/api.ts`가 840줄이며 계약 타입, HTTP 호출, 정규화, 쓰기 토큰, fixture 데이터를 한 파일에 담고 있다.
- `apps/web/src/styles.css`가 2,176줄이며 테스트 일부가 CSS class Implementation을 직접 참조한다.
- 제품/아키텍처 기준은 프론트엔드 계산 최소화와 운영 서버(Operations Server)의 화면용 View Model 제공을 요구한다.

## 설계 메모

- Module은 호출자가 알아야 하는 Interface를 줄이고, Implementation 안에 동작을 집중시켜 Depth를 높인다.
- `api.ts`는 삭제 테스트 기준으로 유효한 Module이지만 책임이 넓다. 데이터 접근 Interface와 Adapter를 분리하면 Locality와 Leverage가 좋아진다.
- HTTP Adapter와 fixture Adapter가 이미 있으므로 운영 데이터 접근은 real Seam이다.
- 수집 대상/백필 workflow는 새 외부 Adapter보다 순수 상태 전이 Module이 적합하다.
- 표시 계산 Module은 백엔드 View Model과 충돌하는 프론트엔드 계산을 찾는 검사 지점이 되어야 한다.
- 테스트 selector는 사용자 행동 Interface와 디자인 계약 검증을 구분한다.

## 실패 케이스

- 운영 상태 첫 화면이 대시보드 요약과 백필 작업만으로 빠르게 렌더링되지 않는다.
- fixture Adapter가 실제 API 데이터처럼 운영 코드 경로에 섞인다.
- 활성 수집 대상 최대 50개 제한이 화면 이벤트에만 숨어 별도 테스트가 어렵다.
- 백필 계획 생성/삭제/승인 순서가 DOM 통합 테스트로만 검증된다.
- 표시 시간 KST/UTC, 퍼센트, 저장량 fallback이 여러 화면에서 서로 다르게 계산된다.
- E2E(End-to-End) 테스트가 CSS class 변경만으로 깨진다.
