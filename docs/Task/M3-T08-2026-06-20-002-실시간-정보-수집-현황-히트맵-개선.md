# 실시간 정보 수집 현황 Heatmap(Heat Map) 구현 계획

I'm using the writing-plans skill to create the implementation plan.

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** `#root > main > section.dashboard-page > div.ops-kpi-grid > section.panel.ops-activity-card`에 표시되는 `정상 수집` 히트맵을, 최근 24시간 기준 최대 50개 관심 코인별 실시간 수집 대상(가격/체결/오더북) 예상수집량 대비 수집량으로 색상 구분되는 **실시간 정보 수집 현황(Realtime Collection Heatmap)** 카드로 전환한다.

**Architecture:** 백엔드 View Model을 source-of-truth로 확장해 (PostgreSQL/SQLite 동일 경로) `DashboardSummary`에 `realtimeCollectionHeatmap`을 추가하고, 프런트엔드는 받아온 값만 렌더링한다. 기존 `collectionActivity`(7일×24)는 하위 호환 유지한다.

**Tech Stack:** FastAPI, Pydantic, SQLite/PostgreSQL, Vitest, Playwright, OpenAPI, React + TypeScript.

`packages/shared/goodmoneying_shared/models.py`에 새 데이터 모델을 추가한다.

```python
@dataclass(frozen=True)
class RealtimeCollectionHeatmapBucket:
    bucket_start_at: datetime
    expected_rows_all: int
    actual_rows_all: int
    expected_rows_by_type: dict[str, int]
    actual_rows_by_type: dict[str, int]
    actual_ratio_percent: Decimal
    status: Literal["none", "low", "collecting", "high"]


@dataclass(frozen=True)
class RealtimeCollectionHeatmapRow:
    instrument: Instrument
    instrument_display_name: str
    hourly_buckets: list[RealtimeCollectionHeatmapBucket]
```

`DashboardSummary`에 `realtime_collection_heatmap: list[RealtimeCollectionHeatmapRow]`를 추가하고, `dashboard_summary()`에서 기본값을 빈 리스트로 초기화하지 않고 실제 집계값으로 채우도록 연결한다.

- PostgreSQL/SQLite에서 공통적으로 `dashboard_summary()`가 호출되는 두 구현부(`_collection_activity_buckets`와 병행)를 참조해, 아래 방식으로 히트맵을 별도 메서드로 분리해 추가한다.
  - 기준 대상: `list_active_targets()` 상위 50개(활성 대상)
  - 시간대: `now_utc() - 24h`부터 24개 시간 버킷
  - 수집 데이터: `source_candles`, `ticker_snapshots`, `orderbook_summaries`
  - ratio 계산: `(actual_rows_all / expected_rows_all) * 100`, 0일 때 0
  - 기존 `status` 규칙 재사용(`none/low/collecting/high`)
  - 기대값(`expected_rows_*`)은 데이터타입별 기대 수집 간격 기반(1분 단위 기반이면 60, 필요한 경우 보정값 테이블 분리)
  - 반환: 대상별 24개 버킷 정렬
