## 범위

- 공유 시간 유틸리티(shared time utility)를 KST 기준으로 전환
- PostgreSQL(PostgreSQL) 연결 세션과 DB 기본 시간대를 KST로 설정
- Docker Compose와 Dockerfile 런타임 시간대를 KST로 고정
- Docker 이미지에 timezone database(`tzdata`) 설치 명시
- 로컬 개발 스크립트(`dev.sh`) 실행 프로세스의 `TZ`, `PGTZ`를 KST로 설정
- 백필 계획 생성 화면 입력을 KST 기준 payload로 변환
- API 계약과 제품/아키텍처 문서의 시간대 정책 갱신

## 비범위

- 업비트(Upbit) 외부 API의 `to` 파라미터 프로토콜 변경
- 기존 운영 데이터의 물리적 재작성
- DB 타입(`TIMESTAMPTZ`) 제거

## 설계 메모

- `TIMESTAMPTZ`는 절대 시점을 보존하는 PostgreSQL 타입으로 유지한다. 대신 세션 timezone과 DB 기본 timezone을 KST로 고정한다.
- 애플리케이션 모델과 API payload는 `+09:00` 오프셋을 가진 ISO 문자열을 사용한다.
- 업비트 외부 API는 UTC(Coordinated Universal Time) `to` 값을 요구하므로, provider boundary에서만 UTC 변환을 유지하고 내부 row는 KST로 변환한다.
- `datetime-local` 입력은 시간대 정보를 갖지 않으므로 화면에서는 KST 입력으로 명시하고 API 요청에는 `+09:00`을 붙인다.

## 계약 변경

- `docs/contracts/db/schema.sql`에 `SET TIME ZONE 'Asia/Seoul'`, `ALTER DATABASE goodmoneying SET timezone TO 'Asia/Seoul'` 추가
- `CollectionPlan.rangeTimeZone` enum을 `KST` 단일값으로 축소
- 저장량/행 수의 “오늘” 기준 설명을 KST 오늘로 변경

## 실패 케이스

- 컨테이너 내부 `date`가 UTC로 나와 로그 기준과 화면 기준이 다르다.
- DB `SHOW TIMEZONE`이 UTC로 남아 `now()`와 일 단위 집계가 KST 오늘과 어긋난다.
- 백필 계획 생성 화면은 KST라고 보이지만 API에는 `Z` 값을 보내 범위가 9시간 밀린다.
