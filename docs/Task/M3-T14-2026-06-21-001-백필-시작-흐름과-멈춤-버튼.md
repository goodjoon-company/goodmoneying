# M3-T14 백필 시작 흐름과 멈춤 버튼

Status: Done
Owner: Codex
Created: 2026-06-21
Updated: 2026-06-21

## 목적

수집 대상/설정 화면의 백필 계획 생성 팝업에서 사용자가 확인 버튼을 누른 뒤 별도 승인 버튼을 다시 누르는 흐름을 제거한다. 팝업의 기본 실행 버튼은 `백필 시작`으로 표시하고, 클릭 즉시 백필 작업(Backfill Job)을 `pending` 상태로 저장해 백필 수집 워커(Backfill Collection Worker)가 DB 생성 순서대로 자동 실행하도록 한다.

## 요구사항

- 백필 계획 생성 팝업의 `확인` 버튼을 `백필 시작` 버튼으로 변경한다.
- `백필 시작` 클릭 시 선택 코인, 데이터 유형, 시작/종료 일시를 서버로 전송해 `pending` 백필 작업을 생성한다.
- 백필 계획 승인 버튼은 제거한다.
- 백필 작업 패널에는 멈춤(Pause), 중지(Stop), 삭제(Delete) 제어를 제공한다.
- 백필 수집 워커는 저장된 `pending` 작업을 DB 생성 순서대로 점유해 실행한다.

## 설계 기준

- 제품(Product): `docs/01_Product.md`의 GM-PROD-023은 백필 후보 선택, 백필 계획 생성 팝업, 백필 시작, `pending` 작업 저장, 워커 지속 수집 흐름을 기준으로 한다.
- 아키텍처(Architecture): `docs/02_Architecture.md`와 `docs/02_Architecture/upbit-collection-pipeline.md`는 백필 수집 워커가 DB 상태 폴링(Polling)으로 `pending` 작업을 읽는 구조를 기준으로 한다.
- API 계약(Contract): `POST /v1/backfill/jobs`는 `CreateBackfillJobRequest`를 받아 `pending` 백필 작업을 반환한다.

## 구현 메모

- 프론트엔드(Frontend)는 더 이상 로컬 승인 대기 목록을 만들지 않고 `startBackfillJob` API를 직접 호출한다.
- 운영 서버(Operations Server)는 요청을 받아 기존 백필 계획 저장 경로를 재사용하되 즉시 `pending` 작업으로 전환해 반환한다.
- 저장소(Repository)의 `claim_next_backfill_job`은 `pending` 작업을 `created_at` 오름차순으로 점유하므로 저장 순서 실행 요구를 만족한다.
- `pause` 제어는 `pending` 또는 `running` 작업에만 노출한다.
