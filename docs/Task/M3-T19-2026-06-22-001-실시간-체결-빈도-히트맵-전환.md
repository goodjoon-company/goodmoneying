## 범위

- 업비트 웹소켓(WebSocket) `trade` 구독 추가
- `trade_events` 원천 테이블과 저장소(repository) 기록 경로 추가
- 최근 24시간, 관심 코인 최대 50개 기준 시간당 체결 빈도 집계
- 분당 평균 체결 빈도별 색상 단계 적용: 10건 미만 빨강, 10~50 미만 주황, 50~100 미만 노랑, 100~200 미만 파랑, 200 이상 초록
- 마우스 오버 상세에 시간 평균 체결빈도, 체결강도, 체결량, 체결금액 표시

## 비범위

- 주문 체결(MyOrder) private 웹소켓(WebSocket) 구독
- 호가 원천 스냅샷 전체 저장
- 체결 이벤트 장기 보존 정책과 파티셔닝(Partitioning) 확정
- 별도 실시간 패널 증분 이벤트 프로토콜

## 현재 맥락

기존 `realtimeCollectionHeatmap`은 `source_candles`, `ticker_snapshots`, `orderbook_summaries`의 시간당 기대 row 대비 실제 row 비율을 표시했다. 새 요구사항은 같은 화면 영역을 체결 이벤트 빈도 관측으로 바꾸는 것이다.

## 설계 메모

- API 필드명 `realtimeCollectionHeatmap`은 유지하되 셀 계약을 체결 빈도 지표로 변경한다.
- 체결강도는 `BID 체결량 / ASK 체결량 * 100`으로 계산하며, ASK 체결량이 0이면 0으로 둔다.
- 웹소켓 구독은 공개 시세(Quotation) `trade` 타입을 사용한다. 내 주문 체결은 범위가 아니다.
- 중복 체결 이벤트는 `(instrument_id, source, sequential_id)` 유니크 키로 차단한다.

## 계약 링크

- DB: `docs/contracts/db/schema.sql`

## 계약 변경

- DB: `trade_events` 테이블 추가, `collection_runs`/`target_collection_results`의 `trade_event` data type 허용
- API: `RealtimeCollectionHeatmapCell`을 `tradeCount`, `averageTradesPerMinute`, `tradeStrength`, `tradeVolume`, `tradeAmount`, `status` 구조로 변경

## 실패 케이스

- `trade` 이벤트가 구독되지 않으면 체결 빈도 히트맵이 계속 0건으로 표시된다.
- 중복 `sequential_id`를 저장하면 체결 빈도가 과대 집계된다.
- 색상 임계값이 분당 평균 기준이 아니면 고빈도 코인의 체결 밀도를 잘못 표현한다.
- ASK 체결량이 0일 때 체결강도 계산에서 division by zero가 발생할 수 있다.
