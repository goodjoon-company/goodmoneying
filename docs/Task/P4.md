# P4 재현 가능한 백테스트 요약

상태: 진행 중
추적: GitHub Issue #31
시작일: 2026-07-18

## 목표

P4는 같은 데이터셋·전략·엔진 버전·파라미터·seed가 같은 결과를 반환하는 백테스트 체계를 만든다. 연구, 백테스트, paper·shadow·live-ready 단계는 같은 전략 평가 의미와 주문 체결 의미를 공유해야 한다.

## P4-1 완료 범위

- `backtest-core-v1` 순수 백테스트 엔진 추가
- dataset content hash, strategy graph hash, parameter hash, deterministic seed 기반 입력 hash 고정
- `(knowledge_at, source_priority, stable_sequence)` 사건 정렬
- wall clock과 직접 난수 사용 금지
- Decimal 기반 수수료·슬리피지(slippage)·지연(latency)·부분 체결(partial fill) 계산
- `orderbook_absent_uses_candle_close`, `partial_fill_by_candle_volume_participation` 가정 기록
- golden replay 신호 동등성 검증

## P4-2 완료 범위

- `backtest_runs`, `backtest_trades`, `backtest_equity_points`, `backtest_metrics`, `backtest_artifacts` DB 계약 추가
- `input_hash`, `result_hash`, dataset content hash, strategy graph hash 영속화
- published 전략 버전과 sealed 데이터셋 버전만 run 입력으로 허용
- terminal run과 결과 행 append-only 봉인
- `PostgresBacktestStore` completed run 저장·조회·멱등 재생 구현
- dbmate migration E2E와 live PostgreSQL Backtest Store E2E 통과

## P4-3 완료 범위

- 저장된 백테스트 run을 조회하는 `GET /v1/backtest-runs/{backtestRunId}` REST 계약 추가
- `BacktestRun`, `BacktestMetric`, `BacktestTrade`, `BacktestArtifact` OpenAPI 스키마 추가
- FastAPI가 Backtest Store 조회 결과를 읽기 전용으로 반환하고 없는 run은 `BACKTEST_RUN_NOT_FOUND` 404로 수렴
- `PostgresBacktestStore.get_run()` 조회 응답에 체결 순번, 발생·지식 시각, artifact media type 보존
- Web API client의 `loadBacktestRun()`과 Backtest Lab UI 추가
- Operations Console에 Backtest Lab 메뉴와 `REST 조회` 갱신 기준 추가
- seeded E2E API와 브라우저 E2E에서 `Run #21`, `finalEquity`, 체결, artifact를 읽기 전용으로 확인

## P4-4 완료 범위

- 저장된 백테스트 run 목록을 조회하는 `GET /v1/backtest-runs` REST 계약 추가
- `BacktestRunSummary`, `BacktestRuns` OpenAPI 스키마 추가
- Backtest Store 목록은 offset이 아니라 ID 상한(`ceiling`)과 마지막 ID(`lastId`)를 담은 불투명 커서(opaque cursor)를 사용한다.
- 첫 페이지 이후 새 run이 삽입돼도 다음 페이지에 섞이지 않도록 첫 페이지의 최대 ID를 cursor에 고정한다.
- cursor envelope는 `backtest-run-list-v1` 문맥과 HMAC-SHA-256 digest로 변조·문맥 불일치를 거부한다.
- FastAPI가 cursor 불일치를 `BACKTEST_CURSOR_CONTEXT_MISMATCH` 409로 수렴한다.
- Backtest Lab은 하드코딩된 run ID 대신 저장된 run 목록을 먼저 읽고 첫 run을 자동 선택하며, `nextCursor`가 있으면 다음 페이지를 누적 로드하고, 선택한 run 상세만 `GET /v1/backtest-runs/{backtestRunId}`로 조회한다.
- 목록은 체결·산출물 대량 payload를 싣지 않는 summary만 반환하고, 상세 조회는 기존 읽기 전용 payload를 유지한다.

## P4-5 완료 범위

- `backtest_runs`에 `attempt_count`, `max_attempts`, `next_retry_at`, `lease_owner`, `lease_expires_at`, `lease_generation`, 마지막 오류와 `dead_letter_reason`을 추가하는 dbmate migration 작성
- `queued`, `retry_wait`, 만료된 `running` run을 `FOR UPDATE SKIP LOCKED`와 generation fencing으로 claim
- claim 시 `attempt_count`·`lease_generation` 증가와 lease owner·expiry 기록
- 완료 시 현재 owner·generation·미만료 lease를 검증한 뒤 결과 row와 artifact를 같은 transaction에 저장하고 `succeeded|failed` terminal 상태로 봉인
- 실패 시 시도 예산이 남으면 `retry_wait`, 소진하면 `dead_letter`로 전이
- `BacktestWorker` core 추가: claim → executor → complete, executor 오류 → fail, lease lost → 늦은 쓰기 없음
- live PostgreSQL E2E에서 claim, stale worker fencing, artifact 저장, retry_wait, dead_letter 전이 확인

## P4-6 완료 범위

- `GET /v1/backtest-runs/{backtestRunId}/trades` 체결 결과 페이지 API 추가
- `GET /v1/backtest-runs/{backtestRunId}/equity-points` 자산곡선 페이지 API 추가
- 체결은 `trade_sequence`, 자산곡선은 `point_sequence` 오름차순 keyset pagination으로 조회
- cursor는 run ID, 첫 페이지 sequence 상한, 마지막 sequence와 HMAC digest로 문맥과 무결성을 검증
- 단건 `BacktestRun` 상세 payload는 기존 호환을 유지하고 대용량 소비자는 새 페이지 API를 사용
- Web client에 `loadBacktestTrades`, `loadBacktestEquityPoints` 추가

## P4-7 완료 범위

- `POST /v1/backtest-runs` 실행 생성 API 추가
- 운영 토큰(`X-Operator-Token`)이 있는 명령만 accepted 상태(`202`)로 처리
- published 전략 version과 sealed 데이터셋 version만 run 입력으로 허용
- strategy graph hash, dataset content hash, dataset 범위·정책, engine version, canonical parameters hash, initial cash, execution model, seed, materialized candle events를 `backtest-run-input-v1` payload로 materialize하고 백테스트 엔진과 같은 `input_hash`를 결정론적으로 생성
- `backtest_runs.input_payload`에 생성 시점 engine input을 보존하고 Worker claim에서 반환
- 새 run은 내부 DB 상태 `queued`, 외부 API 상태 `pending`, `resultHash=null`, `startedAt=null`, `finishedAt=null`로 반환
- 같은 idempotency key와 같은 본문은 기존 run을 재생하고, 같은 key의 다른 본문은 `BACKTEST_IDEMPOTENCY_CONFLICT` 409로 거부
- 같은 semantic input hash가 이미 있으면 다른 idempotency key의 중복 생성으로 보고 409로 거부
- Web client에 `createBacktestRun` 추가

## P4-8 완료 범위

- `WalkForwardFold`, `SensitivityPoint`, `BootstrapSample` 성과 artifact 입력 타입 추가
- `build_performance_artifacts()`가 `walk_forward_summary`, `sensitivity_summary`, `bootstrap_summary` artifact를 Backtest Store 입력 형태로 생성
- artifact metadata는 각각 `backtest-artifact-walk-forward-v1`, `backtest-artifact-sensitivity-v1`, `backtest-artifact-bootstrap-v1` schema version을 가진다.
- `contentHash`는 metadata 정규 JSON SHA-256이며 입력 순서와 dictionary 삽입 순서에 의존하지 않는다.
- walk-forward fold, sensitivity point, bootstrap sample은 안정 정렬 후 저장하고 `finalEquity` min/max/mean 요약을 포함한다.

## P4-9 완료 범위

- `ws://<api-host>/v1/backtest-runs/{backtestRunId}/progress` 진행 WebSocket snapshot 계약 추가
- 연결 직후 현재 `BacktestRunSummary`에서 파생한 `backtest.progress` 메시지를 전송하고 연결을 닫는다.
- 진행률은 별도 progress column 없이 `pending=0`, `running=50`, `succeeded|failed|cancelled=100`으로 상태에서 파생한다.
- 없는 run은 `backtest.error`와 `BACKTEST_RUN_NOT_FOUND`를 전송한다.
- Web client에 `backtestProgressWebSocketUrl()` helper 추가
