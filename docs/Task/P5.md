# P5 포트폴리오·paper/shadow 봇·주문 상태·위험 통제

상태: 진행 중
GitHub Issue: #32

## 목표

전략 버전과 백테스트 결과를 안전한 모의 운영으로 연결한다. P5는 실제 주문 endpoint를 호출하지 않고 portfolio, bot, order intent, paper/shadow execution, reconciliation evidence, risk limit/event, kill switch를 검증 가능한 상태로 만든다.

## 범위 밖

- 실제 Upbit 주문 제출
- private WebSocket과 주문 테스트 API 기반 live-ready 검증
- live 활성화 또는 live capability 변경
- 출금 기능·권한

## P5-1 완료 목표

- 포트폴리오, 정책, 자금 배분 DB 계약
- 봇 정의, 인스턴스, 상태 전이 감사 DB 계약
- paper/shadow 전용 주문 의도, simulated exchange order, fill, position projection DB 계약
- 위험 한도, 위험 이벤트, kill switch DB 계약
- 감사 이벤트와 fill은 append-only trigger로 보호

## P5-2 완료 목표

- `POST /v1/portfolios` 멱등 명령 API
- `GET /v1/portfolios` owner 문맥 cursor 목록 API
- `PostgresPortfolioBotStore` 포트폴리오 생성·목록 저장소
- 포트폴리오 API 명령의 `request_id`, `idempotency_key`, `requested_at`, `request_hash` DB 계약
- 동일 멱등 키와 다른 요청 본문 충돌 409 처리

## P5-3 완료 목표

- `paper_execution_jobs` 임대(lease)·재시도·dead-letter queue DB 계약
- approved paper `order_intents`만 claim하는 `PostgresPortfolioBotStore` 메서드
- claim된 job을 주입된 simulator 결과로 완료하는 `PaperExecutionWorker`
- completion transaction에서 simulated `exchange_orders`, `paper_simulator` fill, `position_projections`, `order_intents.status='paper_filled'` 기록
- 실제 Upbit 주문, private WebSocket, 주문 테스트 API 경로 미사용

## P5-4 완료 목표

- `RiskEvaluationWorker`가 `created` paper/shadow 주문 의도를 한 transaction에서 평가
- 전역·포트폴리오·봇·거래쌍 `risk_limits`와 전역·포트폴리오·봇 `kill_switches` 최신 상태 적용
- 승인 시 `approved`, `policy_approved` 위험 이벤트와 paper 실행 queue 연결
- 한도 초과·계산 증거 부족·kill switch 활성화 시 `risk_rejected`, `limit_rejected|kill_switch_rejected` 위험 이벤트 기록
- kill switch arm 이후 이미 승인된 paper 실행 job도 claim과 completion에서 차단
- 실제 Upbit 주문, private WebSocket, 주문 테스트 API 경로 미사용
