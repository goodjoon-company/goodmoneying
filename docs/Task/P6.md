# P6 Upbit 실거래 준비

상태: 진행 중
GitHub Issue: #33

## 목표

Upbit 주문 테스트(order-test), 실제 주문 어댑터(adapter), private `myOrder` WebSocket, REST 대사(reconciliation), SMP(Self-Match Prevention), 멱등 주문(identifier), 긴급 정지(Kill Switch), live-ready gate를 구현한다.

P6의 구현 목표는 실거래 준비도 검증이다. 실제 자금이 움직이는 live 활성화와 실제 주문 실행은 운영자 승인 gate 전까지 기본 비활성이다.

## 범위 밖

- 주식시장 기능과 타 거래소 실제 연동
- 출금 기능·권한
- AI 또는 CI에 의한 live 활성화
- 안전 gate를 우회한 실제 주문 제출

## P6-1 완료 목표

- Upbit 주문 identifier 최대 64자 제약을 `rest.new-order`, `rest.order-test`, 주문 조회·취소 identifier, `cancel-and-new-order` identifier 계약에 반영
- `time_in_force=post_only`와 `smp_type` 동시 사용을 `rest.new-order`, `rest.order-test` 계약에서 금지
- `new_time_in_force=post_only`와 `new_smp_type` 동시 사용을 `rest.cancel-and-new-order` 계약에서 금지
- Gateway validator가 문자열 `max_length`와 값 기반 금지 조합을 상향 호출 전 검사
- 문서 카탈로그와 패키지 내장 카탈로그를 동일하게 유지
- 공식 스냅숏 fixture와 계약 테스트를 갱신해 drift를 차단

## P6-1 안전 경계

- `rest.order-test`는 여전히 `safety: test`, `rate_limit_group: order-test`다.
- `rest.new-order`, 취소, 취소 후 재주문은 여전히 `safety: blocked`다.
- 새 코드는 실제 주문 제출, 취소, private WebSocket 연결, credential 권한 변경을 수행하지 않는다.
- 주문 테스트 API가 반환한 UUID·identifier를 조회·취소에 사용하는 기능은 아직 추가하지 않았다.

## P6-2 order-test 증적과 실제 주문 식별자 격리 완료 목표

- `exchange_accounts` DB 계약으로 Upbit 계좌의 안정 식별자와 live 비활성 기본 상태를 분리
- `upbit_order_identifier_reservations` DB 계약으로 live identifier와 order-test 응답 식별자를 같은 계좌 namespace에서 원자적으로 예약
- `live_order_identifiers` DB 계약으로 `order_intents`별 deterministic `gm1_` identifier를 계좌 안에서 영구 unique로 예약
- `upbit_order_test_runs` DB 계약으로 order-test 응답 UUID·identifier를 조회·취소 불가 증적으로만 보존
- `upbit_order_test_runs`는 UPDATE·DELETE를 거부하는 append-only 증적으로 유지
- 테스트 주문 응답을 실제 주문 원장·조회·취소 식별자로 재사용하지 못하게 한다.
- `derive_upbit_live_order_identifier()`가 계좌 안정 식별자와 주문 멱등 키에서 56자 live identifier를 결정론적으로 생성
- DB 함수와 trigger가 `live_order_identifiers.idempotency_key`와 `order_intents.idempotency_key` 불일치, 결정론적 identifier 불일치, order-test 응답 식별자의 live 재사용을 거부
- live PostgreSQL E2E에서 order-test 증적과 live identifier table이 분리되고, order-test lookup/cancel 허용, 비-live identifier 삽입, 양방향 식별자 재사용, live/test 동시 재사용 경쟁이 DB에서 거부됨을 검증
- dbmate schema snapshot을 새 migration 기준으로 갱신

## P6-2 안전 경계

- 새 테이블은 실제 주문 제출 outbox가 아니라 식별자·증적 경계만 만든다.
- order-test 응답 UUID·identifier는 `lookup_allowed=false`, `cancel_allowed=false` 제약으로만 저장된다.
- live identifier는 `order_intents`와 연결되지만 아직 Upbit `POST /v1/orders` 상향 호출에 사용하지 않는다.
- 출금 권한·출금 endpoint와 연결하지 않는다.

## 후속 범위

1. `live_disabled` capability gate: 배포 SHA, 만료, 조회 실패 폐쇄형 평가, CI/AI 거부를 DB 계약과 E2E로 고정한다.
2. private `myOrder` 대사: 무이벤트 정상 처리, REST snapshot 복구, `prevented`·부분 체결 적용, `outcome_unknown` 재제출 금지를 구현한다.
3. 안전 주문 adapter: outbox, 주문·조회 권한 준비도, 출금 권한 미사용 검증을 추가한다.
