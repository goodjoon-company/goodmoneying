# P6 Upbit 실거래 준비

상태: 진행 중
GitHub Issue: #33

## 목표

Upbit 주문 테스트(order-test), 실제 주문 어댑터(adapter), private `myOrder` WebSocket, REST 대사(reconciliation), SMP(Self-Match Prevention), 멱등 주문(identifier), 긴급 정지(Kill Switch), live-ready gate를 구현한다.

P6의 구현 목표는 실거래 준비도 검증이다. 실제 자금이 움직이는 live 활성화와 실제 주문 실행은 운영자 승인 gate 전까지 기본 비활성이다.

## 범위 밖

- 주식시장 기능과 타 거래소 실제 연동
- 출금 기능·권한
- AI 또는 CI에 의한 live 활성화
- 안전 gate를 우회한 실제 주문 제출

## P6-1 완료 목표

- Upbit 주문 identifier 최대 64자 제약을 `rest.new-order`, `rest.order-test`, 주문 조회·취소 identifier, `cancel-and-new-order` identifier 계약에 반영
- `time_in_force=post_only`와 `smp_type` 동시 사용을 `rest.new-order`, `rest.order-test` 계약에서 금지
- `new_time_in_force=post_only`와 `new_smp_type` 동시 사용을 `rest.cancel-and-new-order` 계약에서 금지
- Gateway validator가 문자열 `max_length`와 값 기반 금지 조합을 상향 호출 전 검사
- 문서 카탈로그와 패키지 내장 카탈로그를 동일하게 유지
- 공식 스냅숏 fixture와 계약 테스트를 갱신해 drift를 차단

## P6-1 안전 경계

- `rest.order-test`는 여전히 `safety: test`, `rate_limit_group: order-test`다.
- `rest.new-order`, 취소, 취소 후 재주문은 여전히 `safety: blocked`다.
- 새 코드는 실제 주문 제출, 취소, private WebSocket 연결, credential 권한 변경을 수행하지 않는다.
- 주문 테스트 API가 반환한 UUID·identifier를 조회·취소에 사용하는 기능은 아직 추가하지 않았다.

## P6-2 order-test 증적과 실제 주문 식별자 격리 완료 목표

- `exchange_accounts` DB 계약으로 Upbit 계좌의 안정 식별자와 live 비활성 기본 상태를 분리
- `upbit_order_identifier_reservations` DB 계약으로 live identifier와 order-test 응답 식별자를 같은 계좌 namespace에서 원자적으로 예약
- `live_order_identifiers` DB 계약으로 `order_intents`별 deterministic `gm1_` identifier를 계좌 안에서 영구 unique로 예약
- `upbit_order_test_runs` DB 계약으로 order-test 응답 UUID·identifier를 조회·취소 불가 증적으로만 보존
- `upbit_order_test_runs`는 UPDATE·DELETE를 거부하는 append-only 증적으로 유지
- 테스트 주문 응답을 실제 주문 원장·조회·취소 식별자로 재사용하지 못하게 한다.
- `derive_upbit_live_order_identifier()`가 계좌 안정 식별자와 주문 멱등 키에서 56자 live identifier를 결정론적으로 생성
- DB 함수와 trigger가 `live_order_identifiers.idempotency_key`와 `order_intents.idempotency_key` 불일치, 결정론적 identifier 불일치, order-test 응답 식별자의 live 재사용을 거부
- live PostgreSQL E2E에서 order-test 증적과 live identifier table이 분리되고, order-test lookup/cancel 허용, 비-live identifier 삽입, 양방향 식별자 재사용, live/test 동시 재사용 경쟁이 DB에서 거부됨을 검증
- dbmate schema snapshot을 새 migration 기준으로 갱신

## P6-2 안전 경계

- 새 테이블은 실제 주문 제출 outbox가 아니라 식별자·증적 경계만 만든다.
- order-test 응답 UUID·identifier는 `lookup_allowed=false`, `cancel_allowed=false` 제약으로만 저장된다.
- live identifier는 `order_intents`와 연결되지만 아직 Upbit `POST /v1/orders` 상향 호출에 사용하지 않는다.
- 출금 권한·출금 endpoint와 연결하지 않는다.

## P6-3 live_disabled capability gate 완료 목표

- `trading_capabilities` DB 계약으로 global live capability 권위 상태를 append-only 증적으로 저장
- 기본 상태, 권위 행 없음, DB 조회 실패, 배포 SHA 불일치, 승인 만료는 모두 `live_disabled`로 평가
- CI·AI·service actor는 live capability를 변경할 수 없다.
- live capability 평가는 환경변수나 API Key 존재가 아니라 DB 권위 행, 배포 SHA, 만료 시각만 사용
- live PostgreSQL E2E에서 operator 승인 행만 `live_enabled`가 되고, SHA mismatch·만료·CI actor·UPDATE가 모두 거부됨을 검증

## P6-3 안전 경계

- 이 단계는 live 활성화 명령 API를 만들지 않는다.
- 실제 Upbit 주문 제출, 취소, private WebSocket, 출금 권한 관련 변경은 없다.
- `live_enabled` 평가는 후속 실제 주문 adapter의 입력 guardrail일 뿐, 현재 코드 경로에서 주문을 전송하지 않는다.

## P6-4 private myOrder 대사 입력 계약 완료 목표

- `docs/contracts/upbit/myorder-event.md`로 private `myOrder` 이벤트 내부 대사 입력 계약을 고정
- `myOrder`가 initial snapshot을 보내지 않으므로 무이벤트를 정상 상태로 처리하고 REST snapshot 대사를 요구
- `prevented_volume`, `prevented_locked`, nullable `trade_fee`, nullable `is_maker`를 보존
- 부분 체결(`state=trade`, `remaining_volume > 0`)을 `partial_fill` 관측 상태로 분류
- 모든 `myOrder` 대사 계획은 `can_resubmit=false`로 동일 주문 재제출을 금지

## P6-4 안전 경계

- 실제 private WebSocket 연결을 새로 열지 않는다.
- 실제 Upbit REST 조회, 주문 제출, 취소, outbox 생성은 후속 범위다.
- parser는 내부 대사 입력만 만들며 계좌 비밀이나 주문 권한을 요구하지 않는다.

## P6-5 private myOrder REST snapshot 적용 완료 목표

- `docs/contracts/upbit/rest-order-reconciliation.md`로 주문조회 권한 기반 REST snapshot 대사 계약을 고정
- `GET /v1/order`, `GET /v1/orders/open`, `GET /v1/orders/closed`, `GET /v1/orders/uuids`의 조회 경계와 권한을 문서화
- REST 주문 snapshot을 내부 대사 입력으로 정규화하고 terminal snapshot만 기존 `reconcile_exchange_order()`에 적용
- `wait|watch|trade` 진행 중 snapshot은 원장 변경 없이 observe-only로 처리
- REST `paid_fee`, `prevented_volume`, `prevented_locked`, `tradeUuid`, `sourceEndpoint`를 대사 증거에 보존
- 모든 REST snapshot plan은 `can_resubmit=false`로 동일 주문 재주문을 금지

## P6-5 안전 경계

- 실제 Upbit REST client, private WebSocket 연결, 주문 제출, 주문 취소를 추가하지 않는다.
- 기존 `paper|shadow` 원장 대사 경계를 `live`로 확장하지 않는다.
- live 주문 UUID·identifier와 `exchange_orders`의 영구 결합은 후속 안전 주문 adapter 계약에서 다룬다.

## P6-6 안전 주문 adapter 완료 목표

- `upbit_api_key_permission_attestations` DB 계약으로 주문하기·주문조회 권한이 모두 있고 출금 권한이 없는 API Key 준비도 증적만 허용
- `upbit_order_outbox` DB 계약으로 live identifier가 예약된 주문 의도를 실제 제출 전 outbox 증적으로만 저장
- CI·AI·service actor는 권한 증적과 outbox를 생성할 수 없다.
- outbox는 `ready|blocked`만 저장하며 실제 submit attempt는 `0`으로 고정한다.
- `ready` outbox는 `approved` 주문 의도(order intent)만 허용하며, 권한 증적(permission attestation)을 참조하면 `blocked` 상태라도 같은 거래소 계좌(exchange account)에 귀속돼야 한다.
- shared adapter는 live capability, 권한 준비도, 권한 만료, 출금 권한 존재, kill switch를 fail-closed로 평가한다.

## P6-6 안전 경계

- 실제 `POST /v1/orders` 호출, 취소, private WebSocket, REST client를 추가하지 않는다.
- 출금 권한이 있는 API Key는 준비도 증적으로 저장할 수 없다.
- `ready` outbox도 제출 가능 상태가 아니라 후속 worker가 소비할 감사 가능한 후보일 뿐이다.

## P6-7 live 주문 결합 완료 목표

- `exchange_orders.execution_mode='live'`를 DB 계약에 추가하되 binding 없이 커밋되는 live 주문 행을 지연 제약 트리거(deferrable constraint trigger)로 거부한다.
- `upbit_live_exchange_order_bindings`로 Upbit 주문 UUID와 내부 deterministic `gm1_` identifier를 내부 `exchange_orders`에 결합한다.
- 결합은 `live_order_identifiers`, `upbit_order_outbox`, `exchange_orders`가 같은 거래소 계좌(exchange account)와 주문 의도(order intent)에 귀속될 때만 허용한다.
- order-test 응답 UUID·identifier는 live exchange order로 결합할 수 없다.
- 결합 증적 생성 시 `live_order_identifiers.status`를 `submitted`로 전이한다.
- shared adapter는 결합 후보를 정규화하지만 실제 제출(`can_submit`), 취소(`can_cancel`), 동일 주문 재제출(`can_resubmit`)은 계속 금지한다.

## P6-7 안전 경계

- 실제 `POST /v1/orders` 호출, 취소, private WebSocket 연결을 추가하지 않는다.
- P6-7은 이미 관측된 표준 UUID·identifier를 내부 원장에 결합하는 계약만 추가한다.
- live `exchange_orders`는 같은 트랜잭션 안에서 binding을 만들 때만 커밋 가능하다.
- live 주문 대사(reconciliation) 적용과 submit worker는 후속 범위다.

## 후속 범위

1. 안전 주문 submit worker와 Upbit Gateway 실제 주문 제출 전 dry-run/rehearsal 경계를 추가한다.
2. live 주문 대사(reconciliation) 적용 경계를 추가한다.
