## 수동 확인

- `rg`로 `rateLimitRemainingPercent`, `duplicateRows24h`, `Rate limit 여유`, `중복 행`, 고정 `quality_status="normal"`, 고정 `missing_segment_count=1`, `collectionRangeDisplay: "2024"` 참조를 확인했다.
- 남은 참조는 이번 Task 설명 또는 이전 Task의 이력 문맥뿐이며, 실행 코드와 API 계약에는 남아 있지 않다.

## 남은 리스크

- PostgreSQL 물리 저장량은 아직 `pg_total_relation_size`가 아니라 행 수 기반 추정값이다. 이번 범위에서는 UI 라벨과 계약 설명을 “오늘 저장 행 기반 추정”으로 좁혔다.
- Upbit rate limit 헤더 영속화는 별도 관측성 테이블이 필요하므로 이번 범위에서 제거했다.

## 추가 검증: 격리 로컬 전체 시스템

요청에 따라 PostgreSQL, API, Web, Worker를 모두 로컬에서 격리 실행했다. 기존 개발 DB와 Podman volume을 사용하지 않고, 임시 PostgreSQL 컨테이너에 tmpfs를 연결해 테스트 후 삭제했다.

실행 방식:

```text
PostgreSQL: podman postgres:18, tmpfs /var/lib/postgresql, 임시 포트
Worker: fixture client로 1회 seed 후 2초 간격 loop 실행
API: GOODMONEYING_DATABASE_URL=임시 PostgreSQL
Web: VITE_API_BASE_URL=임시 API
E2E: E2E_SKIP_WEBSERVER=1, 실행 중인 임시 API/Web 사용
Browser QA: Playwright Chromium 직접 실행
```

확인된 런타임 데이터:

```text
DASHBOARD activeTargets=50 targets=50 coverage=150
DASHBOARD totals_keys=activeTargetLimit,activeTargets,delayedTargets,failedRuns24h,failureRate24h,incidentTargets,missingRangesOpen,normalTargets,recentRequestCount,storageBytesToday,storageBytesTodayDisplay,warningTargets
DB_COUNTS collection_runs=4
```
