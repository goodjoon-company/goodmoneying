## 확인 내용

- `/v1/dashboard/summary/stream?once=true`는 `event: dashboard`와 `DashboardSummary` JSON payload를 반환한다.
- 운영 화면은 `subscribeDashboardSummary()` 구독 콜백을 통해 기존 `OperationsSnapshot`의 `dashboard`와 `notifications`를 갱신한다.
- SSE 연결과 일반 `/v1/dashboard/summary` 요청이 겹칠 때 로컬 SQLite 저장소가 같은 연결(connection)을 동시에 쓰지 않도록 대시보드 요약 계산을 재진입 잠금(reentrant lock)으로 직렬화했다.
- live 업비트 프로필(`GOODMONEYING_LIVE_UPBIT=1`)의 `realtime_collection_worker.main()`은 웹소켓 스트림 워커로 위임한다.
- 웹소켓 메시지 변환은 `ticker`, `orderbook`, `candle.1m` 기본 포맷(DEFAULT)을 기존 `TickerSnapshot`, `OrderbookSummary`, `SourceCandle` 모델로 변환한다.
- 웹소켓 스트림이 flush 주기 전에 예외로 종료되어도 직전까지 버퍼에 있던 시세 데이터는 기존 incremental 저장 경로로 flush한 뒤 예외를 호출자에게 전파한다.
- 웹소켓 버퍼 flush가 실제 row를 저장하면 `realtime_collection` heartbeat를 `running`으로 갱신해 운영 화면의 Realtime worker 상태가 `지연`으로 오판되지 않도록 했다.
- 오늘 저장 Row Count, `storageBytesToday`, `storageRowsToday`, `storageBreakdown`, `operationsTrend.storageBytes`는 `target_collection_results`를 제외하고 원천 시장 데이터 저장 테이블 3종(`source_candle`, `ticker_snapshot`, `orderbook_summary`)만 집계한다.
- 수집 결과 로그(`target_collection_results`)는 DB 스키마(DB schema)를 바꾸지 않고 최근 요청 수(`recentRequestCount`), worker 24시간 수집량, 활동 버킷 같은 운영 지표로 유지한다.
- fixture 프로필은 기존 REST 기반 1회 수집 경로를 유지한다.

## 남은 리스크

- 실제 업비트 웹소켓 live 연결은 이번 자동 검증에서 외부 API를 호출하지 않았다.
- 재연결 backoff, 누락 이벤트 보정, 연결별 구독 재설정, 메시지 큐(Message Queue) 분리는 후속 Task 범위다.
- 캔들 웹소켓은 체결이 없으면 이벤트가 없으므로 히트맵 기대 건수 계산은 별도 개선이 필요하다.
