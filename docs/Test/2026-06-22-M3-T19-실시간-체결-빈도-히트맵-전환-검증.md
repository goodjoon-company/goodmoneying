## 확인 내용

- 웹소켓 구독 요청은 기존 `ticker`, `orderbook`, `candle.1m`에 `trade` 타입을 추가한다.
- `trade` 메시지는 `TradeEvent`로 변환되며, `(instrument_id, source, sequential_id)` 유니크 키로 중복 저장을 차단한다.
- `trade_events` 저장 시 `collection_runs`와 `target_collection_results`에는 `trade_event` data type을 기록한다.
- `realtimeCollectionHeatmap` 셀은 시간당 `tradeCount`를 `averageTradesPerMinute = tradeCount / 60`으로 환산해 색상 상태를 결정한다.
- 색상 상태는 10건 미만 `red`, 10~50 미만 `orange`, 50~100 미만 `yellow`, 100~200 미만 `blue`, 200 이상 `green`이다.
- 체결강도는 `BID 체결량 / ASK 체결량 * 100`으로 계산하고, ASK 체결량이 0이면 0으로 둔다.
- 마우스 오버 상세에는 시간 평균 체결빈도, 체결강도, 체결량, 체결금액을 표시한다.
- OpenAPI `RealtimeCollectionHeatmapCell` 계약은 체결 빈도 지표 필드와 새 색상 enum을 요구한다.
- 기존 API 필드명 `realtimeCollectionHeatmap`은 유지해 대시보드 응답 구조의 상위 호환성을 보존했다.

## 남은 리스크

- 실제 업비트 웹소켓 live 연결은 이번 자동 검증에서 외부 API를 호출하지 않았다.
- `trade_events` 장기 보존 정책, 파티셔닝(Partitioning), 압축 집계 테이블은 후속 설계 범위다.
- 체결 빈도 기준은 거래량이 많은 코인에 민감하므로 운영 화면에서 전체 시장과 관심 코인 범위를 함께 해석해야 한다.
