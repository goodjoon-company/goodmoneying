# P0-5 아키텍처와 계약 정렬 검증

Date: 2026-07-10
Related Task: GitHub Issue #7
Environment: macOS 로컬 저장소, `codex/p0-5-architecture-contract-alignment`

## 검증 대상

- 아키텍처 기준이 M1을 현재 구현 범위로 설명하지 않고 M1~M3를 완료 기준선으로 설명하는지 확인한다.
- 관심종목과 코인 상세는 사용자 가치 화면, 운영 상태와 Backfill은 내부 운영 표면으로 구분됐는지 확인한다.
- 호가 원천, 다중 워커, 메시지 큐, 고가용성, 시장 확장이 고정 일정이 아니라 측정 가능한 결정 게이트로 정의됐는지 확인한다.
- 기존 ADR의 구현 결정은 보존하고 미래 일정만 ADR-0007로 정제(refine)했는지 확인한다.
- OpenAPI, DB, 메시지 계약 변경 필요 여부를 실제 계약과 런타임 소비 코드로 확인한다.
- `timestamptz` 절대 시각과 KST 세션·계산·표시 정책이 ADR, 아키텍처, 계약 테스트에서 일치하는지 확인한다.

## 코드·문서 리뷰

- 요구사항 적합성: Issue #7의 아키텍처, 업비트 수집 파이프라인, 계약, ADR 검토 범위를 모두 수행했다.
- 제품 기준 적합성: M1~M3를 완료 기준선으로 두고 현재 Post-MVP 제품화에서 관심종목과 코인 상세를 우선한다.
- 아키텍처 적합성: 현재 런타임 구조와 데이터 흐름은 바꾸지 않고 미래 확장 진입 조건만 정렬했다.
- 계약 적합성: API·DB·메시지 계약과 런타임 소비 코드 사이에 드리프트가 없다.
- 시간대 적합성: 저장 형식은 절대 시각 `timestamptz`, DB 세션·계산·표시는 KST라는 경계를 ADR과 파이프라인에 동일하게 기록했다.
- ADR 적합성: 과거 결정을 덮어쓰지 않고 ADR-0007에서 미래 일정만 결정 게이트로 정제했다.
- 테스트 적합성: 계약·API 테스트 39개가 통과했고 기존 테스트 기대값을 변경하지 않았다.
- 보안·성능·운영 안정성: 현재 신뢰 네트워크, 단일 인스턴스, DB 상태 제어를 유지하며 확장 임계값을 명시했다.
- 프론트엔드 품질: 화면 코드·스타일·동작을 변경하지 않아 브라우저·디자인 QA는 미적용이다.
- 추적성: Issue #7, Product, Architecture, ADR-0007, 이 Test, `docs/History/2026-07-10-P0-5-아키텍처와-계약-정렬.md`를 연결했다.

## 종단 간 테스트와 미검증 항목

이번 작업은 런타임 코드와 사용자 화면 동작을 변경하지 않으므로 새 자동화 종단 간 테스트(E2E Test)는 실행하지 않았다. 전체 CI는 push 후 확인하고 Issue #7 댓글에 기록한다.

확장성 진입 임계값의 구체적인 수치는 아직 정하지 않았다. 실제 처리량·복구·저장 문제가 관찰되거나 새 시장·전략 실험이 승인될 때 별도 Task와 ADR에서 수치화한다.

## 결론

제품과 현재 아키텍처의 충돌을 제거했다. 호가 원천, 확장성, 주식·LLM·봇은 고정 일정이 아니라 근거 기반 결정 게이트이며, 현재 API·DB·메시지 계약은 변경할 필요가 없다.
