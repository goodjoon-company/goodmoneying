## 종단간 시험(E2E Test)

### 기본 명령 실패와 원인

Issue에 적힌 `npm --prefix apps/web run test:e2e`는 `apps/web/package.json`에 해당 스크립트가 없어 현재 저장소 계약과 다르다. 실제 루트 명령인 `npm run e2e`를 실행했지만, Playwright 기본 서버가 루트 `.env`의 외부 `GOODMONEYING_DATABASE_URL`을 읽어 다음 오류로 API 시작에 실패했다.

```text
psycopg.OperationalError: connection failed: password authentication failed for user "goodmoneying"
Error: Process from config.webServer was not able to start. Exit code: 1
```

제품 결함이 아니라 로컬 E2E 하네스(Harness)가 외부 DB 설정과 결합된 문제이며 GitHub Issue #10으로 분리했다.

### 격리 재검증

운영 DB에는 접근하지 않았다. `SQLiteOperationsRepository`에 `FixtureUpbitClient`를 명시적으로 직접 주입한 테스트 전용 API를 `18000` 포트에, 웹을 `15173` 포트에 실행한 뒤 다음 명령으로 검증했다.

```bash
E2E_SKIP_WEBSERVER=1 \
E2E_API_BASE_URL=http://127.0.0.1:18000 \
E2E_WEB_BASE_URL=http://127.0.0.1:15173 \
E2E_OPERATOR_TOKEN=local-dev-token \
npm run e2e
```

결과: `1 passed (4.6s)`. 시작 데이터는 수집 후보군 100개와 활성 수집 대상 50개였고, 테스트 종료 뒤 격리 API·웹 서버를 중지했다. SSE(Server-Sent Events) 연결 종료 중 `CancelledError` 로그가 있었지만 테스트 결과와 프로세스 정리에는 영향을 주지 않았다.

## 최종 리뷰

- 요구사항 적합성: 완료된 M1~M3 기준선과 현재 P1·P2 사용자 가치 방향이 분리돼 있다.
- 제품 적합성: 운영 상태와 Backfill은 내부 운영 도구, 관심종목과 코인 상세는 사용자 가치 화면이다.
- 아키텍처 적합성: 현재 단일 저장소·워커·운영 서버 구조를 사실대로 유지하고 확장은 측정 가능한 결정 게이트를 따른다.
- 계약 적합성: P0 최종 검증에서 API·DB·메시지 계약 동작 변경은 없으며 계약 테스트를 포함한 전체 Python 테스트가 통과했다.
- ADR 적합성: 현재 미래 결정은 ADR-0007을 따르고 과거 ADR의 역사적 문구와 구분된다.
- 작업·테스트·인계 이력(Task/Test/History) 추적성: Issue #3~#9, 단일 기준 문서, 단계별 테스트·인계 이력과 최종 증적을 연결했다.
- 테스트 적합성: Python 145개, 웹 38개, 빌드, 격리 E2E 1개가 통과했다.
- 보안·운영 적합성: 외부 DB 인증 정보를 출력하거나 운영 데이터를 변경하지 않았고 E2E를 로컬 메모리 저장소로 격리했다.
- 프론트엔드 품질: 화면 코드는 변경하지 않았으며 기존 E2E에서 주요 흐름, 접근 가능한 이름, 콘솔·페이지 오류 0건을 확인했다.
