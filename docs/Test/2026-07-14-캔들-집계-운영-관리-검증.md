## 수동 확인 경로

Podman이 설치된 환경에서 `./dev.sh infra start` 후 API·web·실시간 수집·백필·캔들 집계 워커를 시작한다. 브라우저의 `시스템 관리` 메뉴에서 WebSocket 연결, 실시간 수집 대상의 데이터 유형, Backfill 대상, 집계 대상별 행 수와 전체 진행률을 확인한다.

현재 작업 환경은 `podman-not-found`여서 로컬 PostgreSQL 장기 실행 검증은 하지 못했다. 격리 SQLite 기반 Playwright E2E가 API/WebSocket/UI 경로를 대신 검증했다.

### 계약·아키텍처 판단

- DB 스키마 변경은 없다. 기존 heartbeat 테이블이 이미 `candle_aggregation` 유형을 허용하고 집계 대상 테이블이 `pending` 상태를 저장한다.
- `OperationsRepository`가 워커 runtime 상태를 읽는 공개 경계를 제공하고 SQLite/PostgreSQL 구현이 같은 상태 판정 규칙을 사용한다.
- WebSocket은 작업 상태인 `aggregation`과 워커 생존 상태인 `aggregationWorker`를 분리해 ADR-0009의 실제 heartbeat 표시 결정을 따른다.
- `pendingTargetCount`는 집계 대상 DB 상태에서 계산하며 전체 대상 수는 완료·실행·대기·실패 수의 합과 일치한다.
