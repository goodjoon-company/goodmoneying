### 계약·아키텍처 판단

- DB 스키마 변경은 없다. 기존 heartbeat 테이블이 이미 `candle_aggregation` 유형을 허용하고 집계 대상 테이블이 `pending` 상태를 저장한다.
- `OperationsRepository`가 워커 runtime 상태를 읽는 공개 경계를 제공하고 SQLite/PostgreSQL 구현이 같은 상태 판정 규칙을 사용한다.
- WebSocket은 작업 상태인 `aggregation`과 워커 생존 상태인 `aggregationWorker`를 분리해 ADR-0009의 실제 heartbeat 표시 결정을 따른다.
- `pendingTargetCount`는 집계 대상 DB 상태에서 계산하며 전체 대상 수는 완료·실행·대기·실패 수의 합과 일치한다.

### 구현·설계 판단

- 집계 작업을 claim한 뒤 5초 간격의 독립 heartbeat 실행기를 시작한다. 본 처리 스레드가 원천 조회·집계·UPSERT 어느 구간에 있든 주기는 처리 행 수와 무관하다.
- 정상 반환과 materialize 예외 모두 중지 이벤트를 설정하고 ticker 스레드를 합류해 고아 스레드를 남기지 않는다. ticker 내부 heartbeat 오류는 기록하고 다음 주기를 계속 기다려 집계 데이터 처리를 중단하지 않는다.
- 실제 런타임 PostgreSQL 저장소는 호출마다 독립 연결을 사용하므로 집계 트랜잭션과 heartbeat 연결을 공유하지 않는다. SQLite는 격리 테스트 어댑터의 재진입 잠금(RLock)으로 동일 객체의 스레드 접근을 직렬화한다.
- 원천행 fetch 크기 1,000은 메모리 배치 경계로만 유지하고 heartbeat 콜백을 제거했다. 따라서 heartbeat UPSERT 수는 원천행·집계봉 수가 아니라 작업 경과 시간의 상한을 따른다.
