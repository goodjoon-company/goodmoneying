### 구현·설계 판단

- 운영 PostgreSQL은 주 집계 저장소와 하트비트 저장소를 별도로 생성한다. 하트비트 저장소 연결 제한(connect timeout)과 문장 실행 제한(statement timeout)은 모두 2초다.
- 파일 SQLite는 같은 경로의 별도 연결을 만들고 잠금 대기 제한(busy timeout)을 2초로 둔다. 독립 연결로 같은 DB를 공유할 수 없는 `:memory:` 테스트는 같은 저장소를 재사용한다.
- SQLite `materialize_candle_rollups`는 원천 조회부터 모든 집계 UPSERT까지 재진입 잠금(Reentrant Lock)과 연결 컨텍스트의 단일 트랜잭션(transaction)으로 묶는다. 예외 시 전체 롤백되어 이후 상태·하트비트 쓰기가 부분 집계를 커밋하지 않는다.
- 하트비트 콜백은 데몬 스레드(daemon thread)에서 즉시 비동기 실행한다. 종료 시 중지 이벤트(stop event)를 설정하고 운영 기본 3초만 합류(join)하며, 초과하면 `aggregation_heartbeat_shutdown_timeout` 오류를 남기고 본 처리는 반환한다.
- PostgreSQL 연결·문장 실행 제한 2초와 SQLite 잠금 대기 제한 2초가 각 경계의 대기를 제한한다. 이는 모든 DB·네트워크·디스크 I/O(Input/Output)의 완료 상한을 보장하지 않는다. 임의 주입 콜백이 계속 막혀도 데몬 스레드이므로 프로세스 종료를 붙잡지 않으며 저하 상태가 로그로 관찰된다.

### 구현·설계 판단

- SQLite `materialize_candle_rollups`는 재진입 잠금(Reentrant Lock)을 얻은 직후 `BEGIN IMMEDIATE`를 실행한다. 원천 `SELECT`와 집계 UPSERT가 같은 쓰기 트랜잭션(transaction) 스냅샷(snapshot)에 있고 성공 시 명시적 커밋(commit), `BaseException` 포함 실패 시 명시적 롤백(rollback)을 수행한다.
- 5초 하트비트 실행기(heartbeat ticker)는 개별 `run_once`가 아니라 집계 폴링 루프(poll loop) 전체를 감싼다. 영구 차단 상태로 세 작업을 반복해도 데몬 스레드(daemon thread)는 하나이며, 정상 폴링과 집계 예외 종료에서는 남지 않는다.
- PostgreSQL 하트비트 저장소는 연결 제한과 문장 실행 제한을 각각 2초로 설정한다. SQLite 하트비트 연결은 잠금 대기 제한(busy timeout)만 2초로 설정한다. 네트워크·디스크를 포함한 모든 I/O(Input/Output)의 완료 상한으로 문서화하지 않는다.
- 시스템 관리 WebSocket 계약은 `running/stale/failed`, `lastHeartbeatAt`, 30초 `stale` 의미만 정의한다. 저장소 분리, 스레드, 합류(join)는 ADR-0009가 소유한다.
