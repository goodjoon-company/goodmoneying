## E2E 세부 결과

Docker E2E는 매번 격리 네트워크와 빈 PostgreSQL 17.10을 만든 뒤 다음 순서를 자동 실행하고 종료 시 컨테이너·네트워크·익명 볼륨·시험 이미지를 정리한다.

1. 고정 다이제스트(digest)의 dbmate 이미지로 기준선을 적용한다.
2. 시험용 `instruments` 행을 삽입하고 같은 마이그레이션을 재실행한다.
3. 적용 버전 1개와 시험 데이터 1행 보존, DB 시간대 `Asia/Seoul`을 확인한다.
4. 실제 API 이미지로 `/health`, `/v1/dashboard/summary` HTTP 200을 확인한다.
5. 생성 스키마를 정규화된 `docs/contracts/db/schema.sql`과 비교한다.
6. 호스트 `127.0.0.1` URL과 `pg_dump` 없는 조건에서 `dev.sh db dump`의 Docker 폴백 결과가 같은지 비교한다.

## 2026-07-16 GitHub Actions CI 보정

PR #18의 `pull_request`와 `push` 검증은 모두 DB 마이그레이션 E2E의 Docker 덤프 폴백에서 `Error: dial tcp 172.17.0.1:32768: connect: connection refused`로 실패했다. PostgreSQL 시험 컨테이너가 호스트 `127.0.0.1`에만 포트를 공개했는데, Ubuntu Docker Engine의 dbmate 컨테이너가 `host.docker.internal`의 브리지 게이트웨이로 접근한 것이 원인이었다.

- Docker Desktop은 기존처럼 DB URL을 `host.docker.internal`로 변환하고 호스트 게이트웨이(host gateway)를 추가한다.
- Linux Docker Engine은 원래 `127.0.0.1` DB URL과 호스트 네트워크(host network)를 사용한다.
- 그 밖의 Docker 운영체제는 Linux로 추정하지 않고 스키마 덤프 실행 전에 명시적으로 차단한다.
- Linux 경로와 미지원 운영체제 차단 테스트는 수정 전 실패를 확인한 뒤 구현했으며, Docker Desktop 보존 경로도 실행 기반 회귀 테스트를 추가했다.

## 남은 운영 경계

- DB 서버에는 dbmate 설치나 마이그레이션 파일 복사가 필요 없다.
- 실제 운영 적용 전에는 최신 백업, 앱 서버의 DB 연결, DDL 권한을 다시 확인해야 한다.
- 운영은 자동 rollback을 실행하지 않고 확장-축소(expand-contract)와 순방향 수정(forward fix)을 따른다.
