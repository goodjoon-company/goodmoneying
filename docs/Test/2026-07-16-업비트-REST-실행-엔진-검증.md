# 2026-07-16 업비트 REST 실행 엔진 검증

## 검증 대상

- HS512 JWT, 매 요청 UUID nonce, 입력 순서와 배열 키 반복을 보존한 SHA512 쿼리 해시(query hash)
- 저장소 밖 환경 변수 또는 절대 경로 읽기 전용 파일 자격 증명
- 공식 REST 그룹별 요청 제한(rate limit), 상향 `Origin` 제거, `Remaining-Req`, 429·418 냉각(cooldown)
- `read`·`test` 실행과 `blocked`의 자격 증명·제한기·네트워크 이전 차단
- 카탈로그 스칼라·열거형(enum)·범위·배열 `items`의 재귀 검증과 비유한 숫자 거부
- 고정 공식 URL과 명시적 루프백(loopback) 테스트 URL 허용 목록
- 키·JWT·Authorization·쿼리 해시(query hash)의 추적 봉투(Trace Envelope) 마스킹
- `dev.sh`의 `upbit-gateway` 8001 start·stop·restart·status·logs·readiness와 단독 실행 시 DB 비의존성
- 게이트웨이 Dockerfile, 로컬 Compose, 환경 샘플, CI 타입 검사·이미지 빌드

## 테스트 주도 개발 증적

인증, 제한기, 안전 정책, 클라이언트, 추적, 실행기, 개발 스크립트, 컨테이너 계약, 프로세스 E2E 순서로 실패 테스트를 먼저 작성했다. 각 첫 실행은 모듈 누락(`ModuleNotFoundError`), 심볼 누락(`ImportError`), 계약 누락(`KeyError`), 예상 상태 불일치로 실패했고 최소 구현 뒤 같은 범위를 통과시켰다.

[공식 인증 문서](https://docs.upbit.com/kr/reference/auth)의 Python 예시가 `unquote(urlencode(..., doseq=True))`를 사용하는 것을 재확인해 공백의 `+` 표현, 배열 키 반복, 대괄호, 불리언(boolean) 소문자와 입력 순서를 해시 계약으로 고정했다. 같은 정규 토큰에서 해시용 비인코딩 문자열과 전송용 백분율 인코딩(percent-encoding) 문자열을 각각 파생하며, `+09:00`·`id&uuid=extra`·`#fragment` 값이 실제 가짜 상류에서 원문 그대로 디코딩되는지 검증했다. [공식 요청 수 제한 문서](https://docs.upbit.com/kr/reference/rate-limits)를 기준으로 그룹 한도, `Remaining-Req`, 429·418, `Origin` 발동 조건을 재검토했다. `Origin`을 실제 수신한 요청만 1회/10초 대상이므로 브라우저 헤더를 제거한 게이트웨이 상향 요청에는 시세 `market` 그룹 10회/초를 적용했다. 실제 키 파일은 읽거나 복사하지 않았고 모든 인증 테스트는 64바이트 가짜 비밀 키만 사용했다.

독립 감사에서 먼저 추가한 회귀 테스트는 URL과 JWT의 쿼리 불일치, 브라우저 `Origin`의 잘못된 로컬 1회/10초 제한, WebSocket 식별자의 REST 실행 유입, 로컬 오류와 상향 상태의 OpenAPI 불일치, 418 본문 냉각 무시를 각각 재현했다. 첫 실행은 불리언 `True`·URL 대괄호 `%5B%5D`, 두 번째 Origin 요청 대기, WebSocket `INVALID_PARAMETERS`, 500 응답 계약 누락과 컴포넌트 불일치, 418 파서 심볼 누락 및 프로세스 후속 요청 시간 초과로 실패했다. 최소 구현 뒤 같은 테스트와 전체 검증을 통과시켰다.

최종 재감사에서는 예약 문자 값의 URL 주입·절단, `sec=0` 관찰의 기존 418 냉각 단축, 상향 422와 임의 상태의 계약 누락을 실패 테스트로 재현했다. 이어 Compose의 호스트 직접 키 상속, 전체 JWT·`Bearer` 로그 노출, 카탈로그와 런타임 제한 상수 드리프트(drift), `nan`·`inf`·`1e309`의 418 파서 예외를 각각 고정했다. 실패 결과는 잘린 전송 쿼리, 120초에서 1초로 줄어든 동시성 냉각, OpenAPI `default` 누락, Compose `null` 병합, 요청 제한 함수 누락, `OverflowError`였고 항목별 최소 구현 뒤 같은 범위를 통과시켰다.

마지막 계약 감사에서는 카탈로그의 모든 REST 배열 파라미터 17개에 유효 문자열 배열과 잘못된 객체·중첩 목록·객체 요소·정수·불리언 요소를 대입했다. 기존 구현은 바깥 `list`만 검사해 잘못된 요소를 상류까지 전송했으므로 회귀 테스트가 실패했다. `items` 스키마를 재귀 적용하고 숫자 `nan`·`inf`를 거부하도록 수정한 뒤, executor에서 요청 제한·자격 증명·네트워크 호출이 모두 0회이며 실제 프로세스 API가 422를 반환하고 가짜 상류 호출이 0회임을 확인했다. 기존 예약 문자 문자열 배열의 정규 해시와 안전 전송은 그대로 통과했다.

## E2E 안전 증적

가짜 상류 호출 목록을 별도 조회해 연속 브라우저 `Origin` 요청에도 상향 `Origin`이 없고 시세 요청이 1초 안에 연속 처리됐으며 `POST /v1/orders` 호출이 0회임을 확인했다. 취소·자산 이전·입출금·트래블룰 검증을 포함한 모든 상태 변경 endpoint는 카탈로그 계약상 `blocked`이고 실행기 안전 검사가 HTTP 클라이언트보다 먼저 수행된다. WebSocket 식별자는 REST 모델 경계에서 422로 거부됐다. 응답 문자열과 가짜 상류·게이트웨이 프로세스의 표준 출력(stdout)·표준 오류(stderr)를 가짜 접근 키·비밀 키·생성된 전체 JWT·`Bearer` Authorization 값으로 검색해 모두 0건임을 확인했다.

## 코드·보안·설계 리뷰

Issue #20, 제품 P2.2·GM-PROD-032~033, 게이트웨이 아키텍처, OpenAPI, 카탈로그, ADR-0011을 기준으로 직접 리뷰했다. 첫 리뷰에서 한 그룹의 제한 대기가 다른 그룹까지 잠그는 전역 잠금, 호스트 이름·경로가 붙은 테스트 URL 허용, 임의 응답 필드에 반사된 쿼리 해시(query hash), 네 가지 대안 필수 파라미터 조합의 비기계적 표현, 연결 예외 원문 보존, Compose 자격 증명 전달 누락을 발견했다.

각 항목에 실패 테스트를 추가한 뒤 그룹 잠금 해제 후 대기, 루프백 IP 리터럴과 경로 없는 URL만 허용, 계산된 쿼리 해시의 값 기반 마스킹, `any_of_required` 계약, 원인 사슬 없는 고정 연결 오류, 셸 환경 자격 증명 전달로 수정했다.

독립 감사에서 중요(Important) 5건과 경미(Minor) 3건을 추가 검토했다. 배열·불리언 정규 토큰, Origin 제거 후 실제 그룹 처리량, REST 전용 실행 경계, 로컬 오류·상향 상태의 완전한 OpenAPI 컴포넌트 일치, 418 헤더·중첩 JSON·문구·종료 시각의 최장 냉각과 비감소 보장으로 수정했다. Compose 읽기 전용 비밀정보, `dev.sh` DB 우회 설명, 실제 프로세스 보안 E2E도 보완했다. 구현 diff를 요구사항, 게이트웨이 설계, OpenAPI, ADR-0011, 최신 공식 인증·요청 제한 문서에 다시 대조했다.

최종 재리뷰에서는 해시·전송 직렬화의 보안 경계, 동시 요청 제한 갱신의 단조성(monotonicity), 모든 상향 상태의 추적 봉투 계약, Compose 환경 병합, 로그 비밀정보, 카탈로그 요청 제한 단일 기준, 비유한 수 처리까지 대조했다. 422는 로컬 오류와 상향 추적 봉투의 혼합 스키마이며 501·505 같은 임의 상태는 OpenAPI `default` 추적 봉투로 설명한다. 카탈로그의 모든 REST 제한 그룹이 런타임 구성과 정확히 일치하는 테스트도 통과했으며, 최종 보안·설계 재리뷰의 치명적(Critical)·중요(Important) 발견사항은 0건이다.

마지막 계약 재리뷰는 카탈로그 배열 요소 타입을 실행기가 그대로 소비하는지, 잘못된 요소가 제한기·키·네트워크에 도달하지 않는지, 유효 배열의 순서·반복·예약 문자 처리가 유지되는지 확인했다. 스칼라 enum·최솟값·최댓값은 기존 구현과 카탈로그가 일치했고, 일반 숫자 스키마에는 유한성 검사를 추가했다.

## 미검증 항목

- 실제 Upbit 계정·키·네트워크 호출은 안전 정책에 따라 의도적으로 검증하지 않았다.
- WebSocket 중계와 프론트엔드 기능별 시각화는 후속 Issue 범위다.

## 결론

가짜 자격 증명과 가짜 상류만 사용한 자동화 검증에서 REST 실행·인증·제한·마스킹·차단·개발 런타임 경계가 요구사항, 아키텍처, OpenAPI, 카탈로그와 일치한다.
