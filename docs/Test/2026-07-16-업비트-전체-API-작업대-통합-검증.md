## 공식 기준과 요청 제한

2026-07-16 업비트 개발자 센터의 최신 문서를 다시 확인했다.

- `Origin` 헤더가 있는 브라우저 직접 시세 조회(Quotation) REST와 공개 웹소켓(WebSocket)은 10초당 1회다.
- 서버 게이트웨이는 `Origin`을 전달하지 않으므로 시세 조회 그룹별 초당 10회/IP 제한을 적용받는다.
- 거래소(Exchange) 기본 그룹은 초당 30회, 주문과 주문 생성 테스트(order-test)는 각각 초당 8회, 주문 일괄 취소는 2초당 1회다.
- 웹소켓 연결은 초당 5회, 제어 메시지는 연결당 초당 5회이면서 분당 100회다.

참조: [요청 수 제한](https://docs.upbit.com/kr/reference/rate-limits), [WebSocket 사용 및 에러 안내](https://docs.upbit.com/kr/docs/websocket-best-practice), [주문 생성 테스트](https://docs.upbit.com/kr/reference/order-test)

## 인벤토리와 안전 경계

- REST 51종: Quotation 13종, Exchange 38종
- Exchange 안전 등급: 조회 23종, 주문 생성 테스트 1종, 영구 차단 14종
- WebSocket 스트림 14종과 `LIST` 제어
- 실제 주문·취소·자산 이전·입출금·Travel Rule 상태 변경은 프론트엔드(Frontend)와 게이트웨이에서 이중 차단한다.
- 브라우저에는 업비트 키·JWT(JSON Web Token)·Authorization 헤더를 전달하지 않는다.

## 실제 업비트 안전 연동

`scripts/verify_upbit_safe_live.py`로 첨부 키 파일을 읽되 값과 응답 본문을 출력하지 않고 다음 세 요청만 수행했다.

| 기능 | 결과 |
|---|---|
| 페어 목록 조회 | HTTP 200, `market` 그룹 잔여 9 |
| 포켓 잔고 조회 | HTTP 200, `default` 그룹 잔여 29, 자산 값 미출력 |
| 주문 생성 테스트 | HTTP 400, `order-test` 그룹 잔여 7, `under_min_total_bid` 공식 검증 오류 |
| 영구 차단 14종 | 로컬 14종 차단, 상향 호출 0건 |

주문 생성 테스트는 실제 주문을 만들지 않는다. Secret Key 길이가 40바이트라 HS512 권장 길이 64바이트보다 짧다는 라이브러리 경고가 있었으며, 기능 검증 결과에는 영향을 주지 않았다.

## TDD와 QA에서 발견한 결함

1. 모바일 375px에서 문서 너비가 867px로 확장되는 가로 넘침(overflow)을 재현 E2E로 먼저 실패시킨 뒤 390px 이하로 수정했다.
2. 하향 웹소켓이 닫힌 직후 오류 응답을 재전송하면 서버에서 `Cannot call "send" once a close message has been sent.`가 발생하는 경쟁 조건(race condition)을 단위 테스트로 재현하고 정상 종료 처리했다.
3. 실제 UI가 생성한 WebSocket 구독 14종을 정식 카탈로그와 직접 대조하도록 E2E를 강화했다.
4. 모바일 원본 추적 아이콘이 129px로 늘어나던 회귀를 38px로 고정하고 자동 검증했다.
5. 최종 독립 리뷰에서 게이트웨이 REST 실행 경로의 운영자 인증 누락을 발견해, 같은 출처 역방향 프록시가 주입하는 `X-Operator-Token`을 필수화하고 누락 401·불일치 403·정상 토큰 성공을 프로세스 E2E와 OpenAPI 계약으로 고정했다.
6. 백그라운드 WebSocket 읽기 중 하향 연결이 종료되면 자동 재연결이 시작되던 경로를 재현하고, 상향 연결을 닫되 재연결하지 않도록 수정했다.
7. 안전 실연동 검증기의 종료 코드를 공개 조회뿐 아니라 잔고 조회·주문 생성 테스트의 허용 결과와 영구 차단 14종·상향 호출 0건까지 모두 만족해야 0이 되도록 강화했다.
8. 운영 `app.env`의 공용 운영자 토큰을 REST 인증이 읽지 못하던 배포 결함을 실패 테스트로 재현하고, 게이트웨이 전용 토큰 우선·공용 배포 토큰 대체 규칙을 REST와 WebSocket이 함께 사용하도록 통합했다.

## 요구사항·아키텍처·계약·보안 리뷰

- 제품 요구사항: 세 하위 메뉴, 기능 그룹 탭, 공통 거래쌍, 입력 유형별 필드, 시각화, 원본 추적, 캔들 양방향 연속 조회를 모두 연결했다.
- 아키텍처(Architecture): 웹은 같은 출처 프록시를 통해 별도 업비트 게이트웨이에만 접근하고, 운영 배포·상태 점검·개발 스크립트와 다이어그램에 게이트웨이를 반영했다.
- 계약(Contract): 정식 카탈로그를 단일 기준으로 REST 51종과 WebSocket 14종을 UI·실행기·E2E가 함께 소비한다.
- 보안(Security): 키는 서버 전용이며 추적 응답은 마스킹한다. 영구 차단 요청은 자격 증명·요청 제한기·네트워크보다 먼저 거부한다.
- 독립 코드 리뷰: 최초 리뷰의 중요(Important) 3건인 REST 운영자 인증, 백그라운드 WebSocket 종료 처리, 안전 실연동 종료 코드 판정을 모두 테스트 우선으로 수정했다.
