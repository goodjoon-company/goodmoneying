# 시스템 트레이딩 플랫폼 P1 데이터 기반 검증

GitHub Issue: [#28](https://github.com/goodjoon-company/goodmoneying/issues/28)
검증 환경: macOS, Docker PostgreSQL 17.10, Python 3.14, Node.js 24, Chromium

## 적색(RED) 증적

- P1 계약 테스트는 최초 실행에서 시장·정책·coverage migration과 `/v1/data-foundation`이 없어 실패했다.
- 실제 PostgreSQL coverage 테스트는 초기 상태 수와 중첩 범위 처리에서 실패했고 nullable target을 `FOR UPDATE` 하던 쿼리 오류를 드러냈다.
- 시장 동기화 worker 계약은 compose·`dev.sh` 서비스가 없어 실패했다.
- Coverage & Quality 브라우저 테스트는 정책 편집 버튼과 시장별 설정이 없어 실패했다.
- 재시도 E2E는 첫 실패 뒤 기대 `retry_wait`, 실제 `failed`로 실패했다. 이후 같은 instance 중복 claim, 임대 회수 뒤 stale worker 쓰기, 부분 진행 유실, dead-letter와 coverage 불일치를 회귀 테스트로 추가했다.
- 전체 E2E의 과거 `저장 KST` 기대는 내부 UTC 계약과 충돌해 `저장 UTC`로 교체했다.

## 실제 PostgreSQL 검증 범위

- 모든 KRW 기본 정책과 신규 시장 자동 편입, 명시적 제외·일시정지 보존
- 시장 상태 이력 비중첩, 대상·job·coverage 멱등 제약
- 정책별 시작일·데이터 유형·보존·우선순위 저장과 동일 요청 무변경
- 정확한 구독 `(target_spec_id, generation)` 적용 완료
- 캔들·체결·호가·티커 실제 행의 market·사건·수신·저장·지식 시각과 checksum manifest
- 연결별 원본 수신 영수증, 동일 경제 상태 호가 스냅숏 중복 제거, 전체 30레벨과 10레벨 파생 요약의 원자 저장
- 같은 연결·프레임 순번의 동일 payload 멱등 처리와 다른 payload 충돌 rollback, 다른 연결의 동일 payload receipt 2행·snapshot 1행
- 원본 receipt·snapshot의 보존 기간 strict `<` 경계, 다중 정책 무기한 우선·최댓값, level cascade와 summary·manifest 보존
- 백필 페이지의 실제 endpoint·`to` cursor·query·요청/응답 시각·HTTP 상태·원본 JSON과 저장 batch 간 manifest 공유
- 응답 없는 전송 오류(Transport Error)의 실제 요청 cursor, 실패 감지 시각, `NULL` 상태·payload, 오류 유형·메시지 manifest
- 빈 응답의 무거래 합성 금지, 연속 캔들 batch coverage 압축
- 418 응답 차단 시간 보존, 차단 정보가 없으면 저장소 300초 기본값, 429는 1초 기본값, 재시도 대기 전 claim 금지, 예산 소진 dead-letter
- 살아 있는 lease 중복 claim 금지, 만료 회수 attempt 증가, stale owner 진행 쓰기 거부
- 부분 진행 시각·처리 구간·행 수 보존, 최종 작업 범위의 미분류 공백·`unverified → missing` 품질 이벤트와 다른 상태 보존
- 거래지원 중단·카탈로그 누락 뒤 `unavailable`, 재등장 뒤 열린 끝의 `unverified` 전이와 중복 없는 품질 이벤트
- 시장 목록의 실제 `is_details=true` 요청·원문·checksum·수집기/schema 버전 manifest와 모든 상태 이력 연결
- 정책 변경 시 해당 대상만 중지·재생성하고 다른 대상 lease·성공 coverage를 보존하는 다중 대상 수명주기
- 변경 명령 envelope의 동일 key 동시 요청 직렬화, 동일 payload 동일 결과, 다른 payload 409 충돌
- 운영 PostgreSQL·P1 schema fail-fast와 승인 SHA·최근 백업·양수 필요 용량·충분한 여유 용량을 모두 요구하는 기본 폐쇄 백필 gate
- 이미 적용된 구버전 `004`의 감사 행 삭제 가능성을 `007`이 감지하고, `008`이 기존 불완전 확인값을 제거해 백업 비교·복구 확인 전 모든 운영 런타임을 차단하는 복구 gate
- 모든 운영 worker의 최신 migration·복구 gate·실제 SQL 기준 명시·암묵 SELECT/INSERT/UPDATE/DELETE 권한 준비성 검사
- `INSERT RETURNING`, conflict target이 있는 `ON CONFLICT`, `DO UPDATE`, `UPDATE`, `DELETE RETURNING`의 기존 행 읽기 권한을 포함하고, conflict target·RETURNING·기존 행 표현이 없는 `ON CONFLICT DO NOTHING`은 SELECT를 요구하지 않음
- `SELECT ... FOR UPDATE`, `FOR NO KEY UPDATE`, `FOR UPDATE OF alias`의 잠금 대상을 UPDATE 권한으로 분류하고 `coverage_intervals` 실제 행 잠금을 제한 역할로 검증
- for-loop 테이블 바인딩과 `_table_count_between` 리터럴 호출을 확장해 `source_candles`·`ticker_snapshots`·`orderbook_summaries` 동적 FROM 대상을 직접 검증
- 실제 INSERT 대상 테이블이 소유한 identity/serial sequence만 USAGE를 요구하고, 미사용 `collection_coverage_segments`·`collection_coverage_snapshots`·`missing_ranges`·`raw_response_samples` sequence는 요구하지 않음
- 제외 시장 재활성화 시 저장 정책과 구독 세대로 정확히 한 개의 백필 작업 생성, `paused` 대상의 dead-letter 증적 보존
- P1 보존 실행이 호가 원문 receipt·snapshot·level만 삭제하고 캔들·체결·티커 원문을 보존하는 경계

- P1 전체 diff 리뷰에서 감사 행 삭제, 정책 재조정, 성공 대상 보존, 공식 파라미터, 카탈로그 원문 증적, 운영 fail-fast, 명령 멱등성, KST UI, 운영 백필 gate 누락을 발견해 TDD로 반영했다.
- 2차 전체 재리뷰에서 적용 migration 불변성, 최신 `market_event.warning/caution` 계약, 전체 worker 준비성, 시장 동기화 재시도, `paused` 대상 보존, 제외 시장 재활성화 백필, P1 보존 범위, 운영 healthcheck, OpenAPI/Pydantic 엄격성 불일치를 발견해 TDD와 실제 PostgreSQL·브라우저 E2E로 반영했다.
- 3차 전체 재리뷰에서 004 바이트 변조, NULL 삼값 논리로 열린 복구 확인 제약, 런타임 권한 대상 누락, E2E의 과거 Git 객체 의존, OpenAPI `reason` 공백 계약 누락을 발견해 TDD와 008 순방향 마이그레이션으로 반영했다.
- 4차 재리뷰에서 `RETURNING`·`ON CONFLICT`·`UPDATE`·`DELETE`의 PostgreSQL 암묵 읽기 권한과 미사용 테이블 sequence까지 요구하던 과잉 권한 검사를 발견해 정적 계약 테스트와 실제 제한 역할 PostgreSQL 테스트로 반영했다.
- 5차 재리뷰에서 행 잠금의 UPDATE 권한 누락, 무대상 `ON CONFLICT DO NOTHING`의 과잉 SELECT, f-string 동적 식별자 유실을 발견해 범위 인식 정적 분석과 실제 제한 역할 PostgreSQL 테스트로 반영했다.
- 이미 적용된 004·007은 원본 바이트를 보존하고 모든 교정은 007·008 순방향 마이그레이션에만 둔다. 구버전 004의 파괴 가능성은 복구 게이트로 명시적으로 관리한다.
- SQLite는 운영 내구성 구현이 아니라 테스트 호환 어댑터임을 Architecture·ADR·History에 명시했다.

## 미실행과 안전 경계

- 실제 운영 DB의 2024년 이후 KRW 전체 백필은 P7 용량·백업·처리율 gate와 P8 승인 SHA 배포 전 실행하지 않았다.
- 실제 주문·출금·실거래 활성화는 실행하지 않았다.
- 원격 CI run과 최종 SHA는 push 뒤 이 문서의 후속 closeout에 기록한다.
