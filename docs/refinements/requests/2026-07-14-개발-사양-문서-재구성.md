# 개발 사양 문서 재구성 요청

Date: 2026-07-14
Source Request: 사용자 요청
Related Issue: [#15](https://github.com/goodjoon-company/goodmoneying/issues/15)
Related Product: `docs/01_Product.md`의 P0 문서·인계 기준 및 P1·P2·P2.1 현재 제품화 범위

## 목적

제품(PRD, Product Requirements Document), 아키텍처(Architecture), 사용 안내, 계약(Contract)을 개발자가 구현 전후에 같은 순서로 탐색할 수 있는 개발 사양(Development Specification)으로 정리한다. 문서 구조를 정리하는 작업이며 제품 기능, DB·HTTP·WebSocket 계약, 런타임 코드는 바꾸지 않는다.

## 설계 원칙

- 제품 목표·범위·요구사항의 단일 기준은 계속 `docs/01_Product.md`다.
- 시스템 경계·모듈 책임·흐름은 `docs/02_Architecture.md`와 `docs/02_Architecture/`에 두고, 정확한 스키마(schema)는 `docs/contracts/`로 연결한다.
- 계약 문서는 기계 검증 파일을 복제하지 않고 소비자, 변경 순서, 검증 위치를 안내한다.
- 사용 안내는 개발 환경 준비, 프로세스 실행, 화면별 데이터 준비, 장애 확인 순서로 설명한다.
- Mermaid 다이어그램은 현재 구현된 단일 인스턴스·세 워커·PostgreSQL·업비트 경계를 표현하며, 미래 후보 아키텍처를 현재 구현처럼 그리지 않는다.

## 범위

1. `docs/README.md`에 개발 사양 탐색 순서와 문서 책임 지도를 추가한다.
2. 제품 문서를 목표, 범위, 요구사항, 수용 기준, 로드맵을 빠르게 찾을 수 있게 정리한다.
3. 아키텍처 문서에 시스템 문맥, 소프트웨어 구성, 수집·분석 데이터 흐름 Mermaid 다이어그램을 추가한다.
4. 업비트 수집 파이프라인 모듈 문서를 책임, 경계, 런타임 흐름, 실패·복구 기준 중심으로 정리한다.
5. 수집 운영·코인 분석 안내를 공통 실행 절차와 화면별 사용 절차로 정리한다.
6. 계약 루트·API·DB 안내에 계약 소비자와 검증 경로를 명시한다.

## 비범위

- `openapi.yaml`, DB schema, WebSocket JSON Schema의 의미 변경
- API 경로, 데이터 모델, 워커 동작, UI 동작 변경
- ADR 결정 변경 또는 새 아키텍처 결정 추가
- 완료된 과거 검증 증적의 재작성

## 완료 기준(Acceptance Criteria)

1. 개발자는 `docs/README.md`에서 제품 → 아키텍처 → 계약 → 사용·검증 문서의 읽는 순서를 확인할 수 있다.
2. `docs/02_Architecture.md`는 현재 시스템 문맥, 소프트웨어 구성, 수집·집계·분석 흐름을 Mermaid 다이어그램으로 보여 준다.
3. 제품, 모듈, 사용 안내, 계약 문서는 서로의 책임을 복제하지 않고 정확한 단일 기준 파일을 연결한다.
4. 모든 새 링크는 저장소 내부 경로를 가리키며, 문서 변경 검증과 인계 요약을 각각 `docs/Test/`, `docs/History/`에 남긴다.
