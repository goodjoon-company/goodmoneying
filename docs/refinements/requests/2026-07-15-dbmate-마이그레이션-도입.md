# dbmate 버전 DB 마이그레이션 도입

Date: 2026-07-15
Status: Accepted
Source Request: 앞으로 모든 DB 스키마 변경을 dbmate로 관리한다.
Related Issue: [#16](https://github.com/goodjoon-company/goodmoneying/issues/16)
Related ADR: [ADR-0010](../../ADR/ADR-0010-dbmate-버전기반-DB-마이그레이션.md)

## 사용자 가치와 목표

- 개발자는 `dev.sh`를 통해 미적용 DB 변경을 누락하지 않고 앱을 시작한다.
- 운영자는 버전, 적용 순서와 실패 지점이 기록된 DB 변경만 배포한다.
- API와 워커는 스키마 변경 책임과 권한을 갖지 않는다.
- 기존 `goodmoneying-dev` 데이터와 운영 배포 경계를 보존하며 버전 관리로 전환한다.

## 기능 설계

### 변경 이력과 스냅샷

- `docs/contracts/db/migrations/<timestamp>_<description>.sql`이 사람이 작성하고 리뷰하는 변경 이력이다.
- `docs/contracts/db/schema.sql`은 명시적 migrate·dump 또는 CI가 생성하는 현재 결과이며 직접 수정하지 않는다.
- 최초 `20260715000100_initial_schema.sql`은 현재 전체 스키마를 `migrate:up`에 포함하고 파괴적 기준선 `down`은 구현하지 않는다.
- 이미 적용된 파일은 수정하지 않는다. 수정이 필요하면 새 타임스탬프 파일을 추가한다.

### 개발 흐름

| 명령 | 동작 |
|---|---|
| `./dev.sh db new <설명>` | dbmate 형식의 새 SQL 파일 생성 |
| `./dev.sh db migrate` | `DBMATE_STRICT=true`로 미적용 변경 실행 후 결정적 스키마 덤프 생성 |
| `./dev.sh db status` | DB별 적용·미적용 상태 출력 |
| `./dev.sh db dump` | 현재 DB를 `schema.sql`로 덤프 |
| `./dev.sh db rollback` | 개발자가 명시한 경우 최근 안전한 `down` 실행 후 덤프 |
| `./dev.sh app start ...` | `DBMATE_STRICT=true`와 `--no-dump-schema migrate` 성공 후 기존 앱 시작 흐름 실행 |

- dbmate는 루트 npm 개발 의존성으로 정확한 버전을 고정하고 `node_modules/.bin/dbmate`를 직접 실행한다.
- DB URL은 기존 `GOODMONEYING_DATABASE_URL`을 그대로 사용하며 비밀값을 로그에 출력하지 않는다.
- dbmate, DB URL 또는 연결이 없거나 마이그레이션이 실패하면 앱을 시작하지 않는다.

### 운영 배포

1. CI가 `apps/migrations/Dockerfile`로 `goodmoneying-migrations:{release-tag}`를 빌드·배포한다.
2. prod-home 배포가 infra PostgreSQL을 먼저 확인하고 앱 Compose 파일과 이미지를 준비한다.
3. 앱 서버가 `app.env`의 DB URL로 마이그레이션 컨테이너를 일회성 실행한다.
4. 실패하면 기존 API·워커를 유지한 채 배포를 중단한다.
5. 성공하면 같은 release 태그의 API·워커를 갱신하고 healthcheck와 Playwright E2E를 실행한다.

DB 서버에는 dbmate 설치나 마이그레이션 파일 복사가 필요 없다. PostgreSQL 접속 허용, 마이그레이션 권한과 적용 전 백업만 필요하다.

## 오류 처리와 안전 원칙

- 기준선 적용 전 DB 논리 덤프, 테이블 수, 주요 행 수와 최신 시각을 기록한다.
- 기존 DB 기준선 적용 전후 데이터 값이 달라지면 완료하지 않는다.
- 운영 자동 롤백은 제공하지 않는다. 실패한 앱 배포는 이전 앱 이미지로 되돌리고 DB는 호환 상태를 유지한다.
- 컬럼 제거·타입 축소는 최소 두 번의 배포로 나누며, 먼저 새 구조를 추가하고 데이터 전환 뒤 사용이 제거된 구조만 별도 삭제한다.
- 오래 걸리는 DDL은 잠금과 트랜잭션 가능 여부를 검토하고 필요하면 `transaction:false`를 명시한다.

## 비범위

- SQLAlchemy 또는 ORM 도입
- 웹 화면 변경
- 이번 기준선 외의 제품 DB 구조 변경
- 운영 배포의 자동 `dbmate rollback`
- 이번 작업에서 전용 DB 역할을 새로 발급하는 보안 분리

## 승인된 결정

- 개발 앱 시작 전 자동, 운영 배포 전 명시적 마이그레이션
- 운영 실행 주체는 앱 서버의 전용 불변 마이그레이션 이미지
- 운영은 백업과 순방향 수정, 개발만 안전한 `down` 허용

## 인수 조건(Acceptance Criteria)

- 빈 PostgreSQL과 기존 개발 DB가 같은 기준선 버전에 도달한다.
- 기존 데이터와 최신 시각이 보존된다.
- API·워커는 DDL을 실행하지 않는다.
- 개발 명령과 실패 차단이 자동 테스트된다.
- prod-home dry-run에서 migrate 성공이 app `up -d`보다 앞선다.
- CI가 빈 DB 적용, 재적용 멱등성, 상태와 스냅샷을 검증한다.
- 실제 PostgreSQL 마이그레이션 뒤 API 상태와 대시보드 요청이 성공한다.
- 모든 검증 성공 후 README의 `dev.sh` 사용법에 migration 가이드가 추가된다.

## 테스트·QA 전략

- RED/GREEN 계약 테스트로 디렉터리, 기준선 지시자, dbmate 버전, 런타임 DDL 제거를 검증한다.
- `dev.sh` 가짜 실행기 테스트로 성공·실패·비밀값 미노출을 검증한다.
- 배포 dry-run과 GitHub Actions 계약 테스트로 이미지 빌드·migrate→app 순서를 검증한다.
- 빈 PostgreSQL에 전체 마이그레이션을 실행하고 테이블·제약·KST·버전·재실행을 검증한다.
- 기존 개발 DB는 백업과 데이터 지표 비교 뒤 기준선을 적용한다.
- 마이그레이션 완료 DB에서 실제 API `/health`, `/v1/dashboard/summary`를 자동 검증한다.
- 프론트엔드 코드는 변경하지 않으므로 디자인 QA는 미적용하고 기존 Playwright E2E로 회귀를 확인한다.
