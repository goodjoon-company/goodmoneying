## 목표와 사용자 가치

개발자는 업비트 API 테스트 작업대에서 공식 파라미터 의미와 제한을 별도 문서 왕복 없이 이해하고, 유효한 경계값을 빠르게 채우며, 날짜·시간대·숫자 정밀도·배열 개수·필드 조합 오류를 업비트 호출 전에 발견한다. Quotation과 Exchange 화면은 같은 계약 메타데이터와 입력 동작을 사용해 기능별 폼 차이를 없앤다.

## 핵심 시나리오

1. 개발자가 캔들 `to`의 달력과 시계에서 KST 날짜·시·분·초를 선택하거나 `현재 시각(Now)`을 눌러 클릭 시점 값을 채운다.
2. 작업대는 표시값이 KST임을 알리고 요청 미리보기와 게이트웨이 전송값을 시간대가 포함된 ISO 8601로 직렬화한다.
3. 개발자가 `count`, `limit`, `page`, `days_ago` 같은 정수 입력에서 공식 최소·기본·최대와 단위를 확인하고 최소·기본·최대 빠른 입력을 사용한다.
4. 개발자가 `start_time`과 `end_time`을 입력하면 선후 관계와 최대 7일 같은 기능별 기간 제한을 즉시 확인한다.
5. 배열 입력은 최대 항목 수와 허용값을 표시하고, 초과·중복·빈 요소를 실행 전에 설명 가능한 오류로 차단한다.
6. 주문 가격·수량처럼 숫자 형식 문자열(Number String)은 정밀도를 보존하며, 페어·계정에 따라 변하는 제한은 주문 가능 정보와 호가 정책 같은 동적 정책 출처를 표시한다.
7. 입출금 목록의 `from`·`to`처럼 날짜가 아닌 페이지네이션 커서(Cursor)는 텍스트 입력으로 유지한다.

## 기능 설계

### 계약 메타데이터

`docs/contracts/upbit/upbit-api-catalog.yaml`을 단일 기준(source of truth)으로 두고 파라미터에 필요한 경우 다음 메타데이터를 제공한다.

- `default`: 공식 미입력 기본값
- `minimum`, `maximum`, `step`: 정적 숫자 범위와 증분
- `max_items`, `unique_items`: 배열 개수와 중복 정책
- `format`: `date-time`, `time`, `cursor`, `decimal-string` 같은 입력 의미
- `timezone`: 표시 또는 공식 의미의 시간대
- `range_with`, `range_max_seconds`: 시작·종료 필드 관계와 최대 기간
- `dynamic_constraint_source`: 페어·계정 상태에 따라 달라지는 제한을 조회할 카탈로그 기능
- `description`, `unit`: 화면 안내에 필요한 짧은 의미와 단위

공식 문서가 최소값을 명시하지 않은 경우 임의로 `0`이나 `1`을 만들지 않는다. 정적 최소·최대가 아니라 지원값 목록인 파라미터는 `enum` 또는 동적 선택기를 사용한다.

### 공통 입력 동작

- 날짜·시간은 달력과 시계가 포함된 네이티브 입력을 공통 래퍼로 제공하고 `step=1`로 초 단위를 허용한다.
- `현재 시각(Now)`은 클릭 시점의 KST 날짜·시·분·초를 채운다. 선택 파라미터에는 `지우기`를 제공한다.
- 도우미 문구는 레이블과 연결된 `aria-describedby`로 최소·기본·최대·단위·시간대·기간·배열 제한을 제공한다.
- 숫자 입력은 알려진 제약만 `min`, `max`, `step`에 적용하고 `최소`, `기본`, `최대` 빠른 입력 버튼을 조건부 표시한다.
- 숫자 형식 문자열은 브라우저 `Number`로 바꾸지 않고 원문 문자열을 유지하며 숫자 키보드를 유도한다.
- 제출 시 필드별 오류를 표시하고 첫 오류로 포커스를 이동하며, 오류가 있으면 업비트 게이트웨이를 호출하지 않는다.

### 공식 제약 반영 범위

- 캔들 `count`: 기본 1, 최소 1, 최대 200. `to` 미입력은 현재 시각이며 입력값 이전을 조회한다.
- 체결 `count`: 기본 1, 최소 1, 최대 500. `days_ago`: 1~7.
- 호가 `count`: 기본 30, 최소 1, 최대 30. `level`은 숫자 형식 문자열이며 호가 정책 조회를 동적 제한 출처로 사용한다.
- 체결 대기 주문: `page` 기본·최소 1, `limit` 기본 100·최대 100.
- 종료 주문: 최대 조회 기간 7일, `limit` 기본 100·최대 1000.
- 포켓 자산 이전 목록: 최대 조회 기간 7일, `limit` 기본 20·최대 100, UUID·identifier 배열 최대 20개.
- 입출금 목록: `limit` 기본 100·최대 100, `page` 기본·최소 1, UUID·TXID 배열 최대 100개. `from`·`to`는 날짜가 아닌 UUID 커서다.
- 주문 가격·수량·최소 주문 금액·호가 단위는 정적 하드코딩하지 않고 `rest.available-order-information`과 `rest.list-orderbook-instruments`를 동적 제약 출처로 연결한다.
- 자산 이전·입출금 `amount`는 숫자 형식 문자열로 입력하되 정적 제한을 추정하지 않는다. 디지털 자산 출금 수량은 출금 가능 정보 조회를 동적 제한 출처로 연결한다.
- ID 주문 목록 조회의 UUID·identifier 배열은 최대 100개, ID 주문 목록 취소는 최대 20개다. 일괄 취소 `count`는 기본 20·최대 300이며 페어·제외 페어는 각각 최대 20개다.

## 비범위

- 상태 변경 API의 안전 등급과 업비트 상향 차단 정책 변경
- 실제 주문·취소·자산 이전·입출금 실행 허용
- 외부 날짜 선택기 또는 폼 라이브러리 도입
- 공식 문서에 없는 최소값·최대값 추정

## 인수 조건(Acceptance Criteria)

1. 모든 `format: date-time` 입력에 달력·시계, KST 안내, `현재 시각(Now)`, 선택값 비우기가 보인다.
2. `현재 시각(Now)`은 고정 시계 테스트에서 KST 날짜·시·분·초를 채우고 요청에는 동일 순간의 ISO 8601 값이 전달된다.
3. 숫자 입력은 공식 제약이 있는 경우 화면 문구와 HTML `min`·`max`·`step`이 일치하고 최소·기본·최대 빠른 입력이 동작한다.
4. 공식 기본값과 경계값 테스트가 구분되며, 캔들 `count`의 초기값은 공식 기본값 1이다.
5. 기간 역전, 최대 기간 초과, 숫자 범위 초과, 배열 최대 개수 초과는 게이트웨이 호출 없이 차단된다.
6. 입출금 `from`·`to`는 텍스트 커서 입력이고 날짜 선택기가 아니다.
7. Quotation과 Exchange가 같은 제약 표현·직렬화·검증 규칙을 사용한다.
8. 390px와 데스크톱에서 필드·도우미·빠른 입력이 겹치지 않고 키보드만으로 조작할 수 있다.
9. 계약 테스트, 프론트엔드 단위·통합 테스트, 실제 개발 서버 자동화 E2E, 전체 회귀 검증이 통과한다.

## 결정과 가정

- 사용자는 `현재 시각(Now)`을 KST 기준 날짜·시·분·초 전체 입력으로 승인했다.
- 구현은 외부 UI 라이브러리 없이 현재 디자인 시스템과 네이티브 날짜·시간 입력을 확장한다.
- 공식 문서가 문자열로 정의한 숫자 형식 값은 계약 전송 타입을 바꾸지 않고 화면 의미만 강화한다.
- 기존 캔들 `count=200` 자동 초기화 기대값은 새 요구사항에 따라 공식 기본값 `1`로 변경한다.
