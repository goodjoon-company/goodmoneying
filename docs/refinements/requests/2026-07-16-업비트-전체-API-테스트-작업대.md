## 목표와 성공 기준

개발자는 하나의 작업대에서 업비트 공식 시세 조회(Quotation) REST, 거래·자산 관리(Exchange) REST, 공개·비공개 WebSocket 기능을 선택하고 요청 파라미터, 시각화 결과, 실제 원본 요청·응답을 연결해 추적한다. 브라우저의 `Origin` 요청 제한과 키 노출 문제를 제거하기 위해 별도 서버 모듈이 업비트 연결을 전담한다.

성공 기준은 다음과 같다.

1. 공식 기능 개요와 `llms.txt`에 수록된 현재 REST·WebSocket 기능이 카탈로그에 누락 없이 존재한다.
2. 사이드바의 `업비트 API 테스트` 아래에 `Quotation API`, `Exchange API`, `WebSocket API` 2레벨 메뉴가 있고, 화면 상단에는 페어·캔들·체결·현재가·호가 등 기능 그룹 탭이 있다.
3. 좌측 요청 패널은 공통 페어·자산과 기능별 파라미터를 타입에 맞는 선택기·날짜 입력·숫자 입력·직접 입력으로 제공한다.
4. 우측 결과 패널은 캔들 차트, 체결·자산·주문 목록, 현재가 카드, 호가 사다리 등 응답 종류에 맞는 시각화를 제공한다.
5. 모든 시각화에는 추적 아이콘이 있으며 레이어 팝업에서 API 기능, 메서드·경로, 마스킹된 파라미터, 상태, 제한 헤더, 소요 시간, 원본 응답을 확인한다.
6. 캔들처럼 시간 커서를 지원하는 결과는 화면 가장자리 이동 시 다음 페이지를 계속 읽고 중복 제거·뷰포트 보존·끝 감지를 수행한다.
7. 키·JWT·Authorization 헤더는 브라우저, 응답, 로그, 테스트 스냅샷, Git에 나타나지 않는다.
8. 실제 주문·취소·자산 이전·입출금·트래블룰 검증은 실행하지 않는다. 주문은 `POST /v1/orders/test`로만 정상·오류 응답을 검증한다.

## 공식 API 범위

### Quotation REST

- 페어: 목록 조회
- 캔들: 초·분·일·주·월·연
- 체결: 페어 체결 이력
- 현재가: 페어 단위·마켓 단위
- 호가: 호가·호가 정책

### Exchange REST

- 포켓·자산: 포켓 정보, 포켓별 키, 포켓 잔고, 자산 이전 내역, 포켓 잔고
- 주문: 주문 가능 정보, 주문 생성 테스트, 개별·목록·체결 대기·종료 주문 조회
- 출금: 가능 정보, 허용 주소, 개별·목록 조회
- 입금: 가능 정보, 주소, 개별·목록 조회
- 트래블룰: 지원 거래소 조회
- 서비스: 입출금 상태, API Key 목록
- 상태 변경 API도 카탈로그와 입력 화면에는 표시하되 `blocked` 안전 등급으로 두고 업비트 상향 호출을 금지한다.

### WebSocket

- 공개: `ticker`, `trade`, `orderbook`, `candle.1s/1m/3m/5m/10m/15m/30m/60m/240m`
- 비공개: `myAsset`, `myOrder`
- 연결 운용: `LIST_SUBSCRIPTIONS`, 연결·구독·일시정지·해제·재연결, 네 가지 응답 포맷

## 요청 제한 결론

업비트 공식 요청 제한 문서에 따르면 `Origin` 헤더가 포함된 Quotation REST와 공개 WebSocket은 10초당 1회다. 일반 서버 요청은 Quotation 기능 그룹별 IP당 초당 10회이며, Exchange `default`는 포켓당 초당 30회, `order`와 `order-test`는 초당 8회, 일괄 취소는 2초당 1회다. WebSocket 연결은 초당 5회, 데이터 요청은 연결당 초당 5회이면서 분당 100회다.

따라서 브라우저 직접 호출을 폐기하고, 업비트 상향 요청에 브라우저 `Origin`을 전달하지 않는 별도 게이트웨이를 사용한다. 게이트웨이는 한도를 없애는 프록시가 아니라 그룹별 제한과 `Remaining-Req`, HTTP 429·418 처리를 중앙화하는 단일 호출 경계다.

## 설계 대안과 결정

### 대안 A — 브라우저 직접 호출 유지

- 장점: 서버 구현이 가장 작다.
- 단점: 10초당 1회, 비공개 WebSocket 인증 헤더 제약, 브라우저 키 노출 위험 때문에 전체 기능 요구를 충족하지 못한다.

### 대안 B — 기존 운영 서버에 업비트 중계 결합

- 장점: 프로세스 수가 늘지 않는다.
- 단점: 제품 데이터 API와 개발용 외부 API 시험·키·요청 제한 책임이 결합되고 장애·보안 경계가 흐려진다.

### 대안 C — 별도 업비트 API 게이트웨이

- 장점: 키와 업비트 연결, 그룹별 제한, 안전 정책, 추적 정보를 한 경계에 모으고 브라우저·운영 서버와 독립적으로 검증·배포할 수 있다.
- 단점: 개발 스크립트와 배포 구성에 프로세스 하나가 추가된다.
- 결정: 대안 C를 채택한다.

## 구성과 데이터 흐름

- `apps/upbit_gateway/`: 허용 목록 기반 REST 실행, JWT 인증, 그룹별 제한, WebSocket 중계, 응답 추적, 안전 정책을 담당한다.
- `docs/contracts/upbit/`: 공식 기능 카탈로그와 파라미터·안전 등급의 단일 기준이다. 게이트웨이가 이 계약을 읽고 프론트엔드에는 카탈로그 API로 제공한다.
- `apps/web/src/components/upbit-api-test/`: 2레벨 메뉴, 탭, 동적 파라미터 폼, 결과 렌더러, 추적 팝업을 담당한다.
- 브라우저는 키를 알지 못하고 게이트웨이의 HTTP·WebSocket만 사용한다.
- 게이트웨이는 임의 URL을 받지 않고 카탈로그의 `endpoint_id`만 실행한다.

## 안전 정책

| 등급 | 처리 |
|---|---|
| `read` | 업비트 조회 API를 실행한다. |
| `test` | 공식 `POST /v1/orders/test`만 실행한다. |
| `blocked` | 입력·최종 요청 미리보기와 로컬 정책 오류를 반환하고 업비트에는 전송하지 않는다. |

`POST /orders`, 모든 주문 취소, 자산 이전, 출금·입금 생성·취소, 트래블룰 검증은 `blocked`다. 요청 제한 위반을 의도적으로 유발하는 429·418 부하 테스트도 수행하지 않는다.

## 오류와 추적 계약

- REST 응답은 `trace_id`, `endpoint_id`, `request`, `response`, `rate_limit`, `duration_ms`, `received_at`을 가진 추적 봉투(Trace Envelope)로 반환한다.
- 요청의 인증 정보와 민감 필드는 값 대신 `<redacted>`를 반환한다.
- WebSocket은 연결 상태, 구독 요청, 원본 프레임, 오류를 동일한 `trace_id` 계열로 연결한다.
- 네트워크·인증·권한·검증·정책 차단·업비트 오류를 서로 다른 오류 코드로 표시한다.

## 테스트·QA 전략

- 계약: 카탈로그의 전체 기능 목록, 메서드·경로·요청 제한 그룹·안전 등급을 공식 문서 기준으로 검사한다.
- TDD: JWT HS512·nonce·query hash, 키 마스킹, 허용 목록, 안전 차단, 그룹별 제한을 RED/GREEN으로 구현한다.
- 통합: 가짜 업비트 HTTP·WebSocket 서버로 정상·400·401·429·418·형식 오류를 검증한다.
- 프론트엔드: 공통 파라미터 전파, 타입별 입력, 렌더러 선택, 추적 팝업, 캔들 추가 조회와 뷰포트 보존을 검증한다.
- E2E: Quotation, 인증 읽기, 주문 테스트, 상태 변경 차단, 공개·비공개 WebSocket, 원본 추적, 390px·데스크톱 접근성을 Playwright로 검증한다.
- 실제 연동: 공개 조회와 인증 읽기, 주문 생성 테스트만 실행한다. 키 권한이나 허용 IP 때문에 실패하면 정확한 오류 응답을 증적으로 남기며 실제 상태 변경 API는 호출하지 않는다.
