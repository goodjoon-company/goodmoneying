## 선택한 접근과 대안

선택: 기존 모놀리식 저장소와 PostgreSQL을 유지하면서 역할별 worker와 도메인 경계를 확장하고 P0~P8 수직 기능 조각으로 배포한다.

대안 1은 전면 재작성이다. 과거 계약 충돌은 줄지만 검증된 수집·Gateway·배포 기반과 기존 데이터를 잃을 위험이 커서 기각했다. 대안 2는 초기부터 Kafka·시계열 DB·별도 연구 서비스를 도입하는 방식이다. 확장성은 높지만 현재 처리량 근거가 없어 ADR-0012에 따라 보류했다.

## 명시 가정

- 단일 운영자 인증 모델을 유지하되 명령별 권한을 분리한다.
- 실거래 코드는 P6에서 구현할 수 있지만 live activation은 이 작업 범위 밖의 별도 운영 승인이다.
- 운영 secrets가 없으면 private·주문 준비 검증만 비활성화하고 다른 기능과 모의 adapter를 완료한다.
- 공식 API가 과거 데이터를 제공하지 않는 구간은 `unavailable`로 완료 처리한다.
- 운영 DB의 파괴적 변환은 백업·복구 리허설 전까지 수행하지 않는다.

- 개발 DB: migration 1 applied·0 pending, 약 688MB. 운영 DB: PostgreSQL healthy, 약 17GB, 운영 image는 `release-1db7d40`이며 새 gateway·aggregation worker는 없다.
- 공식 market API 실측: 2026-07-17 전체 783개, KRW 269개, BTC 307개, USDT 207개다. 이 숫자는 정책 상한이 아니라 시점 snapshot이다.
- 2024-01-01부터 2026-07-17까지 1분 slot 상한은 market당 1,336,320개다. REST count 200 기준 KRW 269개 전체의 이론상 최소 요청은 1,797,458회, candle 10 req/s를 전용으로 완전 사용해도 약 49.9시간이다.
- 원천 candle 상한은 약 359,470,080행이다. 행·index·TOAST를 포함한 실제 byte를 P1 spike에서 측정하기 전 계획 범위는 row당 250~500B와 index 1.5~2배를 적용한 약 135~360GB다. 따라서 운영 17GB DB에 바로 기본 정책을 활성화하지 않고 disk 여유, partition·index query plan, backfill throttle과 보존 정책을 검증한다.
- 실운영 backfill 목표는 Quotation 다른 consumer 여유와 retry를 포함해 최소 3~5일로 잡고 실제 처리량·응답 빈 구간·DB write latency로 재산정한다.
