## 배경

기존 `docs/01_Product.md`는 최종 제품 비전은 넓게 담고 있었지만, 1차 MVP(Minimum Viable Product)에 국내 주식, 해외 주식, 업비트(Upbit), 가격/거래량, 수급, 공매도(Short Selling), 재무지표, 뉴스, 공시, 리포트, 호가(Orderbook), LLM(Large Language Model) 신호까지 포함해 초기 범위가 과도했다.

리뷰 결과 다음 문제가 확인됐다.

- MVP가 너무 넓어 첫 구현 단위로 실행하기 어렵다.
- "24시간 365일 국내/해외 시장 호가 수집" 표현은 주식시장 운영시간과 맞지 않는다.
- 프론트엔드(Frontend) 화면이 마일스톤 완료 조건에 포함되어 있지 않아 화면 구현이 뒤로 밀릴 수 있다.
- 업비트 중심 MVP로 좁히면 국내/해외 주식 요구사항이 누락될 위험이 있다.

## 결정한 방향

MVP는 업비트 KRW 마켓 데이터 파이프라인과 운영 화면으로 좁힌다. 최종 제품은 원래 요구사항인 코인, 국내 주식, 미국 주식, 뉴스/공시/리포트, LLM 신호, 통합 분석, 전략, 봇(Bot), 시뮬레이션(Simulation), 모의매매(Paper Trading), 실거래(Live Trading)를 모두 포함한다.

마일스톤(Milestone)은 데이터 능력과 프론트엔드 화면을 함께 묶는 제품 단위 세로 절편으로 재수립한다. 백엔드 데이터 수집만 끝난 상태나 빈 화면만 만든 상태는 마일스톤 완료로 보지 않는다.

## MVP 범위

MVP는 M1부터 M3까지로 정의한다.

### 포함 범위

- 업비트 KRW 마켓 24시간 거래대금 기준 상위 100개 후보를 표시한다.
- 시스템은 상위 100개 후보 중 기본 50개를 자동 체크한다.
- 사용자는 상위 100개 범위 안에서 체크를 변경해 최종 수집 대상 50개를 조정할 수 있다.
- 수집 데이터는 캔들(Candle), 현재가, 거래대금, 호가 요약(Orderbook Summary)으로 제한한다.
- 호가 요약은 최우선 매수/매도 가격과 수량, 스프레드(Spread), 상위 5호가 또는 10호가 누적 매수/매도 잔량, 호가 불균형(Imbalance), 갱신 시각, 수집 지연을 포함한다.
- 데이터 품질(Data Quality)은 성공/실패, 지연, 결측, 중복, 최신성, 저장량을 포함한다.
- 첫 화면은 파이프라인 건강도 중심 운영 상태 대시보드다.
- 추가 MVP 화면은 수집 대상/설정 화면, 시장 리스트 화면, 코인 상세 화면이다.

### 제외 범위

- 국내 주식과 미국 주식 수집
- 뉴스, 공시, 증권사 리포트 수집
- LLM 요약과 구조화 신호(Signal)
- 호가 원천 스냅샷(Snapshot) 전체의 고빈도 장기 보존
- 호가 원천 재생(Replay) 기반 정밀 백테스트(Backtest)
- 시장 스캐닝(Scanning), 전략 설계, 봇 설계, 모의매매, 실거래

제외 범위는 버리는 요구사항이 아니라 후속 마일스톤으로 이동하는 요구사항이다.

## 수집 운영 정책

시스템 프로세스는 24시간 365일 상시 실행된다.

코인 데이터는 업비트 시장 특성에 맞춰 24시간 365일 지속 수집한다. 국내 주식과 미국 주식은 시스템이 상시 실행되더라도 각 거래소의 정규장, 지원 가능한 확장장, 휴장일, 데이터 공급원 제약에 맞춰 수집 가능 시간에만 지속 수집한다. 수집 가능 시간이 아닌 구간은 오류가 아니라 시장 휴장 또는 수집 대기 상태로 기록한다.

MVP의 정상 수집 판정은 실용적 운영 기준으로 둔다.

- 현재가와 거래대금은 최근 1분에서 3분 안에 갱신되면 정상으로 본다.
- 캔들은 선택한 단위의 마지막 완성 캔들이 저장되면 정상으로 본다.
- 호가 요약은 최근 1분에서 3분 안에 갱신되면 정상으로 본다.

## 프론트엔드 화면 로드맵

제품 문서에는 기술 스택을 못박지 않고, 마일스톤별 화면과 사용자 경험(UX, User Experience) 요구사항을 기록한다. React, Next.js, Streamlit 같은 기술 선택은 아키텍처(Architecture) 문서 또는 ADR(Architecture Decision Record)에서 결정한다.

| 단계 | 화면 | 사용자가 확인하는 것 |
|---|---|---|
| M1 | 운영 상태 대시보드 | 수집이 정상인지, 어떤 코인이 지연/결측/실패 상태인지, 저장량과 최신성이 어떤지 확인한다. |
| M2 | 수집 대상/설정 화면 | 상위 100개 후보 중 기본 50개 체크 상태를 보고 최종 수집 대상 50개를 조정한다. |
| M2 | 시장 리스트 화면 | 수집 대상 코인의 현재가, 거래대금, 등락률, 캔들/호가 최신성, 품질 상태를 표로 탐색한다. |
| M3 | 코인 상세 화면 | 단일 코인의 캔들 차트, 현재가/거래대금 추이, 호가 요약, 수집 품질 이력을 확인한다. |
| M4 | 국내 주식 리스트와 상세 화면 | 국내 주식의 가격/거래량, 시가총액, 수급, 공매도, 재무지표와 품질 상태를 확인한다. |
| M5 | 미국 주식 리스트와 상세 화면 | 미국 주식의 가격/거래량, 시가총액, 재무지표와 품질 상태를 확인한다. |
| M6 | 신호/이벤트 타임라인 | 뉴스, 공시, 리포트, LLM 신호를 가격 흐름과 같은 시간축에서 비교한다. |
| M7 | 통합 분석과 시장 스캐닝 화면 | 코인과 주식을 같은 기준으로 비교하고 후보를 발굴한다. |
| M8 | 전략 작업대 | 조건 조합, 백테스트 결과, 지표별 기여를 비교한다. |
| M9 | 봇 설계/시뮬레이션 화면 | 봇 설정, 전략 파이프라인(Pipeline), 포지션(Position), 손익, 판단 근거를 시뮬레이션한다. |
| M10 | 모의매매/실거래 준비 화면 | 주문 전 점검, 리스크 한도, 감사(Audit) 로그, 운영 상태를 확인한다. |

## 재수립 마일스톤

| 단계 | 목표 | 완료 기준 |
|---|---|---|
| M0 | 제품/계약/아키텍처 기준 정리 | 제품 범위, 업비트 MVP 범위, 시장별 확장 순서, 계약(Contract) 위치, Task/Test/History 구조가 정리된다. |
| M1 | 업비트 수집 운영 MVP | 업비트 KRW 상위 100개 후보 중 기본 50개 수집 대상의 캔들, 현재가, 거래대금, 호가 요약을 수집/저장하고 운영 상태 대시보드에서 최신성, 지연, 결측, 실패, 저장량을 확인한다. |
| M2 | 업비트 수집 대상 관리와 시장 리스트 | 사용자가 상위 100개 안에서 수집 대상 50개를 조정하고, 시장 리스트에서 현재가, 거래대금, 등락률, 품질 상태를 탐색한다. |
| M3 | 업비트 코인 상세와 호가 원천 저장 | 단일 코인의 캔들 차트, 현재가/거래대금 추이, 호가 요약, 품질 이력을 확인하고 호가 원천 스냅샷 저장을 시작한다. |
| M4 | 국내 주식 데이터 기반 | 국내 주식의 가격/거래량, 시가총액, 수급, 공매도, 재무지표 수집/저장/품질 확인을 추가하고 국내 주식 리스트와 상세 화면을 제공한다. |
| M5 | 미국 주식 데이터 기반 | 미국 주식의 가격/거래량, 시가총액, 재무지표와 가능한 범위의 시장 데이터를 수집/저장/품질 확인하고 미국 주식 리스트와 상세 화면을 제공한다. |
| M6 | 뉴스/공시/리포트와 LLM 신호 | 코인, 국내 주식, 미국 주식의 뉴스, 공시, 리포트를 수집하고 LLM 요약과 구조화 신호를 생성해 이벤트 타임라인 화면에서 가격 흐름과 함께 본다. |
| M7 | 통합 분석과 시장 스캐닝 | 코인과 주식 데이터를 같은 분석 기준으로 비교하고, 시장 스캐닝, 관심 대상 심층 분석, 신호 영향 분석 화면을 제공한다. |
| M8 | 전략 설계와 백테스트 | 가격/거래량, 수급, 공매도, 재무지표, 호가, LLM 신호를 조합해 전략 조건을 만들고 백테스트 화면에서 검증한다. |
| M9 | 봇 설계와 시뮬레이션 | 봇별 투자 금액, 대상 시장, 리스크 한도, 전략 파이프라인을 설정하고 실제 주문 없이 판단과 손익을 시뮬레이션한다. |
| M10 | 모의매매와 실거래 준비 | 모의매매, 주문 계약, 리스크 제한, 감사 로그, 실거래 전 점검 화면을 갖춘다. |

## 문서 반영 계획

`docs/01_Product.md`를 다음 방향으로 수정한다.

- 1차 MVP 범위를 업비트 KRW 마켓 중심으로 축소한다.
- 기존 MVP 요구사항 중 국내 주식, 미국 주식, 뉴스/공시/리포트, LLM, 전략, 봇 항목은 후속 마일스톤으로 이동한다.
- 호가 요구사항은 MVP 호가 요약과 M3 호가 원천 스냅샷으로 분리한다.
- 시스템 24/365 실행 정책과 시장별 수집 가능 시간 정책을 분리해 기록한다.
- 요구사항 표에 MVP 포함 여부와 후속 마일스톤을 명시한다.
- 로드맵을 M0부터 M10까지 재작성하고 각 단계의 프론트엔드 화면을 완료 기준에 포함한다.

## 검증 계획

문서 수정 후 다음을 검증한다.

- `docs/01_Product.md`에 원래 요구사항이 모두 남아 있고 각 요구사항이 MVP, 후속 마일스톤, 제외 범위 중 하나로 분류되어 있다.
- MVP 범위가 업비트 KRW 마켓 중심으로 축소되어 있다.
- 국내 주식, 미국 주식, 뉴스/공시/리포트, LLM, 전략, 봇, 모의매매, 실거래가 후속 마일스톤에서 누락되지 않았다.
- 마일스톤별 프론트엔드 화면이 명시되어 있다.
- 주식 데이터 수집은 거래소 장 운영시간, 휴장일, 데이터 공급원 제약을 따른다고 표현되어 있다.

## 결정 보류 항목

다음 항목은 제품 문서가 아니라 후속 아키텍처 문서나 ADR에서 결정한다.

- 프론트엔드 기술 스택
- DB 종류와 스키마
- 업비트 API 사용 방식과 rate limit 처리
- 국내/미국 주식 데이터 공급원
- LLM provider와 prompt/schema 설계
- 실거래 주문 API와 리스크 제한 방식
